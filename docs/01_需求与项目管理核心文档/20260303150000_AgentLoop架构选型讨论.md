# Agent Loop 架构选型讨论

> 日期: 2026-03-03 | 状态: **已决策 → 方案 B（LangChain 调用层 + 自研 Agent Loop）**

---

## 背景与核心诉求

ZCli 的 Agent Loop 是整个产品的核心引擎。LLM 调用层的选型直接影响：

- **多模型能力**：任何支持 OpenAI 兼容协议的供应商都能接入（F2）
- **可控性**：Agent Loop 的编排逻辑（SubAgent、会话恢复、权限确认等）必须完全自主
- **社区生态**：作为通用解决方案，遇到问题能快速找到解决方案
- **CLI 启动速度**：目标冷启动 < 500ms

---

## 四个候选方案

### 方案 A：纯 OpenAI 兼容协议（原生 HTTP）

自研 HTTP Client，只对接 `/v1/chat/completions` 协议。

| 维度 | 评估 |
|------|------|
| 依赖 | 零 |
| 多模型 | ✅ OpenAI 兼容协议天然支持 |
| 可控性 | ✅ 最高 |
| 开发成本 | ❌ 高，SSE + tool calling 边界情况多 |
| 社区生态 | ❌ 无，问题全靠自己解决 |
| 高级特性 | ❌ Anthropic 原生特性无法直接用 |

---

### 方案 B：LangChain（调用层）+ 自研 Agent Loop ✅ 已决策

用 `@langchain/core` 作为 LLM 调用层的统一抽象，**不使用 LangGraph**，Agent Loop 编排逻辑完全自研。

```typescript
import { ChatOpenAI } from "@langchain/openai"
import { ChatAnthropic } from "@langchain/anthropic"

// OpenAI 兼容协议的任意供应商
const model = new ChatOpenAI({
  configuration: { baseURL: "https://your-proxy.com/v1/" },
  apiKey: "xxx",
  modelName: "claude-opus-4-6",
})

// 绑定工具
const modelWithTools = model.bindTools([readFileTool, grepTool, bashTool])

// 调用 — 流式
const stream = await modelWithTools.stream(messages)
```

**关键边界：只用 LangChain 的 LLM 调用层，不用 LangGraph**

```
✅ 使用:
  @langchain/core      — 基础类型、消息格式、BaseMessage 等
  @langchain/openai    — ChatOpenAI，支持任意 OpenAI 兼容 baseURL
  @langchain/anthropic — ChatAnthropic，解锁 Claude 原生高级特性
  @langchain/google-genai — ChatGoogleGenerativeAI

❌ 不使用:
  @langchain/langgraph — Agent 状态机（Agent Loop 由 ZCli 自研控制）
  LangChain 的 Agent executor、Chain 编排等高层抽象
```

| 维度 | 评估 |
|------|------|
| 依赖 | 中（按需引入各 Provider 包） |
| 多模型 | ✅ 内置多 Provider 抽象，`ChatOpenAI` 可接任意 OpenAI 兼容服务 |
| 可控性 | ✅ Agent Loop 完全自研，LangChain 只做调用层 |
| 社区生态 | ✅ **最大**，Stack Overflow / GitHub Issues / 博客解决方案密度最高 |
| 高级特性 | ✅ `@langchain/anthropic` 原生支持 extended thinking、cache control |
| 调试体验 | 中（调用层抽象可接受，比 LangGraph 全托管好得多）|
| 版本维护 | 中（需关注 breaking change，但 `@langchain/core` 相对稳定） |
| 启动速度 | 中（按需加载可控，避免全量引入） |

**选择原因**：ZCli 定位为通用解决方案，社区生态是长期竞争力的重要支撑。LangChain 作为纯调用层使用（不用 LangGraph），既获得社区红利，又通过自研 Agent Loop 保住核心控制权。

---

### 方案 C：Vercel AI SDK

用 `ai` 核心包 + `@ai-sdk/openai`（OpenAI 兼容协议封装）。

| 维度 | 评估 |
|------|------|
| 依赖 | ✅ 轻 |
| 多模型 | ✅ OpenAI 兼容协议优先 |
| 可控性 | ✅ Agent Loop 自主控制 |
| 社区生态 | 中（Vercel 生态为主，通用解决方案密度不如 LangChain） |
| 高级特性 | ✅ 可选原生 Provider |

社区生态是与方案 B 的核心差距。对于追求通用解决方案的项目，社区密度是重要考量。

---

### 方案 D：Claude Agent SDK（已分析排除）

**D1: `claude-agent-sdk`（Python SDK）**
本质是 subprocess 控制 Claude Code CLI，不是 LLM 调用库，绑死 Claude，无法多模型。**排除。**

**D2: `@anthropic-ai/sdk`（TypeScript 官方 SDK）**
Claude only，不支持多模型。可作为 `@langchain/anthropic` 的底层依赖被间接使用，不直接引入。

## 最终决策架构

```
┌─────────────────────────────────────────────────────────┐
│              Agent Loop (完全自研)                        │
│                                                          │
│  消息管理 → 轮次控制 → SubAgent 调度 → 权限确认            │
│  上下文压缩 → 并行工具调用 → 会话恢复                      │
│                                                          │
│  ⚠️  不使用 LangGraph，编排逻辑 100% 自研                  │
├─────────────────────────────────────────────────────────┤
│              ZCli LLMProvider 接口 (自研隔离层)            │
│  chat(request) → AsyncIterable<ChatChunk>                │
│  隔离 LangChain 类型，上层 Agent Loop 不感知底层 SDK       │
├───────────────────────┬─────────────────────────────────┤
│  OpenAI 兼容通道       │  各厂商原生通道                   │
│  (主通道, 90% 场景)    │                                  │
│                       │                                  │
│  @langchain/openai    │  @langchain/anthropic            │
│  ChatOpenAI +         │  extended thinking               │
│  自定义 baseURL        │  cache control, citations        │
│                       │                                  │
│  接入任意供应商:        │  @langchain/google-genai         │
│  Anthropic ✓          │  grounding, code execution       │
│  OpenAI ✓             │                                  │
│  Gemini ✓             │  (后续按需引入其他)               │
│  Ollama ✓             │                                  │
│  任意代理 ✓            │                                  │
└───────────────────────┴─────────────────────────────────┘
```

## 风险与缓解措施

| 风险 | 缓解方式 |
|------|----------|
| LangChain 版本 breaking change | `ZCli LLMProvider` 隔离层屏蔽上层，只需改适配器 |
| 依赖体积影响启动速度 | 按需引入各 Provider 包，不全量 import |
| LangGraph 的诱惑（功能蔓延） | 严格约束：不引入 `@langchain/langgraph`，Agent Loop 边界在文档中明确 |
