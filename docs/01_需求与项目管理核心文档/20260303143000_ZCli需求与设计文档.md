# ZCli — 多模型 AI 编程 CLI 助手 · 需求与设计文档

> 版本: v0.4.0 | 日期: 2026-03-03 | 最后修订: 2026-03-13（Provider 重构 + 启动初始化 + 项目级权限机制）

---

## 1. 产品定位

**ZCli** 是一个开源的、支持多模型的 AI 编程 CLI 助手。对标 Claude Code 的交互体验，但不绑定单一 LLM 提供商。

**核心目标**：让开发者在终端中自由选择 Claude / OpenAI / Gemini / 本地模型，获得一致的 AI 辅助编码体验，同时拥有完全的可控性和可扩展性。

**目标用户**：需要在终端环境中使用 AI 辅助编码、调试、重构的开发者。

## 3. 技术选型

### 3.1 运行时环境

| 项目 | 选择 | 说明 |
|------|------|------|
| 语言 | TypeScript (strict mode) | 类型安全，Claude Code 同源技术 |
| 运行时 | Node.js >= 20 LTS (初期) | 稳定可靠，后续预留 Bun.js 兼容 |
| 版本管理 | nvm + `.nvmrc` | 项目根目录放置 `.nvmrc` (内容 `20`)，`nvm use` 自动切换 |
| 包管理 | pnpm | 快速，磁盘效率高，Node/Bun 均支持 |

**运行时演进路线：**

```
Phase 1: Node.js 20+ (当前阶段)
Phase 2: 验证 Bun.js 兼容性 (观察期)
Phase 3: 可选切换到 Bun.js (性能收益明确时)
```

**模块系统约束：**
- **必须使用 ECMAScript Modules (ESM)**，禁止 CommonJS（`require`/`module.exports`）
- `package.json` 声明 `"type": "module"`
- `tsconfig.json` 设置 `"module": "ESNext"` + `"verbatimModuleSyntax": true`
- 所有相对导入必须带 `.js` 后缀（ESM 规范要求）
- 类型导入必须使用 `import type`（verbatimModuleSyntax 强制）
- 优先使用 ECMAScript 原生特性（`#` 私有字段、`Promise.withResolvers()`、`using` 声明等）而非 TypeScript 独有语法（`private`/`enum`/`namespace`）

**Bun 兼容性约束：**
- 不使用 Node.js 独有的非标准 API
- 优先使用 Web 标准 API（fetch, URL, crypto.subtle 等）
- 使用 `execa` 而非直接 `child_process`
- `package.json` 的 scripts 不假设特定运行时

### 4.3 工具执行三级降级链

```

     │
     ▼

  优势: 跨平台一致, 不依赖外部 Shell
     │ 仍然失败
     ▼
Level 3: 报告错误 + 建议

```

| 场景 | L1 Shell | L2 Node.js 补偿 |
|------|----------|-----------------|

## 5. 核心模块详设

### 5.1 对话引擎 (Agent Loop)

AgentLoop 是多轮执行引擎，通过 `AsyncGenerator<AgentEvent>` 将所有中间状态 yield 出去，由 useChat（UI）和 SessionLogger（日志）各取所需。

```
用户输入
  │
  ▼
useChat.submit() → AgentLoop.run(history)
  │
  for turn < MAX_TURNS (20):
  │
  ├── yield* #callLLM(history)
  │     ├── yield llm_start
  │     ├── for await (chunk of provider.chat()):
  │     │     yield text / 收集 tool_calls / 提取 usage
  │     ├── yield llm_usage / llm_error
  │     └── return { toolCalls, aborted }
  │
  ├── if no toolCalls → yield done, return
  │
  └── for each toolCall:
        yield* #executeOneTool(tc, history)
          ├── yield tool_start
          ├── yield* #checkPermission(tc)
          │     ├── yield permission_request (暂停等待 UI 确认)
          │     └── yield permission_grant
          ├── registry.execute() → push result to history
          └── yield tool_done(durationMs, success, resultSummary)
```

**AgentEvent 类型系统（11 种）：**

| 分类 | 类型 | 用途 |
|------|------|------|
| 业务事件 | `text`, `tool_start`, `tool_done`, `permission_request`, `error`, `done` | UI 渲染 |
| 观测事件 | `llm_start`, `llm_usage`, `llm_error`, `tool_fallback`, `permission_grant` | SessionLogger 日志 |

关键能力：
- **AsyncGenerator yield\* 委托**：run() 通过 `yield*` 委托子方法，事件直接穿透到消费者，return 值传递结构化结果
- **流式输出**：逐 chunk 渲染到终端
- **观测与业务分离**：两类事件走同一个流，消费者各自过滤，AgentLoop 不知道谁在消费

#### 5.1.1 并行工具执行 — F17

**现状问题**：LLM 一次回复可能请求多个工具调用（如同时 `glob("*.ts")` + `grep("TODO")`），当前 AgentLoop 逐个串行执行：

```typescript
// 当前实现 — 串行
for (const tc of llmResult.toolCalls) {
  yield* this.#executeOneTool(tc, history)
}
```

串行执行意味着 3 个工具各耗时 500ms = 总耗时 1500ms。并行可压缩到 ~500ms。

**目标**：同一轮内多个 tool_calls 并行执行，但保留事件流的有序 yield（UI 不乱序）。

**核心难点 — AsyncGenerator 与并行的矛盾**：

`yield*` 是同步委托，无法在多个 generator 之间并行推进。要并行执行工具同时保持 yield 事件流，需要改变事件分发方式。

**方案：事件队列 + Promise.allSettled**

```
LLM 返回 toolCalls: [A, B, C]
  │
  ▼
分类判断:
  ├── 需要权限确认的工具 → 串行执行（无法并行等待用户交互）
  └── 安全工具 → 并行执行
        │
        ▼
  ┌─────────────────────────────────────────┐
  │  并行启动:                               │
  │   Promise A: execute(A) → 结果入队列      │
  │   Promise B: execute(B) → 结果入队列      │
  │   Promise C: execute(C) → 结果入队列      │
  │                                          │
  │  事件队列 (保序 yield):                   │
  │   yield tool_start(A)                    │
  │   yield tool_start(B)                    │
  │   yield tool_start(C)                    │
  │   ... 等待完成 ...                        │
  │   yield tool_done(A)  ← 先完成先 yield    │
  │   yield tool_done(C)                     │
  │   yield tool_done(B)                     │
  └─────────────────────────────────────────┘
        │
        ▼
  所有结果追加到 history → 下一轮 LLM 调用
```

**执行策略：**

| 工具类型 | 执行方式 | 原因 |
|----------|----------|------|
| 安全工具（read_file, glob, grep 等） | 并行 | 无副作用或副作用互不干扰 |
| 危险工具（bash, write_file, edit_file） | 串行 | 需要逐个权限确认 + 可能存在文件竞争 |
| 混合（同时有安全和危险工具） | 先并行安全工具，再串行危险工具 | 最大化并行度 |

**分组策略**：

```
toolCalls: [glob, grep, bash, read_file, write_file]
    │
    ▼ 按 isDangerous 分组
    │
    ├── 安全组 (并行): [glob, grep, read_file]
    │     └── Promise.allSettled → 收集结果
    │
    └── 危险组 (串行): [bash, write_file]
          └── 逐个 yield* executeOneTool
```

**事件流保证**：
- `tool_start` 在工具启动时**立即** yield（并行工具的 start 事件几乎同时发出）
- `tool_done` 在工具完成时 yield（完成顺序不固定，但每个 done 一定在对应 start 之后）
- UI 端通过 `toolCallId` 匹配 start/done，不依赖顺序
- SessionLogger 通过 timestamp 排序，不依赖 yield 顺序

**history 追加顺序**：并行工具全部完成后，按**原始 toolCalls 顺序**追加结果到 history（保证 LLM 看到确定性的结果顺序）。

**并发限制**：

```typescript
/** 最大并行工具数，防止同时打开过多文件/进程 */
const MAX_PARALLEL_TOOLS = 5
```

超过限制时分批执行（chunk by MAX_PARALLEL_TOOLS）。

**配置项（`~/.zcli/config.json`）：**

```jsonc
{
  "agent": {
    "parallelTools": true,         // 是否启用并行工具执行（默认 true）
    "maxParallelTools": 5          // 最大并行数
  }
}
```

**性能收益预估**：

| 场景 | 串行耗时 | 并行耗时 | 加速比 |
|------|----------|----------|--------|
| 3 个 glob 搜索 | ~600ms | ~200ms | 3x |
| 2 个 grep + 1 个 read_file | ~900ms | ~300ms | 3x |
| 5 个 read_file | ~500ms | ~100ms | 5x |
| 1 个 bash + 2 个 read_file | ~1200ms | ~700ms | 1.7x (bash 串行) |

**对现有架构的影响**：
- AgentLoop.run() 中工具执行循环从 `for...of yield*` 改为并行调度
- `#executeOneTool` 拆分为：`#startTool`（不含 yield*）+ 事件通过队列回传
- 不影响 LLM 调用流程、不影响事件类型定义、不影响消费者（useChat / SessionLogger / TokenMeter）
- 向后兼容：`parallelTools: false` 退回串行模式

### 5.2 多模型 LLM Provider

```typescript
interface LLMProvider {
  readonly name: string
  chat(messages: Message[], options: ChatOptions): AsyncGenerator<StreamChunk>
  countTokens(messages: Message[]): number
  supportedFeatures: FeatureSet  // tool_use, vision, streaming...
}

/** Provider 返回的流式 chunk，统一接口 */
interface StreamChunk {
  type: 'text' | 'tool_call' | 'usage' | 'done' | 'error'
  text?: string
  toolCall?: ToolCallContent
  usage?: TokenUsage      // 流结束前最后一个 chunk，从 SDK 的 usage_metadata 提取
  error?: string
}
```

- **双协议架构**：Anthropic 原生协议（`@anthropic-ai/sdk` 直连）+ OpenAI 兼容协议（`@langchain/openai`）
- **config 驱动路由**：`ProviderConfig.protocol` 字段决定走哪条协议，新增 provider 只改配置文件，零代码改动
- Anthropic 协议：官方 Claude + 兼容 API（MiniMax 等），通过 `baseURL` + `defaultHeaders` 适配认证
- OpenAI 兼容协议：覆盖 OpenAI / GLM / DeepSeek / Ollama 等
- 运行时通过 `/model` 指令切换（见下方交互设计）
- 各 Provider 的差异（tool_use 格式、token 计算、usage 提取路径）在适配器内部消化
- usage chunk 中包含 `inputTokens` / `outputTokens` / `cacheReadTokens` / `cacheWriteTokens` 四维统计

**协议路由机制：**

```typescript
// src/providers/registry.ts
function resolveProtocol(providerName: string, protocol?: 'anthropic' | 'openai') {
  if (protocol) return protocol                    // 显式声明优先
  return providerName === 'anthropic' ? 'anthropic' : 'openai'  // 默认推断
}
```

**第三方 Anthropic 兼容 API 接入：**


**Token 四维计价体系**：

| 维度 | 说明 | 典型场景 |
|------|------|----------|
| **输入 (input)** | 发送给模型的 token | 每次请求的 prompt + 历史消息 |
| **输出 (output)** | 模型生成的 token | 回复文本 + 工具调用 |
| **缓存写入 (cache_write)** | 首次写入缓存的 token | 长 system prompt、大文件首次注入 |

### 5.6.1 会话与数据清理 — F16

随着使用时间增长，`~/.zcli/` 下会积累大量 JSONL 会话文件和 SQLite 用量记录。F16 提供完整的数据生命周期管理，包括手动清理指令和自动清理策略。

**清理对象（三层）：**

| 层级 | 存储位置 | 数据类型 | 默认保留 |
|------|----------|----------|----------|
| L1 | `sessions/<slug>/*.jsonl` | 会话文件（对话 + 日志事件） | 30 天 |
| L2 | `data/zcli.db → usage_logs` | Token 用量记录 | 90 天 |
| L3 | `data/zcli.db → pricing_rules` | 计价规则 | 永久（不清理） |

**`/gc` 指令（交互模式）：**

```
> /gc

  ── 数据清理 ──

  会话文件:  42 个文件, 共 12.3 MB
    ├── 30 天前: 28 个文件 (8.1 MB) ← 将被清理
    └── 保留:   14 个文件 (4.2 MB)

  用量记录:  1,203 条
    ├── 90 天前: 856 条 ← 将被清理
    └── 保留:   347 条

  确认清理？ (y/N)
> y
  ✓ 已清理 28 个会话文件 (8.1 MB)
  ✓ 已清理 856 条用量记录
```

**`/gc` 子命令：**

| 命令 | 说明 |
|------|------|
| `/gc` | 交互式清理（显示统计 → 确认 → 执行） |
| `/gc --dry-run` | 仅预览，不实际删除 |
| `/gc --days 7` | 清理 7 天前的数据（覆盖默认 30 天） |
| `/gc sessions` | 仅清理会话文件 |
| `/gc usage` | 仅清理用量记录 |

**CLI 模式支持（非交互/脚本场景）：**

```bash
# 非交互模式下无确认直接执行
zcli gc --yes
zcli gc --days 14 --yes

# 开发者模式
pnpm dev:pipe -- gc --dry-run
```

**自动清理策略：**

| 触发时机 | 行为 | 可配置 |
|----------|------|--------|
| 每次启动（REPL 模式） | 后台静默清理过期会话文件 | `config.json → gc.autoCleanOnStart` (默认 true) |
| 每次启动（Pipe 模式） | 不触发自动清理（保持快速启动） | — |
| 手动 `/gc` | 清理会话 + 用量记录 | 保留天数可覆盖 |

**配置项（`~/.zcli/config.json`）：**

```jsonc
{
  "gc": {
    "autoCleanOnStart": true,      // REPL 启动时自动清理过期会话
    "sessionRetentionDays": 30,    // 会话文件保留天数
    "usageRetentionDays": 90       // 用量记录保留天数
  }
}
```

**已有基础设施：**
- `sessionStore.cleanup(retentionDays)` — 已实现，按文件 mtime 清理过期 JSONL + 空目录
- 待实现：`/gc` 指令注册、usage_logs 清理 SQL、dry-run 统计、自动清理触发

**上下文感知设计要点：**
1. **不清理当前活跃会话**：自动清理跳过当前 sessionId 关联的文件
2. **不清理有分支的会话**：如果一个会话有 fork 分支（通过 parentUuid 关联），且分支在保留期内，则主干也不清理
3. **SQLite 清理用 DELETE + VACUUM**：清理 usage_logs 后执行 `VACUUM` 回收磁盘空间
4. **统计信息缓存**：`/gc --dry-run` 的统计结果缓存 5 秒，避免重复扫描文件系统
5. **日志记录**：清理操作写入 SessionLogger（type: `gc_executed`），可在 Dashboard 中追溯

### 5.7 父子 Agent (SubAgent)

主 Agent 在处理复杂任务时，可派生子 Agent 并行或专项执行子任务，每个子 Agent 拥有独立的上下文和 Agent Loop，执行完毕后将结果回收到父 Agent。

**核心流程：**

```
用户提出复杂任务
  │
  ▼
父 Agent (Main Agent Loop)
  │  分析任务 → 判断需要拆分
  │
  ├──→ spawn SubAgent A (专项: 搜索代码库)
  │      独立上下文 / 独立 Agent Loop / 可用工具子集
  │
  ├──→ spawn SubAgent B (专项: 查阅文档)   ← 可并行
  │      独立上下文 / 独立 Agent Loop / 可用工具子集
  │
  │  等待子 Agent 完成 (并行或顺序)
  │
  ├──← SubAgent A 返回结果摘要
  ├──← SubAgent B 返回结果摘要
  │
  ▼
父 Agent 汇总子 Agent 结果 → 继续推理 → 响应用户
```

**SubAgent 类型体系：**

| 类型 | 职责 | 可用工具 | 典型场景 |
|------|------|----------|----------|
| `explore` | 代码探索与搜索 | Read, Glob, Grep (只读) | 快速了解代码库结构 |
| `general` | 通用任务执行 | 全部工具 | 复杂多步骤子任务 |
| `plan` | 方案设计与规划 | Read, Glob, Grep, WebFetch (只读) | 架构设计、方案对比 |
| `review` | 代码审查 | Read, Glob, Grep (只读) | 代码质量检查 |
| `custom` | 用户自定义 | 由 Skills/Plugin 声明 | 通过插件扩展的专项 Agent |

**SubAgent 与父 Agent 的关系：**

```typescript
interface SubAgentConfig {
  type: "explore" | "general" | "plan" | "review" | "custom"
  prompt: string                    // 子任务描述
  model?: string                    // 可指定不同模型 (如用便宜模型跑探索)
  tools?: string[]                  // 可用工具白名单 (默认按类型)
  maxTurns?: number                 // 最大轮次限制
  timeout?: number                  // 超时 (ms)
  inheritContext?: boolean          // 是否继承父 Agent 的对话历史
  runInBackground?: boolean         // 后台执行，不阻塞父 Agent
}

interface SubAgentResult {
  agentId: string
  status: "completed" | "timeout" | "error"
  result: string                    // 子 Agent 的最终回复 (摘要)
  tokenUsage: TokenUsage            // 子 Agent 消耗的 token
  turnsUsed: number                 // 实际执行的轮次
  duration_ms: number
}
```

**关键设计决策：**

**1. 上下文隔离**
- 子 Agent 默认拥有独立上下文（不继承父 Agent 完整历史）
- 父 Agent 通过 `prompt` 字段传递必要的任务描述和上下文摘要
- 可选 `inheritContext: true` 让子 Agent 看到父 Agent 的对话历史（成本更高）

**2. 并行与顺序**
- 多个子 Agent 之间无依赖时并行执行（Promise.all）
- 有依赖关系时顺序执行，前一个的结果可注入后一个的 prompt
- 支持后台执行模式（`runInBackground`），父 Agent 不阻塞继续处理其他事

**3. 模型降级**
- 子 Agent 可指定不同的模型，实现成本优化
- 例：父 Agent 用 Opus 做决策，子 Agent 用 Sonnet/Haiku 做探索和搜索
- 模型选择策略可由 Skills 或用户配置控制

**4. 资源管控**
- 每个子 Agent 有独立的 maxTurns 和 timeout 限制
- 子 Agent 的 token 消耗独立计量，归到父会话的 usage 中
- 父 Agent 可设置全局的子 Agent 并发上限

**5. 结果回收**
- 子 Agent 完成后返回结构化的 `SubAgentResult`
- 父 Agent 拿到的是结果摘要（不是子 Agent 的完整对话历史）
- 子 Agent 的完整执行日志写入日志系统，可通过 Dashboard 回溯

**执行拓扑示例：**

```
Main Agent (Opus)
├── SubAgent #1 [explore] (Haiku) ──── 并行 ────┐
├── SubAgent #2 [explore] (Haiku) ──── 并行 ────┤
│                                                │
│  ← 收集 #1, #2 结果                            │
│                                                │
├── SubAgent #3 [general] (Sonnet) ── 依赖 #1,#2 结果
│
│  ← 收集 #3 结果
│
▼ 汇总所有结果 → 最终响应用户
```

### 5.9 非交互模式 (Pipe Mode) — F15

除了交互式 REPL，还支持单次执行模式：命令行直接传入问题，执行完自动退出，纯文本输出到 stdout。

**使用方式：**

```bash
# 位置参数：直接提问
zcli "帮我看看这个项目的目录结构"

# -p / --prompt 参数：等价写法
zcli -p "把这个函数重构成 async"

# 管道输入：stdin 作为上下文
cat error.log | zcli "分析一下这个报错"
git diff | zcli "帮我写 commit message"

# 组合：stdin 上下文 + 位置参数提问
cat src/utils.ts | zcli "这个文件有什么问题？"

# 输出重定向：结果写入文件
zcli "生成一个 .gitignore" > .gitignore
```

**与 REPL 模式的对比：**

| 维度 | REPL 模式（默认） | 非交互模式 |
|------|-------------------|-----------|
| 启动 | `zcli` | `zcli "问题"` 或 `zcli -p "问题"` |
| UI | Ink 渲染（WelcomeScreen、InputBar 等） | 无 UI，纯文本输出到 stdout |
| 交互 | 多轮对话 | 单次问答，执行完退出 |
| 工具权限 | 交互式确认弹窗 | `--yes` 自动批准，否则跳过危险工具 |
| 退出码 | N/A | 0 = 成功，1 = 错误 |
| 适用 | 日常开发对话 | 脚本、CI/CD、管道组合 |

**CLI 参数扩展：**

```
zcli [options] [prompt]

位置参数:
  prompt                     直接提问（如有则进入非交互模式）

选项:
  -p, --prompt <text>        提问内容（等价于位置参数）
  -m, --model <name>         指定模型
  --provider <name>          指定供应商
  --yes                      自动批准所有工具执行（危险操作慎用）
  --no-tools                 禁用工具调用，纯对话模式
  --json                     输出 JSON 格式（含 token 用量等元数据）
  --resume [sessionId]       恢复历史会话（无 prompt 时进入 REPL）
  --show-resume              展示可恢复的会话列表
```

**模式判断逻辑：**

```
CLI 启动
  │
  ├── 有 prompt（位置参数或 -p）？
  │     ├── 是 → 非交互模式
  │     │     ├── 检测 stdin 是否为管道 → 有则读取作为上下文前缀
  │     │     ├── 构建消息 → AgentLoop.run()
  │     │     ├── 输出到 stdout（纯文本 / JSON）
  │     │     └── exit(0) 或 exit(1)
  │     │
  │     └── 否 → 交互模式（Ink REPL）
  │           └── 正常渲染 UI
```

**输出格式：**

默认纯文本输出（只输出 LLM 回复文本），适合管道和重定向：

```json
    "inputTokens": 12,
```

```bash
git diff | pnpm dev:pipe -- "帮我写 commit message"

# 输出重定向
npx tsx bin/zcli.ts "生成 .gitignore" > .gitignore
```

> **注意**：`pnpm dev` / `pnpm dev:pipe` 后面的 `--` 是必须的，它告诉 pnpm 将后续参数透传给脚本而非自己消费。直接用 `npx tsx bin/zcli.ts` 则不需要 `--`。

**关键设计决策：**
1. **stdin 检测**：通过 `process.stdin.isTTY` 判断是否为管道输入，是则先读完 stdin 再发送
2. **无 Ink 渲染**：非交互模式不启动 React/Ink，直接 `process.stdout.write()` 输出，避免终端控制符污染管道
3. **工具权限**：默认跳过需要确认的危险工具（返回 "tool execution skipped: requires interactive approval"），`--yes` 可覆盖
4. **退出码**：AgentLoop 正常完成 exit(0)，出错 exit(1)，方便脚本判断
5. **SessionLogger 复用**：非交互模式同样创建会话、记录日志、写入 usage_logs，保持可观测性一致
6. **动态 import REPL 依赖**：bin/zcli.ts 在 Pipe 模式下不加载 React/Ink（通过 `await import()`），启动更快、无终端控制符污染

## 8. 可观测性与运营管控

本章覆盖 Agent 运行的「上帝视角」能力：知道它做了什么、花了多少钱、出了什么问题。

### 8.1 结构化日志系统（事件流 + 观察者模式）

**设计原则**：AgentLoop 通过 AsyncGenerator yield 出 AgentEvent 事件流，SessionLogger 作为独立观察者订阅事件流，将 AgentEvent 转换为 SessionEvent 并写入 JSONL。业务代码零侵入，日志逻辑集中在 SessionLogger 一处。

**日志存储**：不设独立 `logs/` 目录，所有日志事件直接追加到会话 JSONL 文件中（复用 F8 持久化基础设施）。

```
~/.zcli/sessions/<project-slug>/          # 日志和对话消息在同一个 JSONL 文件中
  └── <uuid>.jsonl
```

**理由：**
- session JSONL 已有 append 基础设施和事件树结构（uuid/parentUuid）
- `loadMessages()` 自动跳过非 user/assistant 事件，日志事件不影响对话恢复
- 一个会话一个文件，无跨文件关联复杂度
- 诊断时直接打开一个文件即可看到完整的对话 + 执行链路

**架构：**

```
AgentLoop (AsyncGenerator<AgentEvent>)
    │
    ├──→ useChat (UI 消费者：更新界面)
    │
    └──→ SessionLogger (日志消费者：写入 JSONL)
              │
              ├── consume(event)          ← AgentEvent → SessionEvent 转换
              ├── logUserMessage()        ← 用户消息
              ├── logAssistantMessage()   ← 助手回复
              ├── logMcpConnect()         ← MCP 连接事件（通过回调，不经过 AgentLoop）
              └── finalize()             ← session_end 汇总
```

**单条事件结构：**

```jsonc
{
  "sessionId": "a1b2c3d4-...",
  "type": "llm_call_end",              // 事件类型 (见下表)
  "timestamp": "2026-03-03T14:30:22.123Z",
  "uuid": "evt-001",                   // 当前事件 ID
  "parentUuid": "evt-000",             // 前一个事件 ID（形成事件链）
  "cwd": "/home/user/project",
  "inputTokens": 1520,                 // 可选字段，按事件类型而定
  "outputTokens": 380,
  "stopReason": "end_turn"
}
```

**事件类型（start/end 配对）：**

| 类型 | 来源 | 说明 |
|------|------|------|
| `session_start` | SessionStore | 会话创建 |
| `session_resume` | SessionStore | 会话恢复 |
| `user` | SessionLogger | 用户消息 |
| `assistant` | SessionLogger | LLM 回复 |
| `llm_call_start` | AgentEvent `llm_start` | LLM 调用开始（provider, model, messageCount） |
| `llm_call_end` | AgentEvent `llm_usage` / `llm_error` | LLM 调用结束（tokens 统计 / 错误标记） |
| `tool_call_start` | AgentEvent `tool_start` | 工具执行开始（toolName, args） |
| `tool_call_end` | AgentEvent `tool_done` | 工具执行结束（durationMs, success, resultSummary） |
| `mcp_connect_start` | McpManager `onConnect` | MCP 连接开始 |
| `mcp_connect_end` | McpManager `onConnect` | MCP 连接结束（success, toolCount） |
| `tool_fallback` | AgentEvent `tool_fallback` | 工具降级（预留，当前无 emit 点） |
| `permission_grant` | AgentEvent `permission_grant` | 权限授予 |
| `error` | AgentEvent `error` | 运行时错误 |
| `session_end` | SessionLogger `finalize()` | 会话结束汇总（cumulative tokens/tools/errors/duration） |

**日志还原对话**：直接打开 `<uuid>.jsonl` 文件，按 `parentUuid` 链或 `timestamp` 排序，即可重建完整的对话时间线。`scripts/inspect-session.mjs` 诊断工具提供事件时间线、配对检查和 parentUuid 链验证。

### 8.2 Token 计量与计费

**计量**：每次 LLM 调用时自动记录 input_tokens / output_tokens / cache_read_tokens / cache_write_tokens，由 Provider 适配器负责从 API 响应中提取。

**计费数据存储**：SQLite，支持多维度聚合查询。

```sql
-- ═══ Token 使用记录表 ═══
-- 每次 LLM 调用产生一条记录，记录四维 token 消耗和费用
CREATE TABLE usage_logs (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,  -- 自增主键
  session_id      TEXT NOT NULL,                      -- 所属会话 ID
  timestamp       TEXT NOT NULL,                      -- 记录时间 (ISO 8601)
  provider        TEXT NOT NULL,                      -- LLM 供应商 (anthropic/openai/glm/deepseek/ollama)
  model           TEXT NOT NULL,                      -- 模型标识 (claude-opus-4-6/gpt-4o/glm-5/...)
  input_tokens    INTEGER NOT NULL,                   -- 输入 token 数
  output_tokens   INTEGER NOT NULL,                   -- 输出 token 数
  cache_read      INTEGER NOT NULL DEFAULT 0,         -- 缓存读取 token 数
  cache_write     INTEGER NOT NULL DEFAULT 0,         -- 缓存写入 token 数
  duration_ms     INTEGER,                            -- LLM 调用耗时（毫秒）
  cost_amount     REAL,                               -- 费用（对应币种金额），无匹配规则时为 NULL
  cost_currency   TEXT NOT NULL DEFAULT 'USD',        -- 费用币种（USD/CNY），取自匹配的 pricing_rule
  pricing_rule_id INTEGER                             -- 匹配的计价规则 ID
);

-- ═══ 计价规则表 ═══
-- 按 provider + model 通配符匹配，支持时间范围、优先级和多币种
CREATE TABLE pricing_rules (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,  -- 自增主键
  provider          TEXT NOT NULL,                      -- LLM 供应商
  model_pattern     TEXT NOT NULL,                      -- 模型匹配模式（支持末尾 * 通配符）
  input_price       REAL NOT NULL,                      -- 输入价格 (货币/百万 token)
  output_price      REAL NOT NULL,                      -- 输出价格 (货币/百万 token)
  cache_read_price  REAL NOT NULL DEFAULT 0,            -- 缓存读取价格 (货币/百万 token)
  cache_write_price REAL NOT NULL DEFAULT 0,            -- 缓存写入价格 (货币/百万 token)
  currency          TEXT NOT NULL DEFAULT 'USD',        -- 价格币种（USD = 美元，CNY = 人民币）
  effective_from    TEXT NOT NULL,                      -- 生效起始日期 (ISO 8601)
  effective_to      TEXT,                               -- 生效截止日期（NULL 表示永久有效）
  source            TEXT,                               -- 价格来源说明
  priority          INTEGER NOT NULL DEFAULT 0          -- 匹配优先级（越大越优先）
);

-- ═══ 字段注释元数据表 ═══
-- SQLite 不支持列级 COMMENT，用此表存储字段说明，可运行时查询
CREATE TABLE schema_comments (
  table_name    TEXT NOT NULL,       -- 所属表名
  column_name   TEXT NOT NULL,       -- 字段名
  comment       TEXT NOT NULL,       -- 字段说明
  PRIMARY KEY (table_name, column_name)
);

CREATE INDEX idx_pricing_lookup ON pricing_rules(provider, model_pattern, effective_from);
CREATE INDEX idx_usage_session ON usage_logs(session_id);
CREATE INDEX idx_usage_timestamp ON usage_logs(timestamp);
```

**多币种设计**：国际厂商（Anthropic/OpenAI/Google）按 USD 计价，国内厂商（GLM/DeepSeek）按 CNY 计价。不做汇率换算，按原始币种存储和展示。UI 自动显示对应货币符号（$ / ¥），多币种用 `+` 连接（如 `$0.0225 + ¥0.0036`）。

**计价匹配逻辑**：

```
LLM 调用完成 → 提取 provider + model + timestamp
  │
  ▼
查询 pricing_rules:
  WHERE provider = ?
    AND model 匹配 model_pattern
    AND effective_from <= timestamp
    AND (effective_to IS NULL OR effective_to > timestamp)
  ORDER BY priority DESC, effective_from DESC
  LIMIT 1
  │
  ▼
计算费用: (input × input_price + output × output_price
         + cache_write × cache_write_price + cache_read × cache_read_price) / 1_000_000
  │
  ▼
写入 usage_logs.cost_amount
```

**典型场景：**

| 场景 | 解决方式 |
|------|----------|
| Anthropic 官方 Claude Opus 调价 | 新增一条 pricing_rule，effective_from 设为调价日期 |
| 通过代理商 X 调用 Claude，价格不同 | provider 设为 "proxy-x"，独立计价规则 |
| 同时用官方和代理商 | 在 LLM Provider 配置中标记 provider 来源，自动匹配对应计价规则 |
| 本地 Ollama 模型免费 | pricing_rule 的 input_price / output_price 均设为 0 |

### 8.4 数据存储策略 (JSONL + SQLite 混合)

```
~/.zcli/
├── sessions/<project-slug>/                   # JSONL - 对话 + 日志（统一存储）
│   ├── <uuid>.jsonl                           # 每个会话一个文件，包含消息和观测事件
│   └── <uuid>.jsonl
└── data/
    └── zcli.db                                # SQLite - 计费 & 统计（已实现，多币种）
        ├── usage_logs                         # Token 使用记录
        ├── pricing_rules                      # 计价规则（21 条默认规则，USD/CNY）
        └── schema_comments                    # 字段注释元数据
```

**关键设计决策：会话与日志合并存储**

原方案 `sessions/` 存 JSON、`logs/` 存 JSONL 分离。实际实现中合并为单一 JSONL 文件：
- 对话消息（user/assistant）和观测事件（llm_call_start/end、tool_call_start/end 等）写入同一个 JSONL
- `loadMessages()` 通过 type 过滤只提取 user/assistant，恢复对话历史
- 诊断时打开一个文件即可看到完整链路，无需跨文件关联
- 消除了 session JSON 与 log JSONL 的同步维护问题

**命名约定汇总：**

| 文件类型 | 命名格式 | 示例 |
|----------|----------|------|
| 会话文件 | `<uuid>.jsonl`（按项目目录隔离） | `sessions/a-dev-work-project/a1b2c3d4.jsonl` |
| 项目文档 | `YYYYMMDDHHMMSS_文档标题.md` | `20260303143000_ZCli需求与设计文档.md` |

**分工逻辑：**

| 存储 | 格式 | 数据 | 理由 |
|------|------|------|------|
| 会话+日志 | JSONL | 对话消息 + Agent 执行事件流 | 统一文件，按 uuid/parentUuid 链排序，追加写入高效 |
| 计费 | SQLite | usage_logs + pricing_rules | 需要 SUM/GROUP BY/JOIN 等聚合查询 |
| 配置 | JSON | config/mcp/skills 配置 | 用户手动编辑友好 |

## 10. 启动初始化与权限机制

### 10.1 启动初始化（`src/core/initializer.ts`）

CLI 入口（`bin/zcli.ts`）在参数解析后、模式判断前调用 `initialize()`，执行前置检查：

| # | 检查项 | 行为 |
|---|--------|------|
| 1 | `~/.zcli/` 目录 | 不存在则创建 |
| 2 | `~/.zcli/config.json` | 不存在 → 写默认模板；存在但缺关键字段 → 补全；JSON 损坏 → 备份 `.bak` + 重置 |
| 3 | `~/.zcli/.mcp.json` | 不存在 → 写空模板 `{ "mcpServers": {} }` |
| 4 | `<project>/.zcli/settings.local.json` | 不存在 → 写空权限模板 `{ "permissions": { "allow": [] } }` |
| 5 | 启动诊断 | 当前 provider 的 apiKey 为空 → stderr 输出警告 |

**设计原则：**

- **幂等**：已存在的文件不会被覆盖
- **非阻塞**：诊断失败、备份失败不阻塞启动
- **stderr 输出**：创建提示和警告走 stderr，不污染交互 UI 或管道 stdout

**返回值：**

```typescript
interface InitDiagnostic {
  warnings: string[]   // 配置问题警告
  created: string[]    // 本次创建的文件路径
}
```

### 10.2 项目级权限机制（`src/config/permissions.ts`）

**核心思想**：通过 `.zcli/settings.local.json` 的白名单，让项目内常用的危险工具（bash、write、edit）免确认直接执行，减少交互摩擦。

**PermissionManager 类：**

```typescript
class PermissionManager {
  constructor(rules: string[], registeredTools: string[])
  isAllowed(toolName: string): boolean
  static fromProjectDir(projectDir?: string, registeredTools?: string[]): PermissionManager
}
```

**规则解析（基于 ToolRegistry 动态匹配，无硬编码映射表）：**

```
FriendlyName(*)  →  提取友好名，转小写，在注册工具中查找：
                     1. 精确匹配（bash → bash）
                     2. 前缀+下划线匹配（read → read_file, edit → edit_file）
                     3. 找不到 → 规则被忽略

通配符后缀       →  mcp__* → 匹配所有以 mcp__ 开头的工具名

精确匹配         →  bash / mcp__ctx7__query-docs → 直接比较
```

**权限检查优先级（5 级决策链）：**

```
工具调用
  │
  ├─ ① isSidechain（子 Agent 模式）────────→ 自动放行
  │
  ├─ ② isDangerous === false ──────────────→ 自动放行
  │
  ├─ ③ 项目级白名单（settings.local.json）─→ 匹配则放行
  │
  ├─ ④ Session 级白名单（用户选 "always"）─→ 匹配则放行
  │
  └─ ⑤ PermissionDialog 弹窗 ─────────────→ 用户决定
```

**两种模式的接入方式：**

| 模式 | 接入点 | 白名单之外的行为 |
|------|--------|------------------|
| 交互模式（REPL） | `useChat.ts` permission_request 事件 | 弹 PermissionDialog 询问 |
| 管道模式（Pipe） | `pipe-runner.ts` permission_request 事件 | 看 `--yes` 参数，有则放行，无则拒绝 |
