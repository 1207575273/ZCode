# ZCli — 多模型 AI 编程 CLI 助手 · 需求与设计文档

> 版本: v0.1.0-draft | 日期: 2026-03-03

---

## 1. 产品定位

**ZCli** 是一个开源的、支持多模型的 AI 编程 CLI 助手。对标 Claude Code 的交互体验，但不绑定单一 LLM 提供商。

**核心目标**：让开发者在终端中自由选择 Claude / OpenAI / Gemini / 本地模型，获得一致的 AI 辅助编码体验，同时拥有完全的可控性和可扩展性。

**目标用户**：需要在终端环境中使用 AI 辅助编码、调试、重构的开发者。

## 3. 技术选型

### 3.1 运行时环境

| 项目 | 选择 | 说明 |
|------|------|------|
| 语言 | TypeScript (strict mode) | 类型安全，Claude Code 同源技术 |
| 运行时 | Node.js >= 20 LTS (初期) | 稳定可靠，后续预留 Bun.js 兼容 |
| 版本管理 | nvm + `.nvmrc` | 项目根目录放置 `.nvmrc` (内容 `20`)，`nvm use` 自动切换 |
| 包管理 | pnpm | 快速，磁盘效率高，Node/Bun 均支持 |

**运行时演进路线：**

```
Phase 1: Node.js 20+ (当前阶段)
Phase 2: 验证 Bun.js 兼容性 (观察期)
Phase 3: 可选切换到 Bun.js (性能收益明确时)
```

**Bun 兼容性约束：**
- 不使用 Node.js 独有的非标准 API
- 优先使用 Web 标准 API（fetch, URL, crypto.subtle 等）
- 使用 `execa` 而非直接 `child_process`
- `package.json` 的 scripts 不假设特定运行时

### 3.2 核心依赖

| 领域 | 技术 | 理由 |
|------|------|------|
| 终端 UI | Ink 5 + React 18 | CLI 组件化渲染，流式输出友好 |
| LLM 调用层 | `@langchain/openai` + `@langchain/anthropic` + `@langchain/google-genai` | 社区生态最大，OpenAI 兼容协议接任意供应商，不使用 LangGraph |
| MCP | `@modelcontextprotocol/sdk` | MCP 官方客户端库 |
| Shell 执行 | `execa` | 比原生 child_process 更好的跨平台支持 |
| 文件搜索 | `fast-glob` + `@vscode/ripgrep` | 高性能文件匹配和内容搜索 |
| Markdown 渲染 | `marked` + `chalk` | 终端 Markdown 高亮 |
| 持久化 (对话/日志) | 本地 JSON 文件 | 可读性好，方便调试和日志还原 |
| 持久化 (计费/统计) | SQLite (`better-sqlite3`) | 支持多维度聚合查询，嵌入式零部署 |
| WebUI | Hono (后端) + React (前端) | Hono 轻量快速，前端复用 React 生态 |
| 测试 | Vitest | 快速，TS 原生支持 |
| 构建 | tsup | 零配置 TS 打包，输出 ESM |

### 4.2 跨平台 Shell 适配

**Shell 选择策略：**

```
ShellExecutor 初始化:

1. 检测 process.platform
   ├─ "win32" →
   │    ├─ 扫描 Git Bash:
   │    │   - C:\Program Files\Git\bin\bash.exe
   │    │   - C:\Program Files (x86)\Git\bin\bash.exe
   │    │   - 环境变量 PATH 中查找 bash.exe
   │    ├─ 找到 Git Bash → 优先使用 (统一 Unix 语法)
   │    └─ 未找到 → 回退 PowerShell
   │
   └─ "linux" / "darwin" →
        └─ /bin/bash (回退 /bin/sh)
```

**用户可通过配置覆盖：**

```jsonc
// ~/.zcli/config.json
{
  "shell": {
    "preferred": "auto",     // "auto" | "gitbash" | "powershell" | "bash"
    "fallbackToNode": true,  // 启用 Node.js 降级补偿
    "timeout": 30000         // Shell 命令超时 (ms)
  }
}
```

### 4.3 工具执行三级降级链

```
Tool 调用请求
     │
     ▼
Level 1: Shell 执行 (首选)
  Git Bash / PowerShell / Bash
  优势: 最接近用户本地环境, 支持管道/通配符
     │ 失败 (超时/命令不存在/权限...)
     ▼
Level 2: Node.js Native (补偿)
  fs / child_process / path / glob 等
  优势: 跨平台一致, 不依赖外部 Shell
     │ 仍然失败
     ▼
Level 3: 报告错误 + 建议
  返回结构化错误信息给 LLM
  由 LLM 决策下一步 (换工具/换方法/请求用户帮助)
```

**降级场景对照表：**

| 场景 | L1 Shell | L2 Node.js 补偿 |
|------|----------|-----------------|
| 查找文件 | `find . -name "*.ts"` | `fast-glob("**/*.ts")` |
| 读取文件 | `cat file.txt` | `fs.readFile()` |
| 搜索内容 | `grep -r "pattern"` | ripgrep 绑定 or 逐文件 match |
| 网络请求 | `curl url` | `fetch()` |
| 进程列表 | `ps aux` / `tasklist` | `child_process.exec` + 平台适配 |
| 文件权限 | `chmod 755 file` | `fs.chmod()` |

**内置工具的两层分类：**

| 层级 | 说明 | 工具 |
|------|------|------|
| Native 层 | 直接用 Node.js API，天然跨平台 | Read, Write, Edit, Glob, Grep |
| Shell 层 | 通过 ShellExecutor，走降级链 | Bash 命令执行, 进程管理 |

## 5. 核心模块详设

### 5.1 对话引擎 (Agent Loop)

```
用户输入
  │
  ▼
消息构建 (system prompt + skills + 历史消息 + 用户输入)
  │
  ▼
LLM 流式调用 ──→ 实时渲染到终端
  │
  ▼
响应解析: 纯文本 or 工具调用?
  ├─ 纯文本 → 渲染输出 → 等待下一轮输入
  └─ 工具调用 → 权限检查 → 执行工具 → 结果回传 LLM → 继续循环
```

关键能力：
- **流式输出**：SSE / stream，逐 token 渲染到终端
- **上下文压缩**：对话超长时自动摘要历史消息，控制 token 开销
- **并行工具调用**：LLM 返回多个工具调用时并发执行

### 5.2 多模型 LLM Provider

```typescript
interface LLMProvider {
  name: string
  chat(messages: Message[], options: ChatOptions): AsyncIterable<StreamChunk>
  countTokens(messages: Message[]): number
  supportedFeatures: FeatureSet  // tool_use, vision, streaming...
}
```

- 内置适配器：Claude (Anthropic API)、OpenAI、Gemini、Ollama（本地模型）
- 运行时通过 `/model` 指令切换（见下方交互设计）
- 各 Provider 的差异（tool_use 格式、token 计算）在适配器内部消化

**`/model` 子命令：**

| 命令 | 说明 | 示例 |
|------|------|------|
| `/model` | 展示模型列表，交互式选择 | `/model` |
| `/model <name>` | 直接切换到指定模型 | `/model gpt-4o` |
| `/model <name> --provider <p>` | 同名模型指定供应商 | `/model claude-opus-4-6 --provider proxyhub` |
| `/model info` | 显示当前模型详情（四维价格、特性、上下文窗口、本会话消耗） | `/model info` |
| `/model list` | 纯列表模式（非交互） | `/model list` |

**价格数据来源**：从 SQLite `pricing_rules` 表实时查询，展示当前生效的计价规则。未配置的价格维度显示 `--`。

**Token 四维计价体系**：

| 维度 | 说明 | 典型场景 |
|------|------|----------|
| **输入 (input)** | 发送给模型的 token | 每次请求的 prompt + 历史消息 |
| **输出 (output)** | 模型生成的 token | 回复文本 + 工具调用 |
| **缓存写入 (cache_write)** | 首次写入缓存的 token | 长 system prompt、大文件首次注入 |
| **缓存读取 (cache_read)** | 命中缓存的 token | 后续轮次复用已缓存的上下文，通常远低于输入价 |

> 并非所有模型/供应商都支持缓存计价。不支持的维度在列表中显示 `--`，计费时该项为 0。

**模型去重与排序逻辑**：
- 同一模型在多个供应商下分别显示（如上方 claude-opus-4-6 同时出现在 Anthropic 和 ProxyHub）
- 默认排序：当前选中置顶 → 按供应商分组 → 组内按模型名排序
- 本地模型（Ollama）价格显示 `FREE`

### 5.3 Skills 混合系统

**Prompt Skills（Markdown 层）：**

```
~/.zcli/skills/
  ├── my-skill.md          # 用户自定义
  └── community/           # 社区分享
      └── tdd.md
```

- 格式兼容 Claude Code 的 skill markdown（frontmatter + 正文）
- `/skill load <name>` 和 `/skill unload <name>` 动态管理
- 注入时将 markdown 内容插入 system prompt

**Code Plugins（JS/TS 层）：**

```
~/.zcli/plugins/
  └── my-plugin/
      ├── package.json
      └── index.ts          # 实现 PluginInterface
```

- 可注册新工具（Tool）、新指令（Command）、新 Hook
- 沙箱隔离，限制文件系统和网络访问范围
- `/plugin install/remove` 管理生命周期

```
内置指令:  /help /model /skill /plugin /mcp /resume /clear /config

```

- 指令解析器支持参数、子命令、Tab 自动补全
- Skills 可通过 frontmatter 声明 `slash_command` 注册为斜杠指令

### 5.6 会话持久化与恢复

```
~/.zcli/sessions/                                    # 命名格式: sess_YYYYMMDD_HHMMSS_SSS
  ├── sess_20260303_143022_317.json
  └── sess_20260303_160815_042.json
```

- 命名精确到毫秒 `sess_YYYYMMDD_HHMMSS_SSS.json`，避免同秒内创建冲突
- 每次对话自动保存到本地 JSON 文件
- `zcli --resume` 列出历史会话供选择继续
- 存储内容：消息历史、当前模型、已加载 Skills、工具调用记录
- 对应日志文件同名存放在 `~/.zcli/logs/` 下（.json → .jsonl）

### 5.7 父子 Agent (SubAgent)

主 Agent 在处理复杂任务时，可派生子 Agent 并行或专项执行子任务，每个子 Agent 拥有独立的上下文和 Agent Loop，执行完毕后将结果回收到父 Agent。

**核心流程：**

```
用户提出复杂任务
  │
  ▼
父 Agent (Main Agent Loop)
  │  分析任务 → 判断需要拆分
  │
  ├──→ spawn SubAgent A (专项: 搜索代码库)
  │      独立上下文 / 独立 Agent Loop / 可用工具子集
  │
  ├──→ spawn SubAgent B (专项: 查阅文档)   ← 可并行
  │      独立上下文 / 独立 Agent Loop / 可用工具子集
  │
  │  等待子 Agent 完成 (并行或顺序)
  │
  ├──← SubAgent A 返回结果摘要
  ├──← SubAgent B 返回结果摘要
  │
  ▼
父 Agent 汇总子 Agent 结果 → 继续推理 → 响应用户
```

**SubAgent 类型体系：**

| 类型 | 职责 | 可用工具 | 典型场景 |
|------|------|----------|----------|
| `explore` | 代码探索与搜索 | Read, Glob, Grep (只读) | 快速了解代码库结构 |
| `general` | 通用任务执行 | 全部工具 | 复杂多步骤子任务 |
| `plan` | 方案设计与规划 | Read, Glob, Grep, WebFetch (只读) | 架构设计、方案对比 |
| `review` | 代码审查 | Read, Glob, Grep (只读) | 代码质量检查 |
| `custom` | 用户自定义 | 由 Skills/Plugin 声明 | 通过插件扩展的专项 Agent |

**SubAgent 与父 Agent 的关系：**

```typescript
interface SubAgentConfig {
  type: "explore" | "general" | "plan" | "review" | "custom"
  prompt: string                    // 子任务描述
  model?: string                    // 可指定不同模型 (如用便宜模型跑探索)
  tools?: string[]                  // 可用工具白名单 (默认按类型)
  maxTurns?: number                 // 最大轮次限制
  timeout?: number                  // 超时 (ms)
  inheritContext?: boolean          // 是否继承父 Agent 的对话历史
  runInBackground?: boolean         // 后台执行，不阻塞父 Agent
}

interface SubAgentResult {
  agentId: string
  status: "completed" | "timeout" | "error"
  result: string                    // 子 Agent 的最终回复 (摘要)
  tokenUsage: TokenUsage            // 子 Agent 消耗的 token
  turnsUsed: number                 // 实际执行的轮次
  duration_ms: number
}
```

**关键设计决策：**

**1. 上下文隔离**
- 子 Agent 默认拥有独立上下文（不继承父 Agent 完整历史）
- 父 Agent 通过 `prompt` 字段传递必要的任务描述和上下文摘要
- 可选 `inheritContext: true` 让子 Agent 看到父 Agent 的对话历史（成本更高）

**2. 并行与顺序**
- 多个子 Agent 之间无依赖时并行执行（Promise.all）
- 有依赖关系时顺序执行，前一个的结果可注入后一个的 prompt
- 支持后台执行模式（`runInBackground`），父 Agent 不阻塞继续处理其他事

**3. 模型降级**
- 子 Agent 可指定不同的模型，实现成本优化
- 例：父 Agent 用 Opus 做决策，子 Agent 用 Sonnet/Haiku 做探索和搜索
- 模型选择策略可由 Skills 或用户配置控制

**4. 资源管控**
- 每个子 Agent 有独立的 maxTurns 和 timeout 限制
- 子 Agent 的 token 消耗独立计量，归到父会话的 usage 中
- 父 Agent 可设置全局的子 Agent 并发上限

**5. 结果回收**
- 子 Agent 完成后返回结构化的 `SubAgentResult`
- 父 Agent 拿到的是结果摘要（不是子 Agent 的完整对话历史）
- 子 Agent 的完整执行日志写入日志系统，可通过 Dashboard 回溯

**执行拓扑示例：**

```
Main Agent (Opus)
├── SubAgent #1 [explore] (Haiku) ──── 并行 ────┐
├── SubAgent #2 [explore] (Haiku) ──── 并行 ────┤
│                                                │
│  ← 收集 #1, #2 结果                            │
│                                                │
├── SubAgent #3 [general] (Sonnet) ── 依赖 #1,#2 结果
│
│  ← 收集 #3 结果
│
▼ 汇总所有结果 → 最终响应用户
```

## 8. 可观测性与运营管控

本章覆盖 Agent 运行的「上帝视角」能力：知道它做了什么、花了多少钱、出了什么问题。

### 8.1 结构化日志系统

**设计原则**：每轮 Agent Loop 执行产生一条完整的结构化日志，支持从日志还原完整对话过程。

**日志存储**：JSON Lines 格式 (.jsonl)，一行一条记录，方便流式写入和逐行解析。按会话分割，与 session 文件一一对应。

```
~/.zcli/logs/
  ├── sess_20260303_143022_317.jsonl   # 会话日志 (与 session 文件同名)
  ├── sess_20260303_160815_042.jsonl
  └── system_20260303.jsonl             # 系统日志 (非会话事件: 启动/配置变更/全局错误)
```

- **会话日志** `sess_YYYYMMDD_HHMMSS_SSS.jsonl`：每个 session 一个文件，查某次对话的日志 O(1) 直接打开
- **系统日志** `system_YYYYMMDD.jsonl`：按日分割，记录不属于任何会话的全局事件

**单条日志结构：**

```jsonc
{
  "timestamp": "2026-03-03T14:30:22.123Z",
  "session_id": "sess_20260303_143022_317",
  "turn": 3,                          // 第几轮对话
  "event": "llm_call",                // 事件类型 (见下表)
  "provider": "claude",
  "model": "claude-opus-4-6",
  "input_tokens": 1520,
  "output_tokens": 380,
  "duration_ms": 2340,
  "tools_called": ["file-read", "grep"],
  "error": null,                      // 错误信息 (如有)
  "metadata": {}                      // 扩展字段
}
```

**事件类型：**

| event | 说明 |
|-------|------|
| `session_start` | 会话创建 |
| `session_resume` | 会话恢复 |
| `user_input` | 用户输入 |
| `llm_call` | LLM 调用（含 token 数、耗时） |
| `tool_call` | 工具调用（含工具名、参数摘要、结果状态） |
| `tool_fallback` | 工具降级（L1→L2 触发记录） |
| `skill_load` | Skill 加载/卸载 |
| `mcp_connect` | MCP Server 连接/断开 |
| `error` | 异常事件 |
| `session_end` | 会话结束（含汇总统计） |

**日志还原对话**：直接打开对应的 `sess_YYYYMMDD_HHMMSS_SSS.jsonl` 文件，按 `turn` + `timestamp` 排序，即可重建完整的对话时间线（谁说了什么、调了哪些工具、花了多少 token）。无需从大文件中过滤 session_id。

### 8.2 Token 计量与计费

**计量**：每次 LLM 调用时自动记录 input_tokens / output_tokens / cache_read_tokens / cache_write_tokens，由 Provider 适配器负责从 API 响应中提取。

**计费数据存储**：SQLite，支持多维度聚合查询。

```sql
-- 用量记录表
CREATE TABLE usage_logs (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id    TEXT NOT NULL,
  timestamp     TEXT NOT NULL,
  provider      TEXT NOT NULL,      -- "anthropic" / "openai" / "google" / ...
  model         TEXT NOT NULL,      -- "claude-opus-4-6" / "gpt-4o" / ...
  input_tokens  INTEGER NOT NULL,
  output_tokens INTEGER NOT NULL,
  cache_read    INTEGER DEFAULT 0,
  cache_write   INTEGER DEFAULT 0,
  duration_ms   INTEGER,
  cost_amount   REAL,               -- 按当时计价规则算出的费用
  cost_currency TEXT DEFAULT 'USD',
  pricing_rule_id INTEGER           -- 关联使用的计价规则
);

-- 计价规则表 (核心: 支持同一模型不同供应商、不同时间段的差异化定价)
CREATE TABLE pricing_rules (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  provider        TEXT NOT NULL,      -- 供应商
  model_pattern   TEXT NOT NULL,      -- 模型匹配 (支持通配符, 如 "claude-*")
  input_price     REAL NOT NULL,      -- 输入价格 (每百万 token)
  output_price    REAL NOT NULL,      -- 输出价格 (每百万 token)
  cache_read_price  REAL DEFAULT 0,   -- 缓存读取价格
  cache_write_price REAL DEFAULT 0,   -- 缓存写入价格
  currency        TEXT DEFAULT 'USD',
  effective_from  TEXT NOT NULL,      -- 生效起始时间 (ISO 8601)
  effective_to    TEXT,               -- 失效时间 (NULL = 永久有效)
  source          TEXT,               -- 价格来源备注 ("官网 2026-03", "代理商 X")
  priority        INTEGER DEFAULT 0   -- 优先级 (高优先级规则覆盖低优先级)
);
CREATE INDEX idx_pricing_lookup ON pricing_rules(provider, model_pattern, effective_from);
```

**计价匹配逻辑**：

```
LLM 调用完成 → 提取 provider + model + timestamp
  │
  ▼
查询 pricing_rules:
  WHERE provider = ?
    AND model 匹配 model_pattern
    AND effective_from <= timestamp
    AND (effective_to IS NULL OR effective_to > timestamp)
  ORDER BY priority DESC, effective_from DESC
  LIMIT 1
  │
  ▼
计算费用: (input × input_price + output × output_price
         + cache_write × cache_write_price + cache_read × cache_read_price) / 1_000_000
  │
  ▼
写入 usage_logs.cost_amount
```

**典型场景：**

| 场景 | 解决方式 |
|------|----------|
| Anthropic 官方 Claude Opus 调价 | 新增一条 pricing_rule，effective_from 设为调价日期 |
| 通过代理商 X 调用 Claude，价格不同 | provider 设为 "proxy-x"，独立计价规则 |
| 同时用官方和代理商 | 在 LLM Provider 配置中标记 provider 来源，自动匹配对应计价规则 |
| 本地 Ollama 模型免费 | pricing_rule 的 input_price / output_price 均设为 0 |

|------|------|------|------|
| 日志 | JSON Lines (.jsonl) | Agent 执行事件流 | 按会话分割，O(1) 定位，追加写入高效 |

### 8.5 数据流全景

```
Agent Loop 每轮执行
     │
     ├──→ Logger ──→ ~/.zcli/logs/sess_*_.jsonl         (会话日志)
     │
     ├──→ TokenMeter ──→ 提取 token 数
     │       │
     │       ├──→ PricingEngine ──→ 查 SQLite pricing_rules ──→ 算费用
     │       │
     │       └──→ 写入 SQLite usage_logs                (计费入库)
     │
     └──→ Session Store ──→ ~/.zcli/sessions/*.json     (对话持久化)

WebUI Dashboard (读取)
     │
     ├──→ 读 SQLite ──→ 总览大盘 (聚合统计)
     ├──→ 读 SQLite ──→ 对话详情 (token 明细)
     ├──→ 读 sess_*.jsonl ──→ 日志浏览 (按会话直接定位)
     └──→ 读/写 SQLite ──→ 设置管理 (计价规则 CRUD)
```

## 9. 非功能性需求

| 项目 | 要求 |
|------|------|

| Dashboard | 本地 Web Server 启动 < 2s，页面首屏加载 < 1s |
