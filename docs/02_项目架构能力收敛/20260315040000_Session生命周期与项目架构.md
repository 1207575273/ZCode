> 日期: 2026-03-15
> 状态: 已落地
> 关联: Bridge Server 重构触发此调整，但影响面覆盖整个项目

## 一、Session 在项目中的角色

Session 是 cCli 中贯穿所有子系统的核心概念。它不只是"一次对话"，而是**多个子系统的关联枢纽**。

### 1.1 谁依赖 Session

```
sessionId
  │
  ├── SessionLogger    — JSONL 事件流写入，parentUuid 链条的根
  ├── TokenMeter       — bind(sessionId) 后才能写 usage_logs
  ├── SessionStore     — 文件路径 = projectSlug/sessionId.jsonl
  ├── Bridge Client    — register(cli, sessionId) 声明路由身份
  ├── Bridge Server    — 按 sessionId 隔离消息路由
  ├── Web 前端         — /session/:sessionId URL 路由
  ├── /resume 指令     — 恢复指定 session 的对话历史
  ├── /fork 指令       — 基于 session 内的事件分支
  ├── SubAgent         — 子 Agent 关联 parentSessionId
  └── 退出提示         — "Resume this session with: ccode --resume {sessionId}"
```

### 1.2 Session 的唯一性

- sessionId 格式：UUID v7（时间有序，`019cebb1-552a-727c-9a35-fbea0827b57c`）
- 生成方式：`sessionStore.create()` 内部调用 `randomUUID()`
- 存储路径：`~/.ccode/sessions/{projectSlug}/{timestamp}_{sessionId}.jsonl`
- 一个 CLI 进程在其生命周期内只有一个活跃 session（`/resume` 会切换）

## 二、生命周期演进

### 2.1 阶段一：惰性创建（初始设计）

```
时间线：
  CLI 启动 ──────── 用户输入 ──────── submit() ──────── ensureSession() ──── 对话开始
                                                              ↑
                                                        sessionId 在这里才诞生
```

**代码位置**：`useChat.ts` 的 `submit()` 方法内部

```typescript
const submit = useCallback((text: string) => {
  // ... 省略其他逻辑
  const sid = sessionLogger.ensureSession(currentProvider, currentModel)
  if (sid) tokenMeter.bind(sid, currentProvider, currentModel)
  sessionLogger.logUserMessage(text)
  // ...
})
```

**设计理由**：
- 启动不一定会对话（用户可能只看 `/help`、`/model`、`/usage`）
- 不浪费 JSONL 文件和磁盘空间
- 纯 CLI 模式下完全合理

**这个设计在纯 CLI 模式下运行了很长时间，没有任何问题。**

### 2.2 阶段二：Bridge 引入后暴露矛盾

Bridge Server 架构要求所有消息按 sessionId 路由。但 session 在首次对话前不存在：

```
时间线：
  CLI 启动 ── Bridge 启动 ── Web 连接 ── register(cli, ???) ── 路由失败
                                              ↑
                                        没有 sessionId，无法 register
```

**具体症状**：

| 时间点 | 期望 | 实际 |
|--------|------|------|
| CLI 启动 | sessionId 可用 | null |
| Bridge Client 连接 | register(cli, sessionId) | register(cli, '') → 路由不匹配 |
| Web 连接 | 收到 session_init + 历史 | 收到空响应 |
| 终端显示 Web URL | `http://localhost:9800/session/019cebb1` | `http://localhost:9800/session/` |
| 多终端隔离 | 各自独立 | 全部路由到空 session，消息串台 |

**根本矛盾**：惰性创建假设"session 是对话的附属品"，但 Bridge 架构中"session 是路由的身份凭证"。身份必须在连接建立时就存在。

### 2.3 阶段三：即时创建（当前方案）

```
时间线：
  CLI 启动 ── render() ── useChat mount ── ensureSession() ── sessionId 立即可用
                                                ↓
                                          Bridge Client register(cli, sessionId) ✓
                                          Web 连接 → session_init ✓
                                          URL 完整 ✓
```

**代码位置**：`useChat.ts` hook 内部新增的 `useEffect`

```typescript
// 预创建 session（不等首次 submit），让 Bridge Server 连接时就能拿到 sessionId
// ensureSession 幂等：后续 submit 调用不会重复创建
useEffect(() => {
  if (currentProvider && currentModel) {
    sessionLogger.ensureSession(currentProvider, currentModel)
  }
}, [])
```

**为什么放在 useChat 而不是其他地方**：
- useChat 持有 `currentProvider` 和 `currentModel`（ensureSession 需要）
- useChat 是 session 的"所有者"——submit、resume、fork 都在这里
- 放在 App.tsx 或 bootstrap 里会引入额外的参数传递

## 三、Session 完整生命周期（当前）

```
┌─ 创建 ─────────────────────────────────────────────────────────┐
│                                                                 │
│  useChat mount → ensureSession(provider, model)                 │
│    → sessionStore.create(cwd, provider, model)                  │
│      → 生成 UUID v7                                             │
│      → 写入 session_start 事件到 JSONL                          │
│      → 返回 sessionId                                           │
│    → sessionLogger.#sessionId = id                              │
│    → 后续 ensureSession 幂等返回                                │
│                                                                 │
├─ 绑定 ─────────────────────────────────────────────────────────┤
│                                                                 │
│  submit() → tokenMeter.bind(sessionId, provider, model)         │
│  Bridge Client → connectBridge(port, sessionId)                 │
│  /resume → sessionLogger.bind(existingSessionId, leafUuid)      │
│  /fork → sessionStore.create() 新 session + bind               │
│                                                                 │
├─ 运行中 ───────────────────────────────────────────────────────┤
│                                                                 │
│  每次 submit:                                                    │
│    sessionLogger.logUserMessage(text)          → JSONL           │
│    AgentLoop.run() → AgentEvent 流:                              │
│      sessionLogger.consume(event)              → JSONL           │
│      tokenMeter.consume(event)                 → SQLite          │
│      eventBus.emit(event)                      → Bridge Client   │
│    sessionLogger.logAssistantMessage(text)      → JSONL           │
│                                                                 │
│  Bridge 路由:                                                    │
│    CLI event → Bridge → 同 sessionId 的 Web                     │
│    Web input → Bridge → 同 sessionId 的 CLI                     │
│                                                                 │
├─ 切换 ─────────────────────────────────────────────────────────┤
│                                                                 │
│  /resume <otherSessionId>                                       │
│    → sessionLogger.bind(otherSessionId, leafUuid)               │
│    → Bridge Client 需要重新 register（当前未实现，P2）           │
│                                                                 │
│  /fork <messageId>                                              │
│    → sessionStore.create() 新 sessionId                         │
│    → sessionLogger.bind(newSessionId, messageId)                │
│                                                                 │
├─ 结束 ─────────────────────────────────────────────────────────┤
│                                                                 │
│  CLI 退出:                                                      │
│    sessionLogger.finalize()                                     │
│      → 写入 session_end 事件（汇总统计）                        │
│    closeDb()                                                    │
│    disconnectBridge()                                           │
│    打印 "Resume with: ccode --resume {sessionId}"                │
│                                                                 │
├─ 恢复 ─────────────────────────────────────────────────────────┤
│                                                                 │
│  ccode --resume <sessionId>                                      │
│    → sessionStore.loadMessages(sessionId) 读取 JSONL            │
│    → sessionLogger.bind(sessionId, leafEventUuid)               │
│    → UI 还原消息历史                                            │
│    → 继续对话（后续事件追加到同一 JSONL 文件）                  │
│                                                                 │
├─ 清理 ─────────────────────────────────────────────────────────┤
│                                                                 │
│  /gc 或 ccode gc                                                 │
│    → 删除超过 retention days 的 JSONL 文件                      │
│    → 删除超过 retention days 的 usage_logs 记录                 │
│    → VACUUM 回收磁盘空间                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 四、JSONL 事件链结构

每个 session 对应一个 JSONL 文件，事件通过 `parentUuid` 形成链条：

```
session_start (uuid: A, parentUuid: null)
  └── user (uuid: B, parentUuid: A)
       └── llm_call_start (uuid: C, parentUuid: B)
            └── llm_call_end (uuid: D, parentUuid: C)
                 └── assistant (uuid: E, parentUuid: D)
                      └── user (uuid: F, parentUuid: E)
                           └── llm_call_start (uuid: G, parentUuid: F)
                                └── tool_call_start (uuid: H, parentUuid: G)
                                     └── tool_call_end (uuid: I, parentUuid: H)
                                          └── llm_call_end (uuid: J, parentUuid: I)
                                               └── assistant (uuid: K, parentUuid: J)
                                                    └── session_end (uuid: L, parentUuid: K)
```

**分支支持**：`/fork` 从某个事件节点分出新链，多个事件可共享同一个 `parentUuid`，形成树形结构。

## 五、与其他子系统的接口

### 5.1 SessionLogger（观测层）

```typescript
class SessionLogger {
  ensureSession(provider, model): string  // 创建（幂等）
  bind(sessionId, leafUuid): void         // 绑定已有 session（resume/fork）
  get sessionId(): string | null          // 获取当前 sessionId
  consume(event: AgentEvent): void        // AgentEvent → SessionEvent → JSONL
  logUserMessage(text): void              // 记录用户消息
  logAssistantMessage(text, model): void  // 记录助手回复
  finalize(): void                        // 写入 session_end 汇总
}
```

### 5.2 TokenMeter（计量层）

```typescript
class TokenMeter {
  bind(sessionId, provider, model): void  // 绑定 session + 模型（计费匹配）
  consume(event: AgentEvent): void        // llm_usage → 写入 SQLite
  getSessionStats(): SessionCostStats     // 内存累计（StatusBar 用）
}
```

### 5.3 Bridge Client（网络层）

```typescript
function connectBridge(port, sessionId): void   // 连接 + register(cli, sessionId)
function disconnectBridge(): void               // 断开
```

### 5.4 SessionStore（持久化层）

```typescript
class SessionStore {
  create(cwd, provider, model): string             // 创建 JSONL + 写 session_start
  append(sessionId, event): void                   // 追加事件
  loadMessages(sessionId, leafUuid?): SessionSnapshot  // 读取消息历史
  list(options?): SessionSummary[]                 // 列出会话
  cleanup(retentionDays): void                     // 清理过期文件
}
```

## 六、设计原则与经验教训

### 6.1 惰性初始化不是银弹

**教训**：当一个资源从"可选附属品"变为"基础设施的身份凭证"时，惰性创建反而成为障碍。

**识别信号**：多个模块开始依赖某个资源的**存在性**（而非内容）时，就该改为即时初始化。

**在 cCli 中**：纯 CLI 模式下 session 是"对话后才需要"的。但 Bridge 架构中 session 是"连接路由的身份凭证"，必须在对话前就存在。

### 6.2 幂等是安全改造的前提

`ensureSession` 的幂等设计（`if (this.#sessionId) return`）使得"提前调用"完全安全。无论调用 1 次还是 100 次，行为一致。这让我们可以放心地把调用时机从 `submit()` 提前到 `useEffect(mount)`，不用担心副作用。

**原则**：任何可能被多个调用方在不同时机调用的初始化函数，都应该是幂等的。

### 6.3 Session 是全局单例资源

一个 CLI 进程在任意时刻只有一个活跃 sessionId。`/resume` 和 `/fork` 会切换它，但不会同时存在两个。这简化了很多设计（不需要 session 池、不需要并发控制）。

但 Bridge Server 维度是多 session 的（多个 CLI 进程各自有自己的 session）。**进程内单例 + 跨进程多实例**，这个区分很重要。

### 6.4 维护检查清单

当未来修改 session 相关逻辑时，确认以下模块是否受影响：

- [ ] `sessionLogger.ensureSession` — 创建时机
- [ ] `tokenMeter.bind` — 绑定时机
- [ ] `Bridge Client register` — sessionId 传递
- [ ] `Bridge Server 路由` — sessionId 匹配
- [ ] `Web 前端 URL` — `/session/:id` 路由
- [ ] `/resume` — `sessionLogger.bind` 后是否需要重新 register Bridge
- [ ] `/fork` — 新 sessionId 是否同步到 Bridge
- [ ] 退出提示 — `getCurrentSessionId()` 是否返回正确值
- [ ] `/gc` — 清理逻辑是否识别空 session（即时创建但未对话）
