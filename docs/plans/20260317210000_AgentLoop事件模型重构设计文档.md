> 状态: 设计完成，待实施

## 一、调研结论

### 1.1 各 LLM 的"调用结束"信号

| LLM | 字段名 | 可能值 | 获取方式 |
|-----|--------|--------|---------|
| **Anthropic Claude** | `stop_reason` | `end_turn` / `tool_use` / `max_tokens` / `stop_sequence` | `finalMessage().stop_reason` |
| **OpenAI** | `finish_reason` | `stop` / `tool_calls` / `length` / `content_filter` | `chunk.choices[0].finish_reason`（流式最后一个 chunk） |
| **OpenAI 兼容 (GLM/DeepSeek)** | `finish_reason` | 同 OpenAI | 同 OpenAI（LangChain 包装在 `response_metadata` 中） |

**关键发现**：

- **两套体系**名字不同但语义对齐：
  - Anthropic `end_turn` ≈ OpenAI `stop`（自然结束）
  - Anthropic `tool_use` ≈ OpenAI `tool_calls`（需要调工具）
  - Anthropic `max_tokens` ≈ OpenAI `length`（截断）
- **流式场景**：Anthropic 在 `message_delta` 事件中带 `stop_reason`，OpenAI 在最后一个 chunk 的 `finish_reason` 中
- **LangChain**：`response_metadata` 中有 `finish_reason`，但流式输出时不一定有（[GitHub Issue #33717](https://github.com/langchain-ai/langchain/issues/33717)）

### 1.2 当前项目问题（复查）

```
Anthropic SDK 返回:
  stream → message_delta(stop_reason: "end_turn") → message_stop
  finalMessage() → { stop_reason: "end_turn", usage: {...} }
                            ↓ 丢失！
  Provider.chat() → yield { type: 'done' }  ← 没有带 stopReason

LangChain 返回:
  stream → chunks → final.response_metadata → { finish_reason: "stop" }
                            ↓ 丢失！
  Provider.chat() → yield { type: 'done' }  ← 没有带 stopReason

AgentLoop:
  yield { type: 'llm_usage', ..., stopReason: 'end_turn' }  ← 硬编码！不是从 Provider 拿的
```

**三个断点**：
1. `StreamChunk` 类型没有 `stopReason` 字段
2. 两个 Provider 的 `done` chunk 没有带 `stopReason`
3. AgentLoop 硬编码了 `'end_turn'`

## 二、设计方案：统一拦截 + stopReason 透传

### 2.1 核心思路

**不在每个 Provider 里单独处理**，而是：

1. `StreamChunk` 加 `stopReason` 字段（可选）
2. 各 Provider 在 `done` chunk 中带上自己体系的 `stopReason`
3. AgentLoop 统一从 chunk 中取 `stopReason`，不硬编码
4. `llm_usage` 改名为 `llm_done`

### 2.2 统一 stopReason 映射

Provider 层不做统一映射——各自保留原始值，AgentLoop 层做标准化：

```typescript
/** 将各 Provider 的 stopReason 标准化为统一枚举 */
function normalizeStopReason(raw: string): string {
  switch (raw) {
    // Anthropic 体系
    case 'end_turn': return 'end_turn'
    case 'tool_use': return 'tool_use'
    case 'max_tokens': return 'max_tokens'
    case 'stop_sequence': return 'stop_sequence'
    // OpenAI 体系
    case 'stop': return 'end_turn'         // 对齐 Anthropic
    case 'tool_calls': return 'tool_use'   // 对齐 Anthropic
    case 'length': return 'max_tokens'     // 对齐 Anthropic
    case 'content_filter': return 'content_filter'
    // 其他
    case 'abort': return 'abort'
    default: return raw
  }
}
```

标准化后的值统一为 Anthropic 命名（因为 ZCli 的 JSONL 日志已有 `end_turn` 的历史数据）。

### 2.3 数据流变化

```
Before:
  Provider → StreamChunk(done, 无 stopReason)
    → AgentLoop → llm_usage(stopReason: 硬编码 'end_turn')
      → SessionLogger → llm_call_end(stopReason: 'end_turn')

After:
  Provider → StreamChunk(done, stopReason: 原始值)
    → AgentLoop → llm_done(stopReason: normalizeStopReason(原始值))
      → SessionLogger → llm_call_end(stopReason: 统一值)
      → TokenMeter → 同
      → Web UI → 可展示
```

## 三、改动清单

### 3.1 核心类型 (`src/core/types.ts`)

```diff
 export interface StreamChunk {
   type: 'text' | 'tool_call' | 'usage' | 'done' | 'error'
   text?: string
   toolCall?: ToolCallContent
   usage?: TokenUsage
   error?: string
+  /** LLM 调用结束原因（done 类型时有值） */
+  stopReason?: string
 }
```

### 3.2 Anthropic Provider (`src/providers/anthropic.ts`)

```diff
-     yield { type: 'done' }
+     yield { type: 'done', stopReason: finalMsg.stop_reason ?? 'end_turn' }
```

`finalMsg.stop_reason` 直接从 Anthropic SDK 的 `finalMessage()` 取。

### 3.3 OpenAI 兼容 Provider (`src/providers/openai-compat.ts`)

```diff
+     // 从 LangChain response_metadata 提取 finish_reason
+     const finishReason = (final as any).response_metadata?.finish_reason
+       ?? (final.additional_kwargs as any)?.finish_reason
+       ?? 'stop'

-     yield { type: 'done' }
+     yield { type: 'done', stopReason: finishReason }
```

LangChain 的 `finish_reason` 位置不稳定（[Issue #33717](https://github.com/langchain-ai/langchain/issues/33717)），做 fallback 链。

### 3.4 AgentEvent 类型 (`src/core/agent-loop.ts`)

```diff
- | { type: 'llm_usage'; inputTokens: number; outputTokens: number; cacheReadTokens: number; cacheWriteTokens: number; stopReason: string }
+ | { type: 'llm_done'; inputTokens: number; outputTokens: number; cacheReadTokens: number; cacheWriteTokens: number; stopReason: string }
```

### 3.5 AgentLoop `#callLLM` 方法

```diff
  for await (const chunk of this.#provider.chat(chatRequest)) {
    // ... 处理 text / tool_call / usage / error ...
+   if (chunk.type === 'done') {
+     doneStopReason = chunk.stopReason ?? 'end_turn'
+   }
  }

- yield { type: 'llm_usage', inputTokens, outputTokens, cacheReadTokens, cacheWriteTokens, stopReason: 'end_turn' }
+ yield { type: 'llm_done', inputTokens, outputTokens, cacheReadTokens, cacheWriteTokens, stopReason: normalizeStopReason(doneStopReason) }
```

### 3.6 AgentLoop `run` 方法 — `done` 加 `reason`

```diff
  if (llmResult.toolCalls.length === 0) {
-   yield { type: 'done' }
+   yield { type: 'done', reason: 'complete' }
    return
  }

  // 循环结束（maxTurns）
- yield { type: 'error', error: `超过最大轮次限制 (${maxTurns})` }
+ yield { type: 'done', reason: 'max_turns' }
```

中止时（catch abort）：
```diff
+ yield { type: 'done', reason: 'aborted' }
```

### 3.7 SessionLogger (`src/observability/session-logger.ts`)

```diff
- case 'llm_usage':
+ case 'llm_done':
    this.#appendEvent('llm_call_end', {
      inputTokens: event.inputTokens,
      // ... 其他字段不变
    })
```

JSONL 的 `llm_call_end` 类型名不变（向后兼容已有日志）。

### 3.8 TokenMeter (`src/observability/token-meter.ts`)

```diff
- if (event.type !== 'llm_usage') return
+ if (event.type !== 'llm_done') return
```

### 3.9 Web 前端 (`web/src/types.ts`)

```diff
- | { type: 'llm_usage'; inputTokens: number; outputTokens: number }
+ | { type: 'llm_done'; inputTokens: number; outputTokens: number; stopReason: string }
```

### 3.10 Bridge Client (`src/server/bridge/client.ts`)

事件过滤无需改动——`llm_done` 不在 `SKIP_TYPES` 中，会正常推送给 Web。

## 四、事件模型对比

### 4.1 Before

```
llm_start ────→ text* ────→ llm_usage(stopReason: 硬编码) ────→ [tool_start → tool_done]* ────→ done
                                 ↑ 名字不对                                                        ↑ 无 reason
```

### 4.2 After

```
llm_start ────→ text* ────→ llm_done(stopReason: 透传) ────→ [tool_start → tool_done]* ────→ done(reason)
                                 ↑ 对称命名                                                       ↑ complete / max_turns / aborted
```

### 4.3 生命周期对称性

```
LLM:   llm_start → text* → llm_done (含 token + stopReason)
                          → llm_error (失败时)

Tool:  tool_start → tool_done (含 durationMs + success + resultSummary)

Agent: run() start → [turn]* → done (含 reason)
```

## 五、LangChain finish_reason 获取的坑

### 5.1 问题

LangChain 的 `ChatOpenAI` 在流式模式下，`finish_reason` 的位置不统一（[Issue #33717](https://github.com/langchain-ai/langchain/issues/33717)）：

- 非流式：`response_metadata.finish_reason` ✅
- 流式（聚合后）：可能在 `response_metadata` 里，也可能不在

### 5.2 应对

做 fallback 链：

```typescript
const finishReason = final.response_metadata?.finish_reason  // 优先
  ?? final.additional_kwargs?.finish_reason                    // 备选
  ?? (final.tool_calls?.length > 0 ? 'tool_calls' : 'stop')  // 推断
```

如果工具调用列表非空但 `finish_reason` 拿不到，推断为 `'tool_calls'`。否则默认 `'stop'`。

### 5.3 为什么不在 Provider 层做标准化

Provider 保留原始值（`'stop'` / `'tool_calls'` / `'end_turn'` / `'tool_use'`），AgentLoop 层统一标准化。原因：

1. **调试友好**——日志里能看到 Provider 返回的原始值
2. **不丢信息**——`stop_sequence` 等 Provider 特有值不会被丢弃
3. **单一转换点**——标准化逻辑只在 AgentLoop 一处，不散落在各 Provider

## 六、测试要点

| 测试 | 验证内容 |
|------|---------|
| AgentLoop 正常结束 | `llm_done` 事件的 `stopReason` = `'end_turn'`（不再硬编码） |
| AgentLoop 中止 | `llm_done` 的 `stopReason` = `'abort'` + `done` 的 `reason` = `'aborted'` |
| AgentLoop maxTurns | `done` 的 `reason` = `'max_turns'`（不再是 error） |
| TokenMeter | `llm_done` 事件能正确被消费 |
| SessionLogger | `llm_done` 映射到 `llm_call_end`（JSONL 类型不变） |
| Web 端 | `llm_done` 事件能到达（Bridge 不过滤） |
| OpenAI Provider | `stopReason` = `'stop'`（原始值）→ AgentLoop 标准化为 `'end_turn'` |
| Anthropic Provider | `stopReason` = `'end_turn'`（原始值）→ 不变 |
