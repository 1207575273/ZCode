## 一、背景与目标

### 1.1 问题

对比 Claude Code 的工具交互 UI（图 05、06）和当前 ZCli（图 07），存在显著差距：

**Claude Code 做到了**：
- `Bash(cd "..." && mvn compile)` — 工具头一行传达"做了什么"
- `Write(pom.xml) Wrote 46 lines` — 操作摘要一目了然
- `Update(src/tools/edit-file.ts) +1 -1` — Diff 预览红绿高亮
- `... +36 lines (ctrl+o to expand)` — 长内容折叠
- `(timeout 2m)` — 长命令有超时反馈

**ZCli 当前**：
- `✓ bash (53ms)` — 只有工具名和耗时，用户完全不知道执行了什么
- 无文件操作 diff 预览
- 无 write 内容预览
- AI 文本一大段不分段的文字墙
- Node.js 内部警告泄露到用户界面

### 1.2 目标

让用户在不展开详情的情况下，**一眼看懂每个工具调用做了什么、改了什么、结果如何**。同时保持管道模式的干净输出。

### 1.3 设计原则

- **信息密度优先**：一行工具状态 = 动作 + 对象 + 结果摘要 + 耗时
- **渐进披露**：默认一行摘要，可展开看完整内容（Diff / 文件内容 / 命令输出）
- **双模式适配**：REPL 丰富渲染，Pipe 保持纯净（可选 stderr 进度）
- **最小侵入**：尽量复用已有的事件字段（`args`、`resultSummary`），避免大改核心链路

#### 3.1.3 ToolStatusLine 渲染逻辑重写

核心：根据 `toolName` + `args` 提取参数摘要，结合 `resultSummary` 渲染一行信息。

```typescript
// 参数摘要提取策略（纯 UI 层函数，不改工具代码）
function buildArgsSummary(toolName: string, args?: Record<string, unknown>): string {
  if (!args) return ''
  switch (toolName) {
    case 'bash':
      return truncate(String(args['command'] ?? ''), 60)
    case 'read_file':
    case 'write_file':
    case 'edit_file':
      return String(args['path'] ?? '')
    case 'grep':
      return `pattern: "${args['pattern']}", path: ${args['path'] ?? '.'}`
    case 'glob':
      return String(args['pattern'] ?? '')
    case 'dispatch_agent':
      return String(args['description'] ?? '')
    default:
      return ''
  }
}
```

渲染格式：

```
Running:   ⠋ {Label} {argsSummary}...
Done:      ✓ {Name}({argsSummary})  {resultSummary}  {duration}
Error:     ✗ {Name}({argsSummary})  {errorSummary}   {duration}
```

耗时格式化：`< 1s → "120ms"`，`1~60s → "3.2s"`，`> 60s → "2m 3s"`。

#### 3.1.4 各工具填充 resultSummary（可选优化）

当前 agent-loop 在 tool_done 中已有 `resultSummary` 字段。检查当前填充逻辑：

```typescript
// src/core/agent-loop.ts — #executeTool() 末尾

  ? result.output.slice(0, 200) + '…'
  : result.output
```

如果当前已有截断逻辑则 P0 不需要改工具。如果没有，可在 agent-loop 层统一生成：
- bash → 输出前 3 行
- read_file → `{lineCount} lines`
- write_file → `{lineCount} lines`
- grep → `{matchCount} matches`
- glob → `{fileCount} files`
- edit_file → `+{added} -{removed}`

**优先在 UI 层从 `resultSummary` 文本中解析，避免改工具层。**

```typescript
// src/ui/format-utils.ts（新建或放入已有 utils）
export function formatDuration(ms: number): string {
  if (ms < 1000) return `${ms}ms`
  if (ms < 60_000) return `${(ms / 1000).toFixed(1)}s`
  const min = Math.floor(ms / 60_000)
  const sec = Math.round((ms % 60_000) / 1000)
  return sec > 0 ? `${min}m ${sec}s` : `${min}m`
}
```

## 五、风险与注意事项

### 5.1 性能

- ToolStatusLine 在工具密集场景（SubAgent 内部多工具）会频繁重渲染。已有的 `<Static>` 固化策略可保护已完成消息，但工具事件在执行中仍在动态区域。
- P1 的 Diff 预览需要控制渲染行数，长 diff 必须折叠。

### 5.2 MCP 工具兼容性

- MCP 工具的 args 结构由外部 server 定义，`buildArgsSummary` 无法预知所有字段。
- 策略：MCP 工具走 fallback 分支，显示 `toolName(JSON.stringify(args).slice(0,40))` 截断。

### 5.3 管道模式向后兼容

- `--verbose` 是新增参数，默认关闭，不影响现有管道行为。
- `--json` 模式是否需要扩展工具事件到 JSON 输出？暂不做，保持现有结构。后续按需扩展。

### 5.4 ToolResult.meta 的类型安全

- `meta` 是可选字段，不改动已有工具的返回值（逐步填充）。
- 联合类型用 `type` 字段区分，UI 层用 switch 安全消费。
