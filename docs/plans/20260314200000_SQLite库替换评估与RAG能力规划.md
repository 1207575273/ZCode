# SQLite 库替换评估与 RAG 能力规划

> 日期: 2026-03-14
> 状态: 待实施
> 关联文档: `20260314190000_ECMAScript规范与Bun兼容性审查报告.md`

## 一、问题背景

### 1.1 当前状况

项目使用 `better-sqlite3` (v12.6.2) 作为 SQLite 驱动，用于 token 计量、计价规则、数据清理等持久化场景。

**`better-sqlite3` 是切换 Bun.js 的唯一硬阻塞项**（详见兼容性审查报告 P0 级别），原因是原生 C++ addon 的 N-API ABI 版本不匹配（Node: MODULE_VERSION 131, Bun: 127）。

### 1.2 新增需求

除了解决 Bun 兼容性，还希望利用 SQLite 做小型本地 RAG（向量搜索 + 全文检索），用于：

- 对话历史语义搜索
- 代码片段 / 指令文件的语义匹配
- 项目级知识库（local-first，不依赖云服务）

## 二、当前 better-sqlite3 使用分析

### 2.1 涉及文件（仅 4 个）

| 文件 | 用途 | API 调用 |
|------|------|---------|
| `src/persistence/db.ts` | 建库/建表/种子数据 | `new Database()`, `.pragma()`, `.exec()`, `.prepare().run()`, `.transaction()`, `.close()` |
| `src/observability/token-meter.ts` | token 写入/查询 | `.prepare()` → `.run()/.get()/.all()` (4 条预编译语句) |
| `src/core/cleanup-service.ts` | 过期数据清理 | `.prepare().run()/.get()`, `.exec('VACUUM')` |
| `tests/unit/token-meter.test.ts` | 单元测试 | `createDb()` + 查询验证 |
| `tests/unit/core/cleanup-service.test.ts` | 单元测试 | `createDb()` + 插入/查询验证 |

### 2.2 API 使用面

```
构造:     new Database(path)
配置:     .pragma('journal_mode = WAL'), .pragma('foreign_keys = ON')
执行:     .exec(multiStatementSQL)
预编译:   .prepare(sql) → Statement
读写:     Statement.run(...params) → { changes }
         Statement.get(...params) → row | undefined
         Statement.all(...params) → row[]
事务:     .transaction(fn) → wrappedFn, wrappedFn()
关闭:     .close()
```

### 2.3 数据规模

- 3 张表: `usage_logs`, `pricing_rules`, `schema_comments`
- 写入频率: 每次 LLM 调用写 1 行 usage_log（低频）
- 数据量: 单用户本地数据，千级~万级行量级

## 三、替代方案对比

### 3.1 候选方案

| 方案 | 包名 | Node | Bun | Deno | RAG 能力 | API 兼容 |
|------|------|------|-----|------|---------|---------|
| **A. libsql** | `libsql` | ✅ | ✅ | ✅ | ✅ 内置向量搜索 + FTS5 | better-sqlite3 兼容 |
| B. better-sqlite3 + 适配层 | `better-sqlite3` + `bun:sqlite` | ✅ | ✅(适配) | ❌ | ❌ 需额外扩展 | 需维护两套 |
| C. sql.js | `sql.js` | ✅ | ✅ | ✅ | ❌ | 不同 API |
| D. node:sqlite | 内置 | ✅(22+) | ❌ | ❌ | ❌ | 不同 API |

### 3.2 推荐方案：A. libsql

**理由：**

1. **一库通吃** — 同一个 `libsql` 包在 Node / Bun / Deno 上直接运行，无需适配层
2. **API 兼容** — 设计目标就是 better-sqlite3 的 drop-in replacement，迁移成本最低
3. **RAG 就绪** — 内置向量列类型 + DiskANN 索引 + FTS5，不需要额外扩展
4. **背景可靠** — Turso (YC W23) 出品，libSQL 是 SQLite 的开源 fork，活跃维护
5. **纯本地** — 不需要 Turso 云服务，嵌入式运行，适合 CLI 工具场景

## 四、libsql 详细评估

### 4.1 API 对照表

| better-sqlite3 | libsql | 差异 |
|----------------|--------|------|
| `new Database(path)` | `new Database(path)` | ✅ 相同 |
| `db.pragma('key = val')` | `db.pragma('key = val')` | ✅ 相同 |
| `db.exec(sql)` | `db.exec(sql)` | ✅ 相同 |
| `db.prepare(sql)` | `db.prepare(sql)` | ✅ 相同 |
| `stmt.run(...params)` | `stmt.run(...params)` | ✅ 返回 `{ changes }` |
| `stmt.get(...params)` | `stmt.get(...params)` | ✅ 相同 |
| `stmt.all(...params)` | `stmt.all(...params)` | ✅ 相同 |
| `db.transaction(fn)` | `db.transaction(fn)` | ✅ 相同 |
| `db.close()` | `db.close()` | ✅ 相同 |

**结论：我们用到的全部 API 都兼容，理论上只需改 import 语句。**

### 4.2 迁移改动预估

```diff
# db.ts (1 行改动)
- import Database from 'better-sqlite3'
- import type { Database as DatabaseType } from 'better-sqlite3'
+ import Database from 'libsql'
+ type DatabaseType = InstanceType<typeof Database>

# token-meter.ts (1 行改动)
- import type { Database as DatabaseType, Statement } from 'better-sqlite3'
+ import type Database from 'libsql'
+ type DatabaseType = InstanceType<typeof Database>
+ type Statement = ReturnType<DatabaseType['prepare']>

# cleanup-service.ts (1 行改动)
- import type { Database } from 'better-sqlite3'
+ import type DatabaseImport from 'libsql'
+ type Database = InstanceType<typeof DatabaseImport>

# package.json (1 行改动)
- "better-sqlite3": "^12.6.2"
+ "libsql": "^0.5.0"

# 测试文件 (同理改 import)
```

**总改动量: ~10 行 import 语句，零业务逻辑变更。**

### 4.3 风险评估

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|---------|
| API 细微差异 | 低 | 官方目标是 drop-in，但边缘情况可能有差异 | 跑完现有测试验证 |
| 性能差异 | 低 | libsql 底层也是 SQLite，性能在同一量级 | 我们是低频 IO，不敏感 |
| 类型定义差异 | 低 | TypeScript 类型可能需要小幅调整 | 按编译错误逐个修复 |
| 包体积 | 低 | libsql 包含预编译二进制，与 better-sqlite3 类似 | CLI 工具可接受 |
| 长期维护 | 低 | Turso 有商业模式支撑，libSQL 社区活跃 | 关注版本更新 |

## 五、RAG 能力规划

### 5.1 libsql 内置能力

#### 向量搜索

```sql
-- 创建带向量列的表
CREATE TABLE documents (
  id INTEGER PRIMARY KEY,
  content TEXT NOT NULL,
  embedding F32_BLOB(768)        -- 768 维浮点向量
);

-- 创建向量索引（DiskANN 算法）
CREATE INDEX idx_docs_embedding ON documents(
  libsql_vector_idx(embedding)
);

-- 向量相似度搜索（Top-K 最近邻）
SELECT id, content
FROM vector_top_k('idx_docs_embedding', :query_vector, 10)
JOIN documents ON documents.rowid = id;
```

#### 全文搜索（FTS5）

```sql
-- 创建全文索引
CREATE VIRTUAL TABLE documents_fts USING fts5(content);

-- 关键词搜索
SELECT * FROM documents_fts WHERE documents_fts MATCH '搜索关键词';
```

#### 混合搜索（向量 + 全文，RRF 融合）

```sql
-- 向量搜索结果
WITH vec_results AS (
  SELECT id, row_number() OVER () AS rank_pos
  FROM vector_top_k('idx_docs_embedding', :query_vec, 20)
),
-- 全文搜索结果
fts_results AS (
  SELECT rowid AS id, row_number() OVER (ORDER BY rank) AS rank_pos
  FROM documents_fts WHERE documents_fts MATCH :query_text
  LIMIT 20
)
-- Reciprocal Rank Fusion
SELECT id,
  COALESCE(1.0 / (60 + v.rank_pos), 0) +
  COALESCE(1.0 / (60 + f.rank_pos), 0) AS rrf_score
FROM vec_results v
FULL OUTER JOIN fts_results f USING (id)
ORDER BY rrf_score DESC
LIMIT 10;
```

### 5.2 cCli RAG 应用场景

| 场景 | 数据源 | 检索方式 | 优先级 |
|------|--------|---------|--------|
| **对话历史语义搜索** | session JSONL → 向量化 | 向量 + FTS5 混合 | P1 |
| **代码片段索引** | 项目文件 → 分块向量化 | 向量搜索 | P2 |
| **指令/Skill 语义匹配** | CCODE.md + skills → 向量化 | 向量搜索 | P2 |
| **命令行历史** | 用户输入记录 | FTS5 关键词 | P3 |

### 5.3 Embedding 策略

CLI 工具需要本地化的 embedding 方案：

| 方案 | 维度 | 速度 | 依赖 | 推荐 |
|------|------|------|------|------|
| **Ollama 本地模型** (nomic-embed-text) | 768 | ~10ms/chunk | Ollama 运行中 | ✅ 有 Ollama 时 |
| **transformers.js** (ONNX Runtime) | 384 | ~50ms/chunk | 纯 JS，无外部依赖 | ✅ 通用方案 |
| 远程 API (OpenAI/Anthropic) | 1536/1024 | ~100ms/chunk | 网络 + API Key | ⚠️ 备选 |

建议：优先检测本地 Ollama，降级到 transformers.js，远程 API 作为可选。

## 七、决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| SQLite 库 | libsql 替代 better-sqlite3 | 一库兼容 Node/Bun/Deno，API drop-in，内置 RAG 能力 |
| 不做适配层 | 直接替换 | API 完全兼容，适配层是过度工程 |
| RAG 向量存储 | libsql 内置向量列 | 无需额外扩展，DiskANN 索引开箱即用 |
| Embedding | Ollama 优先 + transformers.js 降级 | CLI 工具优先本地化，不强制依赖网络 |
| 检索策略 | 向量 + FTS5 混合（RRF） | 语义理解 + 关键词精确匹配互补 |
