## 五、阻塞项分级

### 🔴 P0 硬阻塞（必须解决）

**`better-sqlite3` → 需要替换为 `bun:sqlite`**

- 影响文件：`src/persistence/db.ts` 及所有 SQLite 读写逻辑
- 原因：原生 C++ addon，Bun N-API ABI 版本不匹配
- 方案：抽象 `DatabaseAdapter` 接口，运行时检测环境选择实现
  - Node 环境：`better-sqlite3`
  - Bun 环境：`bun:sqlite`（内置，API 类似，性能更好 3-6x）
- 复杂度：中（API 风格相似，但细节有差异）

### 🟡 P1 软阻塞（影响开发体验，不阻塞运行）

| 问题 | 影响 | 方案 |
|------|------|------|
| `ink` yoga.wasm 打包问题 | `bun build --compile` 失败 | 仅用 Bun 做运行时，保留 tsup 打包 |
| `@vitest/coverage-v8` 不可用 | 无法收集覆盖率 | 改用 `@vitest/coverage-istanbul` 或 `bun:test` |
| `vitest` v4 打包冲突 | 边缘场景测试异常 | `bun run vitest`（不加 `--bun`）或迁移 `bun:test` |

### 🟢 P2 无需处理

| 问题 | 说明 |
|------|------|
| `fs.watch` recursive Linux 兼容性 | 已有 try-catch 兜底，低风险 |
| tsup shebang `#!/usr/bin/env node` | Bun 也识别 node shebang |
| `moduleResolution: "bundler"` | Bun 原生支持 tsconfig paths |

  └── 抽象 DatabaseAdapter 接口

  └── 运行时自动选择

Phase 2: 切换运行时（最小改动）
  └── dev 脚本：tsx bin/ccli.ts → bun bin/ccli.ts
  └── 移除 tsx 依赖

  └── 保留 tsup 做生产打包（规避 ink wasm 问题）

项目代码质量优秀，ESM 规范 100% 合规，切 Bun 的**唯一真正阻塞项是 `better-sqlite3`**。写一个 SQLite 适配层后即可无缝切换运行时，预计首帧渲染从 ~1100ms 降至 ~400-500ms。
