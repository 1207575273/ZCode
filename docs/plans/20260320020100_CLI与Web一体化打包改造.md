# CLI 与 Web 一体化打包改造

> 日期: 2026-03-20
> 状态: 已完成

---

## 一、改造目标

将 CLI 主程序和 Web Dashboard 打包为一体，发布到 npm 后用户安装即拥有完整能力。
Web 是**隐藏能力**——不在 README 或 help 中主动宣传，但知道的人可以通过 `--web` 参数唤醒。

---

## 二、改造前的问题

```
改造前的 build 产物：

cCli/
├── dist/bin/ccli.js        ← tsup 打包的 CLI（build 命令产出）
└── web/dist/               ← Vite 打包的 Web（build:web 命令产出，独立目录）
    ├── index.html
    └── assets/...

问题：
1. CLI 和 Web 产物分散在两个目录
2. npm publish 只包含 dist/bin/，Web 产物不在包内
3. 用户安装后执行 ccode --web，Bridge Server 找不到 web/dist/，Web 无法使用
```

---

## 三、改造方案

### 3.1 统一输出到 dist/

```
改造后的 build:all 产物：

cCli/dist/
├── bin/
│   ├── ccli.js           ← CLI 主程序（tsup 打包，378 KB）
│   └── ccli.js.map       ← sourcemap
└── web/                   ← Web Dashboard（Vite 打包）
    ├── index.html         ← SPA 入口
    └── assets/
        ├── index-*.js      28 KB   ← 首屏核心
        ├── index-*.css     40 KB   ← 样式
        ├── vendor-react-*  179 KB  ← React（长缓存）
        ├── vendor-markdown-* 336 KB ← Markdown 渲染（按需）
        ├── vendor-charts-*  373 KB ← recharts（懒加载）
        ├── OverviewPage-*    9 KB  ← 懒加载页面
        ├── SettingsPage-*   24 KB
        ├── ConversationsPage-* 6 KB
        └── LogsPage-*       1 KB
```

### 3.2 改动文件

| 文件 | 改动 | 说明 |
|------|------|------|
| `web/vite.config.ts` | `outDir: '../dist/web'` | Web 构建产物输出到主 dist 下 |
| `src/server/bridge/server.ts` | distDir 路径解析改为相对 `ccli.js` 位置 | 生产模式从 `dist/bin/` 向上找 `dist/web/` |
| `.npmignore` | `/web/` 排除源码，`dist/web/` 保留 | Web 源码不发布，构建产物发布 |

### 3.3 Bridge Server 路径解析逻辑

```typescript
// ccli.js 运行时位于 dist/bin/ccli.js
// Web 产物位于 dist/web/

const scriptDir = import.meta.dirname   // → dist/bin/

// 优先级：
// 1. 打包产物路径：dist/bin/ → ../web/ = dist/web/
// 2. 回退：cwd/web/dist/（兼容未打包的开发环境）

const distDir = existsSync(join(scriptDir, '../web/index.html'))
  ? join(scriptDir, '../web')         // 打包产物
  : join(process.cwd(), 'web/dist')   // 回退
```

## 五、如何唤醒 Web 能力

```bash
# 启动 CLI + Vite Dev Server + Bridge Server

# 浏览器打开
# http://localhost:9800/session/<sessionId>
# http://localhost:9800/overview
```

dev 模式下 Bridge Server 反向代理 Vite（端口 5173），支持 HMR 热更新。

```bash
cd cCli

# 1. 先构建

# 2. 用 Node 直接跑构建产物
```

## 七、设计决策

### 为什么 Web 是隐藏能力？

1. **CLI 是核心价值**——大部分用户只需要终端交互
2. **Web 是增强层**——管理面板、历史回放、设置管理是高级功能
3. **不增加认知负担**——新用户看到的就是一个 CLI 工具，不会困惑"这到底是 CLI 还是 Web 应用"
4. **包体积可控**——Web 只增加 ~300 KB 压缩体积，对 npm 包来说可接受

### 为什么 Web 打包用 Vite 而不是和 CLI 一样用 tsup/esbuild？

CLI 用 tsup（底层 esbuild）打包只要 **0.1 秒**，但 Web 用 Vite（底层 Rollup）要 **12 秒**。
不能换的原因是 Web SPA 有 CLI 不需要的关键需求：

| 能力 | CLI (tsup/esbuild) | Web (Vite/Rollup) |
|------|--------------------|--------------------|
| 代码分割 (React.lazy) | 不需要 | **必须** — 路由懒加载 |
| HTML 入口生成 | 不需要 | **必须** — index.html + script 注入 |
| CSS 处理 (Tailwind/PostCSS) | 不需要 | **必须** |
| Tree-shaking 精度 | 低要求 | **高要求** — recharts 等大依赖 |

esbuild 不支持 Web 需要的 code splitting + HTML 生成 + Tailwind。强行用会丢失懒加载（回到 956KB 单文件）。

**12 秒的耗时分解**：

```
esbuild transform 1164 模块:   ~2 秒   ← Go 写的，已经很快
rollup chunk 生成 + tree-shake: ~6 秒   ← JS 写的，瓶颈（recharts 500+ 模块）
esbuild minify:                ~2 秒   ← Go 写的，已经很快
进程启动 + 文件 IO:            ~2 秒
```

**已做的优化**（19.6s → 12s）：
- build 脚本去掉 `tsc` 类型检查（省 ~9 秒，类型检查移到 `build:check`）
- `reportCompressedSize: false`（跳过 gzip 体积计算）
- `sourcemap: false`（生产包不需要 sourcemap）
- `cssMinify: 'esbuild'`（esbuild 压缩 CSS）

**后续可能的优化方向**：
- Vite 升级到支持 Rolldown（Rust 重写的 Rollup）后可大幅提速
- 替换 recharts 为轻量图表库（砍掉 500+ 模块，预计 build 降到 6-7 秒）

### 生产模式 CLI 注册 Bridge 的时序问题

**问题**：`node dist/bin/ccli.js --web` 后 Web 页面显示"CLI 已离线"。

**根因**：CLI 启动 Bridge Server 后，用 `setTimeout(100ms)` 等待 sessionId 创建再注册。
但生产模式下 bootstrapAll（Skills 发现 + 文件索引 + Hooks）耗时可能超过 100ms，
导致 `getCurrentSessionId()` 返回 null，`connectBridge` 不执行，CLI 永远不注册到 Bridge。

**修复**：改为轮询等待（100ms 间隔，最多 50 次 = 5 秒上限），sessionId 就绪后立即连接。

```typescript
// 修复前：固定 100ms 延迟（生产模式不够）
setTimeout(() => {
  const sid = getCurrentSessionId()
  if (sid) connectBridge(9800, sid)  // sid 可能为 null → 不连接
}, 100)

// 修复后：轮询等待 sessionId 创建完成
const poll = () => {
  const sid = getCurrentSessionId()
  if (sid) connectBridge(9800, sid)
  else if (++attempts < 50) setTimeout(poll, 100)
}
setTimeout(poll, 100)
```

### 为什么不拆成两个 npm 包？

1. **用户体验**——一个命令安装，一个参数唤醒，零配置
2. **版本一致性**——CLI 和 Web 同版本发布，不会出现兼容性问题
3. **Bridge Server 内嵌**——WebSocket 通信依赖 CLI 进程，分包会增加复杂度
