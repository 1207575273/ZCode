# 工具结果完整保存与 Web 展示

> 日期: 2026-03-24
> 状态: 已完成

---

## 一、问题

### 问题 1：工具输出被截断，完整结果永久丢失

工具执行结果在 `agent-loop.ts` / `parallel-executor.ts` 中被截断为 200 字符的 `resultSummary`，
之后所有环节（JSONL 持久化、Web API、Web 渲染）都只能拿到摘要，完整结果**永久丢失**。

CLI 空间有限，200 字符摘要够用。但 Web Dashboard 有充足空间，应展示完整工具输出。

### 问题 2：部分工具的 output 本身就不含实际内容

即使解决了截断问题，`write_file`、`edit_file` 等工具的 `result.output` 也**只是一句摘要**：

```typescript
// write_file 的 output
output: `已写入 ${path}（${content.length} 字符）`

// edit_file 的 output
output: `已更新 ${path}（${count} 处替换）`
```

实际写入/编辑的内容在工具的 **args（输入参数）** 里，而非 output（执行结果）里。
所以仅仅保存完整 output 不够，Web 还需要从 args 中提取详情。

---

## 二、数据流分析

### 改动前

```
工具执行 → result.output（完整，可能几千~几万字符）
     ↓
agent-loop / parallel-executor
     ↓  truncate(output, 200) → resultSummary
     ↓  result.output 丢弃 ✗
     ↓
AgentEvent (tool_done) → 只有 resultSummary
     ↓
SessionLogger → JSONL (tool_call_end) → 只存 resultSummary
     ↓
Web API /api/conversations/:id → 只返回 resultSummary
     ↓
Web ToolStatus 组件 → 只能渲染 200 字符
```

### 改动后

```
工具执行 → result.output（完整）
     ↓
agent-loop / parallel-executor
     ├─ resultSummary = truncate(output, 200)        ← CLI 用，不变
     └─ resultFull = truncate(output, 100_000)        ← 新增，Web 用
     ↓
AgentEvent (tool_done) → { resultSummary, resultFull }
     ↓
SessionLogger → JSONL (tool_call_end) → 同时写入 resultSummary + resultFull
     ↓
loadMessages → SnapshotToolEvent → { resultSummary, resultFull }
     ↓
Web API → 自动带出（直接返回 snapshot）
     ↓
Web ToolStatus → 组合展示：args 详情（输入）+ resultFull（输出）
```

---

## 三、设计方案

### 双字段策略：resultSummary + resultFull

| 字段 | 上限 | 用途 | 消费者 |
|------|------|------|--------|
| `resultSummary` | 200 字符 | CLI 展示（空间有限） | CLI ToolStatusLine、LLM 上下文 |
| `resultFull` | 100,000 字符 | Web 完整展示 + JSONL 归档 | Web ToolStatus、历史回放 |

不改工具本身的 `result.output`（避免影响 LLM 上下文），只在事件层分叉。

### resultFull 上限说明

> **上限：100,000 字符（约 100KB）**
>
> 选择依据：
> - 覆盖 99%+ 的工具输出场景（bash 命令、文件读取、grep 搜索等）
> - 单条 JSONL 行增大可控（极端情况 ~100KB/条，典型场景 <10KB/条）
> - 长期运行的会话 JSONL 文件增长仍可接受
>
> 超过 100,000 字符的输出仍会被截断，尾部追加 `\n... (truncated, total N chars)`。
> 如果未来需要，可以考虑大结果外置存储（单独文件 + JSONL 里存路径引用），但当前阶段不需要。

### Web 端 args 详情提取

对于 output 只有摘要的工具，Web ToolStatus 从 **args** 中提取实际内容展示：

| 工具 | output 内容 | args 中的详情 | Web 展示 |
|------|-------------|---------------|----------|
| `write_file` | "已写入 path（N 字符）" | `args.content`（完整文件内容） | 文件路径 + 完整写入内容 + 执行结果 |
| `edit_file` | "已更新 path（N 处替换）" | `args.old_string` + `args.new_string` | 文件路径 + old/new 对比 + 执行结果 |
| `bash` | 命令输出（完整） | `args.command` | 完整命令 + 完整输出 |
| 其他工具 | 各自的输出 | — | resultFull 完整输出 |

**关键洞察**：args 一直都保存在 JSONL 里（`tool_call_start` 事件），
所以这个策略对**旧数据也生效**——即使没有 `resultFull`，
Web 展开 write_file 时也能看到完整写入内容。

### 第 1 层：事件模型（核心层）

无需改动。`/api/conversations/:id` 直接返回 SessionSnapshot，新增字段自动带出。

### 第 5 层：Web 前端

| 文件 | 改动 |
|------|------|
| `web/src/types.ts` | `ToolEvent` 接口加 `resultFull` |

### 不动的部分

| 部分 | 原因 |
|------|------|
| CLI `ToolStatusLine.tsx` | CLI 空间有限，继续用 `resultSummary`（200 字符 + 4 行预览） |
| 工具实现（write_file 等） | 不改 `result.output`，避免影响 LLM 上下文长度 |
| `resultSummary` 逻辑 | 保持 200 字符截断，完全向后兼容 |

## 五、向后兼容

- 旧 JSONL 文件没有 `resultFull` 字段 → Web 渲染 fallback 到 `resultSummary`
- **args 详情提取对旧数据也生效**（args 一直都保存在 JSONL 里）→ 旧数据展开 write_file 也能看到完整内容
- 新 JSONL 文件同时包含 `resultSummary`（CLI 用）和 `resultFull`（Web 用）
- `resultFull` 是可选字段（`?: string`），不破坏任何现有接口

## 七、经验总结

### 1. output ≠ 完整内容

最初以为"保存完整的 result.output 就够了"，实际发现 `write_file` 等工具的 output
只是一句摘要（"已写入 xxx"），真正的内容在 args 里。

**教训**：不要假设 output 包含你想要的一切。需要区分：
- **args**（工具输入）= 用户/AI 提供的数据（文件内容、命令、搜索模式等）
- **output**（工具输出）= 工具执行后的反馈（可能只是状态消息）

对于 Web 完整展示，两者都需要。

### 2. 双字段策略比改工具更干净

考虑过让 write_file 把完整内容放到 output 里，但这会：
- 膨胀 LLM 上下文（工具结果会回传给模型）
- 改变所有工具的行为约定
- 影响 CLI 的渲染

双字段（`resultSummary` + `resultFull`）只在**事件层分叉**，不影响工具实现和 LLM 交互。

### 3. args 是天然的"完整内容"来源

args 一直都保存在 JSONL 里（`tool_call_start` 事件），只是之前没有在 Web 端利用。
通过 `extractArgsDetail()` 函数从 args 提取关键字段，
这个策略对旧数据也生效——不需要重新执行工具。

### 4. 分层改动，最小影响面

改动贯穿 5 层（事件模型 → 持久化 → CLI 透传 → Web API → Web 渲染），
但每一层都只加了一个可选字段 `resultFull?: string`：
- 事件层：加字段 + 生成
- 持久化层：加写入 + 读取
- CLI 层：只透传，不改展示
- Web API 层：零改动（自动带出）
- Web 渲染层：核心改造集中在 `ToolStatus.tsx` 一个文件

总共改了 **10 个文件**，每个文件改动量都很小（1-30 行），没有大规模重构。

### 5. 向后兼容零成本

`resultFull` 是可选字段，旧 JSONL 没有它就 fallback 到 `resultSummary`。
加上 args 详情提取，旧数据的 Web 展示体验也有质的提升。
