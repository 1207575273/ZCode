# F13 SubAgent（父子 Agent）设计文档

> 日期: 2026-03-12 | 优先级: P1 | 状态: 设计完成，待实现

---

## 一、需求概述

主 Agent 可派发子 Agent 独立执行任务。子 Agent 拥有完整的 AgentLoop（多轮 LLM + 工具调用），
执行完毕后将结果作为 tool_result 返回给主 Agent。

**核心价值：** 让 LLM 能将复杂任务拆分为多个子任务并行/串行执行，提升解决复杂问题的能力。

---

## 二、架构定位

### 2.1 核心认知

> Agent 工程的本质就是：**一个循环 + 一堆工具。**

SubAgent 不是新的架构层，而是一个普通的 Tool——它的 execute 方法内部恰好启动了一个新的 AgentLoop。
对主 AgentLoop 来说，dispatch_agent 和 bash 没有本质区别：调用 → 等结果 → 拿到 output。

### 2.2 架构模式选择

参考 Claude Code 架构哲学图（docs/res/04_claude_code_架构哲学.png），存在两种模式：

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| **Subagents（一期）** | 子 Agent 隔离执行，结果汇报给主 Agent，子 Agent 之间不通信 | 独立任务并行 |
| **Agent Teams（二期）** | 共享 Task List，Teammate 之间可通信和认领任务 | 协作型复杂任务 |

**一期实现 Subagents 模式，架构预留 Agent Teams 扩展点。**

### 2.3 参考：Claude Code 实际实现

通过分析 Claude Code 的 JSONL 日志（主会话 + subagents/ 目录），确认其实现为：

- **一个 `Task` 工具**，参数为 `{ description, prompt, subagent_type }`
- 子 Agent 有独立 JSONL（`subagents/agent-<id>.jsonl`），标记 `isSidechain: true`
- 主会话记录 `progress` 事件（子 Agent 执行期间的实时状态）
- 子 Agent 完成后，最终文本作为 `tool_result` 返回主 Agent
- 没有 task_create / task_list 等分离的任务管理工具

## 三、方案选型

### 3.1 架构方案

| 方案 | 说明 | 优劣 |
|------|------|------|
| A: 复用 AgentLoop + 解耦 Tool 接口 | 子 Agent 实例化新 AgentLoop，Tool 接口支持 Generator 返回 | 真正解耦，事件流天然通过 generator 链路传播 |
| B: 独立轻量循环 | 写 SubAgentRunner，不复用 AgentLoop | 完全解耦，但重复实现核心逻辑 |
| C: AgentLoop 可嵌套 + onProgress 回调 | Tool.execute 不变，通过 ToolContext.onProgress 回调透传事件 | 回调绕开了问题而非解决问题 |

### 3.2 选定方案 A 的理由

**解耦是真正的工程思想。**

当前 Tool 接口 `execute(): Promise<ToolResult>` 假设所有工具都是"调用→等待→返回"的一次性操作。
dispatch_agent 打破了这个假设——它是一个长时间运行、有中间状态的工具。

方案 C 用 onProgress 回调绕开这个限制，本质上是在 Promise 体系上打补丁：
- ToolContext 被迫加入 `onProgress`、`provider`、`registry` 等大量运行时字段
- useChat、pipe-runner 都要传回调
- 事件流的传播路径从统一的 generator yield 变成两条路（yield + 回调），增加理解成本

方案 A 直接解决根本问题：**借鉴 LangChain 的 invoke/stream 分离思想，新增 StreamableTool 接口。**

```typescript
// 基础工具接口（现有，不动）
interface Tool {
  execute(args, ctx): Promise<ToolResult>
}

// 流式工具接口（新增，继承 Tool）
interface StreamableTool extends Tool {
  stream(args, ctx): AsyncGenerator<AgentEvent, ToolResult>
}
```

- 普通工具（read_file、bash）：只实现 Tool，零改动
- dispatch_agent：实现 StreamableTool，`stream()` yield 子 Agent 进度，return 最终结果
- AgentLoop.#executeOneTool 通过 `isStreamableTool()` 判断走 stream 还是 execute

**LangChain 的启发：** LLM 调用分为流式和非流式（invoke/stream），工具同理。
StreamableTool 的 `stream()` 就像 LLM 的流式调用，yield 中间事件，return 最终结果。
而 `execute()` 保持为"一次性调用"，两者分离、互不影响。

事件流传播路径统一为 generator 链路，与现有 #callLLM 的 yield* 模式完全一致：

```
主 AgentLoop.run()
  yield* #callLLM()              ← 现有：LLM 事件透传
  yield* #executeOneTool()
    yield* dispatch_agent.stream()  ← 新增：子 Agent 事件透传
      yield subagent_progress(...)
      return { success: true, output }
```

**改动对比：**

| 对比项 | onProgress 回调方案 | StreamableTool 解耦方案 |
|--------|--------------------|--------------------|
| Tool 接口 | 不变，但 ToolContext 加 6 个字段 | 不变，新增独立的 StreamableTool 接口 |
| AgentLoop | 加 onProgress 到 config + ctx 透传 | #executeOneTool 加 isStreamableTool 判断 |
| useChat | 传 onProgress 回调 | 不变 |
| pipe-runner | 传 onProgress 回调 | 不变 |
| dispatch-agent | 调 ctx.onProgress | 实现 stream() 方法，直接 yield |
| ToolContext | 加 provider/model/registry/onProgress | 只加 provider/model/registry |

## 四、详细设计

### 4.1 DispatchAgentTool 接口

```typescript
// src/tools/dispatch-agent.ts

export class DispatchAgentTool implements Tool {
  readonly name = 'dispatch_agent'
  readonly description =
    'Dispatch a sub-agent to handle a task independently. ' +
    'The sub-agent runs a complete agent loop with its own context, ' +
    'executes tools as needed, and returns the final result. ' +
    'Use this for tasks that can be done in parallel or need isolation.'
  readonly parameters = {
    type: 'object',
    properties: {
      description: {
        type: 'string',
        description: 'Short task description (for logging and progress display)',
      },
      prompt: {
        type: 'string',
        description: 'Complete instructions for the sub-agent.',
      },
      // 二期预留：子 Agent 类型，映射到预设工具集
      // type: { type: 'string', enum: ['general', 'explore', 'plan'] }
    },
    required: ['description', 'prompt'],
  }
  readonly dangerous = false  // 子 Agent 内部的危险工具走各自的权限检查
}
```

**`dangerous = false` 的理由：**
dispatch_agent 本身不执行危险操作。子 Agent 内部调用 bash/write 等危险工具时，
因为 `isSidechain: true` 会自动批准（主 Agent 派发即授权）。

### 4.2 子 Agent 类型系统（二期预留）

一期所有子 Agent 共享主 Agent 的完整工具集（排除 dispatch_agent 防递归）。

二期按工具集划分隐性类型：

| 类型 | 可用工具 | 典型用途 |
|------|---------|---------|
| general | 全部工具 | 通用任务 |
| explore | read / glob / grep | 代码分析、审查（只读） |
| plan | read / glob / grep | 设计方案、架构规划（只读） |

实现方式：`ToolRegistry.cloneWithout()` 配合类型→排除列表映射表。

### 4.3 子 Agent 的 system prompt

```
You are a sub-agent. Complete the assigned task autonomously.
Constraints:
- Focus solely on the given task
- Do NOT dispatch further sub-agents
- Do NOT ask questions — you have no user interaction
- When done, output your final result as plain text
```

**提示词设计说明：**
- "Focus solely" — 防止子 Agent 偏离任务，节省 token
- "Do NOT dispatch" — 提示词层面的递归防护（硬编码层面通过排除工具保障）
- "Do NOT ask questions" — 子 Agent 没有用户交互通道，问了也没人回答
- "plain text" — 确保最终输出可直接作为 tool_result 返回

### 4.4 stream() 内部流程（StreamableTool 模式）

dispatch_agent 实现 `StreamableTool`，其 `stream()` 方法返回 `AsyncGenerator<AgentEvent, ToolResult>`，
通过 yield 实时透传子 Agent 进度，最终 return ToolResult。

`execute()` 方法作为 fallback，内部调用 `stream()` 但丢弃中间事件，只返回最终结果。

```
async *stream(args, ctx):
  1. 校验参数（prompt 非空）
  2. 生成 agentId（17 位 hex，与 Claude Code 格式对齐）
  3. 构建子 Agent 受限工具集：ctx.registry.cloneWithout('dispatch_agent')
  4. 创建子 AgentLoop:
     - provider/model: 从 ctx 继承（一期不降级，二期预留口子）
     - registry: 受限工具集
     - systemPrompt: 子 Agent 专用（见 4.3）
     - maxTurns: SUB_AGENT_MAX_TURNS (15)
     - isSidechain: true（跳过权限检查 + 日志标记）
     - agentId: 生成的 ID
     - signal: 从 ctx 继承（用户中止时子 Agent 同步中止）
  5. 运行子 AgentLoop，消费事件流:
     - text → 累积 finalText
     - tool_start → yield subagent_progress（实时透传到主 AgentLoop generator 链路）
     - tool_done → yield subagent_progress
     - llm_usage → yield 透传（token 计量）
     - permission_request → 不会触发（isSidechain 自动批准）
  6. return ToolResult:
     - 成功: { success: true, output: finalText }
     - 异常: { success: false, output: 已累积的部分文本, error: 错误信息 }

async execute(args, ctx):
  // fallback：消费 stream() 但丢弃中间事件
  const gen = this.stream(args, ctx)
  let result = await gen.next()
  while (!result.done) { result = await gen.next() }
  return result.value
```

### 4.5 递归防护（双重保障）

| 层级 | 机制 | 说明 |
|------|------|------|
| 工具层 | `registry.cloneWithout('dispatch_agent')` | 子 Agent 的工具集中不包含 dispatch_agent，LLM 无法调用 |
| 提示词层 | "Do NOT dispatch further sub-agents" | 即使工具存在也不调用（兜底） |

### 4.6 中止信号传播

```
用户 Ctrl+C → AbortController.abort()
  → 主 AgentLoop signal 中止
    → dispatch_agent.execute 内的子 AgentLoop 共享同一 signal
      → 子 AgentLoop 的 provider.chat() 收到 abort
      → 子 AgentLoop 抛 AbortError
    → dispatch_agent.execute catch → 返回部分结果
  → 主 AgentLoop 正常处理 AbortError
```

## 五、事件流设计

### 5.1 Tool 接口解耦：StreamableTool 模式（借鉴 LangChain invoke/stream 分离）

```typescript
// src/tools/types.ts

import type { AgentEvent } from '@core/agent-loop.js'

/** 基础工具接口（现有，不动） */
export interface Tool {
  readonly name: string
  readonly description: string
  readonly parameters: Record<string, unknown>
  readonly dangerous?: boolean
  execute(args: Record<string, unknown>, ctx: ToolContext): Promise<ToolResult>
}

/** 流式工具接口（新增）—— 支持 yield 中间事件的长时间运行工具 */
export interface StreamableTool extends Tool {
  stream(args: Record<string, unknown>, ctx: ToolContext): AsyncGenerator<AgentEvent, ToolResult>
}

/** 类型守卫：判断工具是否支持流式执行 */
export function isStreamableTool(tool: Tool): tool is StreamableTool {
  return 'stream' in tool && typeof (tool as StreamableTool).stream === 'function'
}
```

**设计说明：**
- Tool 接口完全不变，现有工具（read_file、bash 等）零改动
- StreamableTool 继承 Tool，新增 `stream()` 方法用于 yield 中间事件
- `execute()` 保留作为 fallback（内部消费 stream 但丢弃中间事件）
- AgentLoop.#executeOneTool 通过 `isStreamableTool()` 类型守卫选择路径
- 与 LangChain Runnable 协议（invoke/stream/batch）的思想一致：同步与流式分离

### 5.2 AgentLoop 适配

```typescript
// agent-loop.ts #executeOneTool 改动

async *#executeOneTool(tc, history): AsyncGenerator<AgentEvent> {
  yield { type: 'tool_start', ... }
  const allowed = yield* this.#checkPermission(tc)
  if (!allowed) { ... }

  const start = Date.now()
  const tool = this.#registry.get(tc.toolName)
  const ctx = { cwd: process.cwd(), provider: ..., registry: ..., signal: ... }

  let result: ToolResult
  if (isStreamableTool(tool)) {
    // 流式工具（dispatch_agent）：yield* 透传中间事件，return 值为最终结果
    result = yield* tool.stream(tc.args, ctx)
  } else {
    // 普通工具：await execute()
    result = await tool.execute(tc.args, ctx)
  }

  const durationMs = Date.now() - start
  // ... history push + tool_done yield（与现有逻辑相同）
}
```

**事件流传播路径统一为 generator 链路：**

```
主 AgentLoop.run()
  yield* #callLLM()                    ← LLM 事件
  yield* #executeOneTool()
    yield* dispatch_agent.stream()     ← 子 Agent 事件实时透传
      yield subagent_progress(...)     ← UI 立即收到
      yield subagent_progress(...)
      return { success, output }       ← 最终结果
```

### 5.3 新增 AgentEvent 类型
