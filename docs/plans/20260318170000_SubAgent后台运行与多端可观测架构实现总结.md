## 一、背景与目标

ZCli 的 SubAgent（一期）只支持**前台阻塞式**执行——主 Agent 派发子 Agent 后必须等它跑完，无法并行调度多个子任务。同时，子 Agent 的执行过程对用户是黑盒：CLI 只显示一行 `Agent[描述] turn 3/15`，Web 端完全无感知。

本次迭代的目标是在三个维度上完成突破：

1. **后台运行能力**：SubAgent 支持 `run_in_background`，主 Agent 不再被阻塞
2. **多端可观测性**：CLI 悬浮面板 + Web 折叠卡片，实时查看子 Agent 完整对话流
3. **会话生命周期管理**：CLI 状态感知、session 切换通知、刷新后数据恢复

## 二、架构设计

### 2.1 事件三写架构

本次最核心的架构决策是 **SubAgent 事件三写**——每个子 Agent 的事件同时写入三个通道：

```
SubAgent AgentLoop 产出事件
  │
  ├─→ ① subLogger.consume()      → 独立 JSONL（审计 + 回放）
  │       <sessions>/<project>/<parentSid>/subagents/agent-<agentId>.jsonl
  │
  ├─→ ② consumeAgentEvent()       → subagent-store 内存缓存（CLI 实时查看）
  │       Map<agentId, SubAgentState>
  │
  └─→ ③ eventBus.emit()           → Bridge → WebSocket → Web 端（跨进程推送）
          subagent_event { agentId, detail: { kind, ... } }
```

**为什么是三写而不是从一个源派生？**

| 通道 | 读取场景 | 延迟要求 | 持久化 |
|------|----------|----------|--------|
| JSONL | 回放、审计、resume | 无 | 是 |
| 内存 Store | CLI SubAgentPanel 实时展示 | <1ms | 否 |
| EventBus → WebSocket | Web 端实时展示 | <50ms | 否 |

三个消费方的读取模式完全不同。JSONL 是追加写 + 顺序读；内存 Store 是随机读（按 agentId）；EventBus 是发布-订阅推送。如果从单一源派生会引入不必要的耦合和延迟。

### 2.2 SubAgent 生命周期

```
dispatch_agent(run_in_background=true)
  │
  ├─ registerSubAgent(agentId)          → 内存 Store 注册
  ├─ subLogger.logUserMessage(prompt)    → JSONL 写入 user 消息
  ├─ yield subagent_progress(turn=0)     → 通知 UI 新 Agent 启动
  └─ return { agentId }                  → 主 Agent 立即拿到结果，继续执行
      │
      └─ 后台异步 ─── runSubAgentInBackground() ───
          │
          │  for await (event of subLoop.run(...))
          │    → subLogger.consume(event)       // JSONL
          │    → consumeAgentEvent(agentId)     // 内存
          │    → eventBus.emit(subagent_event)  // Web
          │    → eventBus.emit(subagent_progress) // UI 摘要
          │
          └─ 完成后
              → markSubAgentDone(agentId)
              → subLogger.finalize()
              → eventBus.emit(subagent_done)
```

### 2.3 C+B 双层存储策略

**执行中（C 层 — 内存）**：
- SubAgent 事件实时写入 `subagent-store` 内存 Map
- CLI SubAgentPanel 从 Store 直读，零延迟
- Web 端通过 EventBus → WebSocket 推送实时更新

**执行完（B 层 — JSONL 回放）**：
- 内存 Store 保留到会话结束（供 CLI 查看）
- Web 刷新 / 历史回放 → `session_init` 时从 JSONL 回放重建
- `sessionStore.loadSubagents()` 扫描 `subagents/` 目录，解析每个 JSONL

**两层自然衔接，零重复写入**——JSONL 在执行过程中就已经写完了，回放只是读取。

### 2.4 CLI ↔ Web 会话生命周期

```
CLI 启动 → Bridge Server 启动 → CLI 注册(sessionId)
                                     │
Web 打开 → WebSocket 连接 → register(sessionId)
                                     │
                    Bridge 检查: 有 CLI 在这个 session 上吗?
                      │                        │
                    有 → cliConnected: true    无 → cliConnected: false
                                                    + activeSessionId: 新会话 ID
                                                    → Web 显示 "CLI 离线" 横幅
                                                       + "切换到活跃会话" 按钮

CLI /resume old-session
  → updateBridgeSession(oldSid) → Bridge re-register
    → 广播 cli_status(connected: true) 给 oldSid 的 Web 客户端
      → Web 横幅消失，恢复正常
```

### 4.2 Web 端 — 对话流内联折叠卡片

SubAgentCard 内联在 `dispatch_agent` 的 toolEvent 下方，而不是作为独立块浮在底部。这样保证了**时序正确性**——用户能看到"在哪个对话上下文中启动了这个子 Agent"。

```
┌─ ✓ Agent(写Python文件) 34ms ──────────────────────────── ▶ ─┐
│                                                              │
│  点击展开 ↓                                                   │
│                                                              │
│  ✓ read_file     120ms  ⎿ 15 lines                          │
│  ✓ write_file     85ms  ⎿ demo_python.txt                   │
│  ✓ bash           230ms  ⎿ Hello, World!                     │
│                                                              │
│  │ 文件已创建并验证通过。                    ← AI 文本输出     │
│  │ demo_python.txt 包含 Python Hello World                   │
│  │ 程序，可直接运行。                                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**关键设计决策：agentId 匹配机制**

ToolStatus 渲染 `dispatch_agent` 条目时，从 `resultSummary` 中用正则 `agentId:\s*([a-f0-9]+)` 提取 agentId，再从 `subAgents` Map 查找对应的 SubAgentInfo。这个方案的优势是不需要修改 tool event 的数据结构，利用已有的 resultSummary 字段做关联。

### 4.3 Web 端 — 数据恢复

## 五、会话生命周期感知

### 5.1 问题场景

```
CLI-A(session-A) + Bridge-A → 退出
CLI-B(session-B) + Bridge-B → 启动
Web 重连 → 用老 session-A 注册 → 消息石沉大海
```

### 5.2 解决方案

**不需要心跳**，利用 Bridge 已有的客户端追踪机制：

1. **注册时检查**：Web register 时，Bridge 检查是否有 CLI 在同一 session → `session_init.cliConnected`
2. **断连时通知**：CLI WebSocket 关闭 → Bridge 广播 `cli_status(connected: false)` 给同 session 的 Web
3. **Resume 时恢复**：CLI `/resume` → `updateBridgeSession(newSid)` → Bridge 广播 `cli_status(connected: true)`
4. **切换引导**：`session_init.activeSessionId` 指向当前活跃的 CLI session → Web 显示"切换到活跃会话"按钮

## 七、架构扩展性分析

本次实现为后续功能铺设了关键基础设施：

### 7.1 SubAgent 对话切入

`subagent-store` 内存 + JSONL 保存了完整的对话历史（user + assistant + tool_call）。未来实现"切入 SubAgent 继续对话"只需：
- 从 JSONL 恢复 message history
- 创建新的 AgentLoop，注入历史上下文
- UI 层切换到 SubAgent 视角

### 7.2 Agent Teams

当前架构已支持多个 SubAgent 并行后台运行 + 独立事件流。Agent Teams 只需在此基础上增加：
- 角色定义（team-lead / reviewer / coder）
- 通信协议（agent 间消息传递）
- UI 层 shift+↑↓ 切换视角

### 7.3 CLI ↔ Web 双向编排

`updateBridgeSession` 机制使得 CLI 和 Web 能动态切换 session 关联。后续可扩展为：
- Web 端主动 resume 任意 session
- Web 端创建新 session 指派给 CLI
- 多 CLI 实例负载均衡
