# F8 会话持久化与恢复 — 实现总结

> 完成日期: 2026-03-10 | 需求优先级: P1 | 16 commits | 19 files | +2138 lines

---

## 1. 实现目标

让用户关闭 ZCli 后可恢复之前的对话，继续上下文交流。支持 JSONL 事件流存储、按项目分目录组织、交互式恢复面板、对话分叉（time-travel）。

---

## 2. 架构设计

### 2.1 存储格式：JSONL 事件流

采用 append-only 的 JSONL 格式（非 JSON 快照），每个事件追加一行。

**优势：**
- 断电只丢最后一行，不损坏整个文件
- 天然结构化日志，后续 Agent 日志系统可复用
- 增量写入，无需每轮全量覆写

### 2.2 目录结构

```
~/.zcli/sessions/
  └── <project-slug>/                       # cwd 路径转 slug（/\: → -）
      └── 20260310153022317_<uuidv7>.jsonl   # 时间戳前缀 + UUIDv7
```

### 2.3 事件树结构（支持分叉）

每个事件包含 `uuid`（自身 ID）和 `parentUuid`（父事件 ID），形成事件树：

```
session_start → user₁ → assistant₁ → user₂ → assistant₂        (主干)
                                     ↗ session_resume → user₃ → assistant₃  (分叉)
```

加载时从叶节点沿 `parentUuid` 回溯到根，精确还原任一分支。

## 4. 核心功能

### 4.1 自动持久化

用户发消息和 LLM 回复时自动追加事件到 JSONL，无需手动保存。Session 在首次 submit 时惰性创建。持久化失败不阻断对话流（静默处理）。

### 4.2 两种恢复入口

| 入口 | 行为 |
|------|------|
| `zcli --resume` | 启动后自动弹出 ResumePanel |
| `zcli --resume <id>` | 直接恢复指定会话 |
| `/resume` | 对话中打开 ResumePanel |

### 4.3 ResumePanel 交互

- **搜索框**（TextInput）：带光标，实时模糊过滤
- **Current Session**：第一项，Enter 返回当前对话
- **会话列表**：首条消息 + 相对时间 + git 分支 + 文件大小
- **分支子视图**：多分支会话自动展开分支选择
- **Ctrl+A**：切换当前项目 / 所有项目
- **Esc**：有搜索文字先清空，无则关闭

### 4.4 对话分叉（Time-Travel）

- `/fork` → 显示当前对话消息列表
- 选择任一消息作为分叉点 → 截断到该处 → 后续消息自然形成新分支
- `SessionStore.listBranches()` 通过叶节点检测自动识别所有分支
- `loadMessages(sessionId, leafEventUuid?)` 沿事件树精确加载指定分支

### 4.5 退出提示

Ctrl+C 退出时打印 resume 命令，支持开发模式（`pnpm dev --`）和全局安装（`zcli`）两种格式。使用 `process.on('exit')` 兜底确保可靠输出。

## 5. 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 存储格式 | JSONL 事件流 | Append-only 抗断电、增量写入、天然日志 |
| SessionId | UUIDv7 | 时间有序可排序，全局唯一 |
| 文件名 | `YYYYMMDDHHMMSSMMM_<uuid>.jsonl` | 人可读 + 自然排序 + UUID 唯一性 |
| 分叉机制 | parentUuid 事件树 | 无需额外数据结构，JSONL 天然支持 |
| 恢复范围 | 仅 user + assistant 文本 | 工具调用历史保留在 JSONL 但不加载到 UI |
| 模块级 sessionId | 与 mcpManager 同模式 | 退出时需在 React 组件树外访问 |

## 8. 对话分叉使用指南 & 与 Claude Code 原生 CLI 的对比

### 8.1 使用场景

对话分叉（Time-Travel）解决一个常见痛点：**LLM 某一轮回答不满意，想从那个节点重新来过，但不想丢失已有的对话历史。**

典型场景：
- LLM 第三轮给了一个糟糕的方案 → fork 回第二轮，换个问法重新引导
- 想对比同一个问题的不同回答 → fork 出两个分支，各自深入
- 代码修改走偏了 → fork 回修改前，换个实现思路

### 8.2 操作方式

**从当前对话分叉：**

```
/fork
```

弹出 ForkPanel，显示当前对话的所有消息：

```
Fork Conversation
  Select a message to fork from — conversation will branch from this point

  [You] 帮我实现一个排序算法
  [AI]  好的，我用冒泡排序来实现...
  [You] 能不能用更高效的算法？
❯ [AI]  当然，这里用归并排序重新实现...     ← 选这里

↑↓ navigate · Enter fork from here · Esc cancel
```

按 Enter 后：
1. 对话截断到选中的消息（之后的消息从 UI 中移除）
2. 系统提示「Forked from message — new branch started」
3. 继续输入，新消息自然形成**新分支**（原分支不受影响）

**恢复时选择分支：**

```
/resume
```

如果某个 session 有多个分支，按 Enter 后进入分支选择子视图：

```
Select Branch
  2 branches found

❯ 当然，这里用快速排序重新实现...
  4 msgs · 3m ago · forked
  当然，这里用归并排序重新实现...
  6 msgs · 10m ago · main

↑↓ navigate · Enter select branch · Esc back to sessions
```

每个分支显示：末尾消息预览、消息数、时间、是否为分叉分支。

**快捷键汇总：**

| 操作 | ForkPanel | ResumePanel |
|------|-----------|-------------|
| 导航 | ↑↓ | ↑↓ |
| 选择 | Enter | Enter |
| 关闭 | Esc / q | Esc（有搜索文字先清空） |
| 搜索 | — | 直接输入文字 |
| 切换范围 | — | Ctrl+A（当前项目/所有项目） |
| 回到当前对话 | — | 选择「↩ Current Session」 |

### 8.3 与 Claude Code 原生 CLI 的核心差异

| 维度 | Claude Code CLI | ZCli |
|------|----------------|------|
| **会话数据结构** | 链表（线性事件序列） | 树（`parentUuid` 事件树） |
| **恢复方式** | 只能从对话末尾继续 | 可选择任意分支的任意叶节点恢复 |
| **对话分叉** | 不支持 | `/fork` 从任意消息处分叉，创建新分支 |
| **分支管理** | 无 | `/resume` 自动检测多分支，进入分支选择子视图 |
| **搜索** | 有搜索框 | 有搜索框（TextInput + 模糊匹配） |
| **项目范围切换** | 有（Ctrl+A） | 有（Ctrl+A） |
| **退出提示** | 有（resume 命令） | 有（自适应 dev/global 模式） |
| **存储格式** | JSONL 事件流 | JSONL 事件流（兼容，额外利用 parentUuid 构建树） |
| **Context 压缩** | 有（summary 事件） | 预留（`summary` 类型已定义，待实现） |
| **Token 用量追踪** | 有（usage 字段） | 预留（`usage` 字段已定义，待 F10 聚合） |
| **Tool results 外置** | 有（大文件分离存储） | 暂无（当前工具结果内联在 JSONL） |

### 8.4 技术原理

分叉的核心在于 JSONL 中每个事件的 `uuid` + `parentUuid` 字段：

```jsonl
{"type":"session_start","uuid":"aaa","parentUuid":null,...}
{"type":"user","uuid":"bbb","parentUuid":"aaa","message":{"content":"问题1"},...}
{"type":"assistant","uuid":"ccc","parentUuid":"bbb","message":{"content":"回答1"},...}
{"type":"user","uuid":"ddd","parentUuid":"ccc","message":{"content":"追问"},...}
{"type":"assistant","uuid":"eee","parentUuid":"ddd","message":{"content":"回答2"},...}
{"type":"session_resume","uuid":"fff","parentUuid":"ccc",...}
{"type":"user","uuid":"ggg","parentUuid":"fff","message":{"content":"换个问法"},...}
{"type":"assistant","uuid":"hhh","parentUuid":"ggg","message":{"content":"新回答"},...}
```

上面的 JSONL 包含两个分支：
- **主干**：aaa → bbb → ccc → ddd → eee（叶节点 eee）
- **分叉**：aaa → bbb → ccc → fff → ggg → hhh（叶节点 hhh，从 ccc 处分叉）

加载分支时，从叶节点沿 `parentUuid` 回溯到根（`parentUuid: null`），反转得到时间顺序。两个分支共享 aaa → bbb → ccc 的前缀路径，之后各自延伸。

`listBranches()` 通过找出所有**叶节点**（没有任何事件以其 uuid 为 parentUuid 的事件）来自动识别分支，无需额外的分支元数据。
