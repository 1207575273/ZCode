# F16 会话与数据清理 — 实现总结

## 1. 实现目标

提供会话文件（JSONL）和用量记录（SQLite usage_logs）的生命周期管理，支持 dry-run 预览和实际清理。

**入口**：REPL `/gc` 斜杠指令（不含自动清理策略，按需手动触发）。

---

## 2. 核心架构

```
用户输入 /gc [--dry-run] [--days N] [sessions|usage]
  │
  ▼ CommandRegistry.dispatch()
  │
  ▼ GcCommand.execute() → CommandAction { type: 'run_gc', ... }
  │
  ▼ App.tsx handleSubmit case 'run_gc'
  │
  ├── dryRun=true  → getCleanupStats(opts) → 输出预览信息
  └── dryRun=false → executeCleanup(opts)  → 输出清理结果
```

**清理策略**：

| 目标 | 默认保留天数 | 清理方式 |
|------|-------------|---------|
| sessions（JSONL 文件） | 30 天 | 遍历 baseDir，按 mtime 过期判断，调用 sessionStore.cleanup() |
| usage（SQLite 记录） | 90 天 | DELETE WHERE timestamp < cutoff + VACUUM 回收空间 |

## 4. 关键实现细节

### 4.1 cleanup-service.ts

两个核心导出函数：

**`getCleanupStats(options)`** — dry-run 统计：
- `scanSessionFiles(retentionDays)`: 遍历 sessionStore.baseDir 下所有 `.jsonl` 文件，按 mtime 判断过期
- `scanUsageLogs(retentionDays, db)`: SQL COUNT 统计总量和过期量
- `target` 参数控制只扫描 sessions / usage / 两者

**`executeCleanup(options)`** — 实际清理：
- 会话文件：先 scanSessionFiles 统计，再调用 `sessionStore.cleanup(days)`
- 用量记录：`DELETE FROM usage_logs WHERE timestamp < ?` + 有删除时执行 `VACUUM`
- `_db` 内部参数支持测试注入，避免复杂的 vi.mock

### 4.2 gc.ts（/gc 指令）

参数解析：
- `--dry-run` → 预览模式（不删除）
- `--days N` → 自定义保留天数（覆盖默认值）
- `sessions` / `usage` → 只清理指定目标
- 可组合：`/gc --dry-run --days 14 sessions`

### 4.3 App.tsx 集成

`case 'run_gc'` 分支：
- dry-run 模式：调用 `getCleanupStats()` 输出文件数/大小/行数预览
- 实际清理：先检查是否有过期数据，无则提示"没有需要清理的过期数据"，有则调用 `executeCleanup()` 输出删除结果
- `fmtSize()` 内联工具函数：字节数格式化为 B/KB/MB

## 5. 设计决策

| # | 决策 | 理由 |
|---|------|------|
| 1 | 不实现 REPL 自动清理策略 | 用户明确反馈 REPL 不应支持自动清理，手动 /gc 即可 |
| 2 | `_db` 注入而非 vi.mock | 简单可靠，避免 ESM mock 的复杂性 |
| 3 | VACUUM 仅在有删除时执行 | 无删除时 VACUUM 浪费 I/O |
| 4 | 默认 session 30 天 / usage 90 天 | 会话文件占磁盘多但访问频率低，用量记录小但统计需求更长 |
