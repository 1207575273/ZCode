# 启动初始化并行优化总结

> 日期: 2026-03-14

## 一、问题背景

cCli 随着功能迭代，启动时需要初始化的子系统越来越多：

| 子系统 | IO 类型 | 实测耗时 | 职责 |
|--------|---------|----------|------|
| Skills 发现 | 磁盘扫描 | ~262ms | 扫描四层来源的 skill 定义 |
| Instructions 加载 | 磁盘读取 | ~56ms | 加载 CCODE.md / CLAUDE.md 指令文件 |
| Hooks 发现 | 磁盘读取 | ~5ms | 从插件包/项目/用户级收集 hooks |
| SessionStart Hooks | 子进程 | ~65ms | 执行 hooks 注入 additionalContext |
| MCP 连接 | 网络 IO | ~6000ms | 连接外部 MCP Server（子进程/HTTP） |
| 文件索引 | 磁盘扫描 | ~701ms | fast-glob 全量扫描 + fzf 索引构建 |
| System Prompt | 纯计算 | <1ms | 拼接 instructions + skills + hooks |

初始版本全部串行执行，首次 submit 延迟 ≈ 所有耗时之和 ≈ 7+ 秒。

## 二、优化演变历程

### 第一阶段：submit 内部并行化

**策略：** 在 useChat 的 submit 函数里，将无依赖的任务用 `Promise.all` 并行。

```
Promise.all([Skills, MCP, Instructions])  → Hooks → SessionStartHooks → SystemPrompt
```

**收益：** 总耗时从串行之和降为 max(Skills, MCP) + Hooks 链 ≈ 6.1s
**问题：** 仍然阻塞首次 submit，用户按回车后要等 6 秒。

### 第二阶段：App mount 提前启动

**策略：** 将初始化从 submit 提前到 App 组件挂载时（useEffect），用户看 WelcomeScreen 和打字的时间后台全速执行。

**实现：**
- bootstrap.ts 提供统一的 `bootstrapAll()` 函数（幂等 Promise，只执行一次）
- App mount 时立即调用
- submit 时 await 同一 Promise，大概率已完成，零等待

**收益：** 用户感知延迟从 6s → 0s（打字时间 > 后台初始化时间）
**问题：** MCP 6 秒仍在 `bootstrapAll` 内，如果用户打字很快仍会被阻塞。

### 第三阶段：依赖拓扑重排

**发现：** MCP 和 Skills→Hooks 链是完全独立的，但 `Promise.all` 等 Layer 0 全部完成才开始 Layer 1，MCP 的 6 秒不必要地阻塞了 Hooks。

**优化：** 按真实依赖关系重排为三条独立链路：

```
Promise.all([
  链A: Skills → Instructions → Hooks → SessionStartHooks → SystemPrompt
  链B: MCP 连接
  链C: FileIndex 扫描
])
```

**收益：** 总耗时 = max(链A ~388ms, MCP ~6000ms, FileIndex ~701ms) = MCP 决定

### 第四阶段：MCP 后台脱离（最终方案）

**核心洞察：** MCP 是纯外部依赖（网络/子进程），耗时不可控，且对话开始时不一定需要 MCP 工具。不应该让外部 IO 阻塞核心对话能力。

**策略：** MCP 从 `bootstrapAll` 完全脱离，独立后台静默加载：

```
App mount:
  ├── bootstrapAll()           ← 核心模块：Skills → Hooks → SystemPrompt | FileIndex
  └── startMcpBackground()     ← 后台：MCP 连接，fire-and-forget

submit:
  ├── await bootstrapAll()     ← 大概率已完成（~701ms）
  └── registerMcpTools()       ← MCP 已就绪则注册，未就绪则跳过
```

**最终耗时拓扑：**

```
时间轴 →
0ms                    388ms              701ms              6000ms
 │                       │                  │                   │
 ├── Skills(262) → Hooks(5) → StartHooks(65) → SystemPrompt(<1)│
 │         └── Instructions(56)                                 │
 ├── FileIndex(701) ────────────────────────┤                   │
 │                                          │                   │
 ├── MCP(6000，后台) ───────────────────────────────────────────┤
 │                                          │                   │
 │                                   bootstrapAll 完成     MCP 就绪
 │                                   (用户可对话)         (MCP 工具可用)
```

**收益：**
- bootstrap 耗时：7s+ → 701ms（降低 90%）
- 用户首次 submit：0 阻塞（打字时间覆盖 701ms）
- MCP 6 秒完全不影响任何交互

### 附加优化：bin/ccli.ts dynamic import 并行

交互模式入口有 7 个串行 `await import()`，改为 `Promise.all` 并行加载，减少模块解析等待。

## 三、最终架构

### 启动流程图

```
bin/ccli.ts
  │
  ├── initialize()                  ← 同步：目录/配置/诊断
  ├── parseArgs()                   ← 同步：参数解析
  │
  ├── [Pipe Mode] → bootstrapAll() + ensureMcpInitialized() 并行 → runPipe()
  │
  └── [REPL Mode]
      ├── Promise.all(7 dynamic imports)   ← 并行模块加载
      └── render(<App>)
            │
            └── useEffect (mount)
                  ├── startMcpBackground()   ← fire-and-forget
                  └── bootstrapAll()
                        ├── 链A: Skills → Instructions → Hooks → SystemPrompt
                        └── 链B: FileIndex
```

### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 编排入口 | 单一 `bootstrapAll()` | 一处编排、处处复用，消除散弹式调用 |
| MCP 策略 | 后台脱离，不阻塞 | 网络延迟不可控，不应拖累核心体验 |
| 幂等保障 | Promise 缓存 + 内部 flag | 多入口调用安全，不重复执行 |
| Pipe 模式 | MCP 必须等待 | 一次性执行，无后续机会补注册 |
| Dev 计时 | `performance.now()` 各模块 | tsx 运行时显示，生产模式零开销 |

### 模块间依赖（编排必须遵守）

```
Skills ──→ Hooks ──→ SessionStartHooks ──→ buildSystemPrompt
                                               ↑
Instructions ──────────────────────────────────┘

MCP：完全独立
FileIndex：完全独立
```

## 四、Dev 模式 Timing 输出

启动时 WelcomeScreen 上方显示：

```
[dev] bootstrap 701ms (skills 262 → hooks 5 → startHooks 65 | fileIndex 701 | instructions 56ms) mcp: loading...
```

MCP 就绪后自动刷新：

```
[dev] bootstrap 701ms (skills 262 → hooks 5 → startHooks 65 | fileIndex 701 | instructions 56ms) mcp: 6043ms
```

格式约定：`→` 表示串行依赖链，`|` 分隔并行链路。

## 五、进一步优化建议

### 5.1 FileIndex 增量缓存（预估收益：701ms → <50ms）

当前每次启动都全量 `fast-glob` 扫描。可以：
- 将扫描结果序列化到 `.ccode/file-index.cache`（JSON/MessagePack）
- 启动时读缓存 → 立即可用 → 后台增量 diff 更新
- 缓存失效策略：对比 `.git/index` mtime 或目录 mtime

这是当前最大的可优化项（701ms），实现后 bootstrap 总耗时可降到 ~388ms。

### 5.2 Skills 扫描缓存（预估收益：262ms → <20ms）

同理，Skills 发现涉及四层目录扫描。可以：
- 缓存到 `.ccode/skills.cache`
- 监听 skills 目录变化时失效

### 5.3 MCP 连接池 / 预热

- 对 stdio 类型 MCP Server：启动子进程后保持连接，下次启动复用（daemon 模式）
- 对 HTTP 类型：DNS 预解析 + 连接预热（HTTP/2 连接复用）
- 超时机制：单个 Server 连接超过 10s 自动放弃，避免卡死

### 5.4 懒加载 Hooks

当前 Hooks 发现在启动时执行。如果 hooks 使用频率低，可以推迟到首次需要时才加载（类似 MCP 的后台策略）。但 SessionStartHooks 需要在首次 submit 前执行，所以收益有限。

### 5.5 切换 Bun Runtime（预估收益：整体提速 2-3 倍）

当前 dev 模式使用 Node + tsx（esbuild transpile），有两层隐性开销：

**Runtime 启动开销（dev timing 看不到的部分）：**

| 环节 | Node + tsx | Bun | 原因 |
|------|-----------|-----|------|
| TS → JS 转译 | ~300-500ms | 0ms | Bun 原生理解 TypeScript，零转译 |
| Runtime 初始化 | ~100ms | ~50ms | Bun 基于 JavaScriptCore，冷启动更快 |

**IO 密集型任务收益（Bun 底层 Zig 实现，fs/glob 比 Node libuv 快）：**

| 模块 | Node (当前) | Bun (预估) |
|------|------------|-----------|
| FileIndex 扫描 | 701ms | 200-400ms（原生 `Bun.glob` 可能更低） |
| Skills 扫描 | 262ms | 80-150ms |
| MCP 连接 | ~6000ms | ~6000ms（网络延迟，与 runtime 无关） |

**整体首帧渲染预估：**

| 指标 | Node + tsx | Bun |
|------|-----------|-----|
| 敲命令 → 首帧渲染 | ~1100ms | ~400-500ms |
| 体感 | 能感知到等待 | 几乎瞬开 |

**迁移成本评估：**
- 当前代码兼容性好：`fs.watch` recursive、`fast-glob`、`fzf`、`ignore` 在 Bun 上均兼容
- 渐进式优化路径：先切 runtime（零改动），再逐步替换为 Bun 原生 API
  - `fast-glob` → `Bun.glob`
  - `fs.readFile` → `Bun.file()`
  - `process.stdout.write` → `Bun.write(Bun.stdout, ...)`
- 风险点：Ink (React 终端渲染) 在 Bun 上的兼容性需要验证

### 优先级排序

| 优化项 | 收益 | 复杂度 | 建议优先级 |
|--------|------|--------|-----------|
| FileIndex 缓存 | 701ms → <50ms | 中 | P0 |
| Skills 缓存 | 262ms → <20ms | 低 | P1 |
| 切换 Bun Runtime | 整体 2-3x 提速 | 中 | P1 |
| MCP 超时/daemon | 避免卡死 | 高 | P2 |
| Bun 原生 API 替换 | 额外 30-50% IO 提速 | 低 | P2（切 Bun 后再做）|
| Hooks 懒加载 | ~5ms | 低 | P3（收益过小） |
