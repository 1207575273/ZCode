# Skills 二期 — 外置插件加载与 Hook 系统设计

## 一、目标

在 Skills 一期（内置三层发现 + SkillTool 工具调用）基础上，新增：

1. **外置插件包加载** — 兼容 superpowers 等 Claude Code 插件的目录结构，支持命名空间
2. **Hook 系统** — SessionStart / PreToolUse / PostToolUse 三种事件钩子，支持插件包自带 Hook 和用户自定义 Hook
3. **Skill 支撑文件元数据** — 加载 skill 时附带同目录支撑文件清单，方便 LLM 按需读取

---

## 二、外置插件包加载

### 2.1 目录结构约定

外置插件放在 **plugins 目录** 下，每个子目录是一个插件包：

```
~/.ccode/plugins/
  └── superpowers/            ← 插件包名
      ├── skills/             ← skills 目录（必需）
      │   ├── brainstorming/
      │   │   ├── SKILL.md
      │   │   ├── visual-companion.md
      │   │   └── spec-document-reviewer-prompt.md
      │   ├── writing-plans/
      │   │   └── SKILL.md
      │   └── ...
      ├── hooks/              ← hooks 目录（可选）
      │   ├── hooks.json
      │   └── session-start
      └── plugin.json         ← 插件清单（可选，用于显示信息）
```

项目级也可以放：`.ccode/plugins/<插件名>/`。

**安装方式**：用户手动 symlink 或复制，不做 marketplace。

```bash
# 示例：将 superpowers 项目链接为 cCli 插件
ln -s /path/to/superpowers ~/.ccode/plugins/superpowers
```

### 2.2 命名空间

插件包内的 skill 名自动添加前缀：`<插件名>:<原始name>`。

| 来源 | SKILL.md 中 name | cCli 注册名 |
|------|-------------------|-------------|
| builtin | `commit` | `commit` |
| user `~/.ccode/skills/` | `my-tool` | `my-tool` |
| project `.ccode/skills/` | `deploy` | `deploy` |
| plugin `~/.ccode/plugins/superpowers/` | `brainstorming` | `superpowers:brainstorming` |

**优先级规则**（同名时）：

```
project skill > user skill > plugin skill > builtin skill
```

非插件来源的 skill（builtin/user/project）不加命名空间，保持一期行为不变。

### 2.3 SkillMetadata 扩展

```typescript
export interface SkillMetadata {
  name: string                    // 注册名（可能含命名空间 "superpowers:brainstorming"）
  description: string
  filePath: string
  source: 'builtin' | 'user' | 'project' | 'plugin'  // 新增 'plugin'
  pluginName?: string             // 插件包名（仅 source='plugin' 时有值）
  allowedTools?: string[]
  userInvocable?: boolean
}
```

### 2.4 发现流程变更

SkillStore.discover() 扩展为四源扫描：

```
1. builtin  → src/skills/builtin/*/SKILL.md
2. plugin   → ~/.ccode/plugins/*/skills/*/SKILL.md   ← 新增
              .ccode/plugins/*/skills/*/SKILL.md      ← 新增（项目级插件）
3. user     → ~/.ccode/skills/*/SKILL.md
4. project  → .ccode/skills/*/SKILL.md
```

合并时按上述优先级，后者覆盖前者。plugin 层级在 builtin 之上、user 之下。

### 2.5 Skill 支撑文件元数据

当 `skill` 工具返回内容时，在 body 前附加上下文信息：

```xml
<skill-context name="superpowers:brainstorming" base-dir="/home/user/.ccode/plugins/superpowers/skills/brainstorming">
Supporting files (use Read tool to access):
- spec-document-reviewer-prompt.md
- visual-companion.md
- scripts/start-server.sh
- scripts/server.js
</skill-context>

(SKILL.md body 正文...)
```

**规则**：
- 列出 SKILL.md 同目录（含子目录）下的所有非 SKILL.md 文件
- 排除隐藏文件（`.` 开头）和 `node_modules`
- 使用相对路径展示，base-dir 提供绝对路径供 Read 工具拼接
- 非插件来源的 skill 如果目录下有支撑文件，同样附加

## 三、Hook 系统

### 3.1 事件类型

基础集，三种事件：

| 事件 | 触发时机 | 典型用途 |
|------|----------|----------|
| `SessionStart` | 会话启动、恢复、compact 后 | 插件启动注入、环境检查 |
| `PreToolUse` | 工具执行前 | 权限拦截、参数校验、日志 |
| `PostToolUse` | 工具执行后 | 结果增强、通知、审计 |

### 3.2 Hook 配置格式

```jsonc
// hooks.json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume|compact",   // 正则匹配触发子类型
        "hooks": [
          {
            "type": "command",
            "command": "bash ./hooks/session-start",
            "timeout": 10000    // 超时毫秒，默认 10s
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "bash|write_file",          // 匹配工具名
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"decision\": \"allow\"}'",
            "timeout": 5000
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "bash ./hooks/log-tool-use",
            "timeout": 5000
          }
        ]
      }
    ]
  }
}
```

### 3.3 Hook 来源与优先级

三层来源，同事件 **叠加执行**（不覆盖）：

| 优先级 | 来源 | 路径 |
|--------|------|------|
| 1（先执行） | 插件包 | `~/.ccode/plugins/<name>/hooks/hooks.json` |
| 2 | 项目级 | `.ccode/hooks.json` |
| 3（后执行） | 用户级 | `~/.ccode/hooks.json` |

同层同事件的多个 Hook 按数组顺序执行。

### 3.4 Hook 执行协议

#### SessionStart Hook

**输入**：环境变量
- `CCODE_SESSION_ID` — 当前会话 ID
- `CCODE_CWD` — 工作目录
- `CCODE_TRIGGER` — 触发子类型（`startup` / `resume` / `compact`）
- `CCODE_PLUGIN_ROOT` — 插件根目录（仅插件来源的 hook）

**输出**：JSON stdout，支持两个字段：
```jsonc
{
  // 注入到 LLM 对话上下文的附加内容
  "additionalContext": "...",
  // 输出到用户终端的提示信息（可选）
  "userMessage": "..."
}
```

`additionalContext` 会作为 system-reminder 注入到当前轮对话。

#### PreToolUse Hook

**输入**：环境变量 + stdin（JSON）
- `CCODE_TOOL_NAME` — 工具名
- `CCODE_TOOL_CALL_ID` — 调用 ID
- stdin：`{ "toolName": "bash", "args": { "command": "rm -rf /" } }`

**输出**：JSON stdout
```jsonc
{
  "decision": "allow" | "block" | "modify",
  // block 时必填
  "reason": "危险操作已阻止",
  // modify 时可选，替换原始参数
  "modifiedArgs": { ... }
}
```

缺省（无输出或非 JSON）= `allow`。

#### PostToolUse Hook

**输入**：环境变量 + stdin（JSON）
- `CCODE_TOOL_NAME` — 工具名
- `CCODE_TOOL_CALL_ID` — 调用 ID
- stdin：`{ "toolName": "bash", "result": { "success": true, "output": "..." } }`

**输出**：JSON stdout（可选）
```jsonc
{
  "additionalContext": "...",   // 追加到工具结果后的提示信息
  "userMessage": "..."          // 终端展示（可选）
}
```

### 3.5 错误处理

- Hook 超时：静默跳过，记录警告日志
- Hook 执行失败（非零退出码）：静默跳过，记录错误日志
- Hook 输出非法 JSON：视为无输出，静默跳过
- **Hook 永远不阻断主流程**（PreToolUse 的 block 决策除外）

### 3.6 SessionStart 与插件 Using-Skill 注入

Superpowers 的 SessionStart Hook 核心功能是注入 `using-superpowers` skill 的全文。cCli 的处理方式：

1. 发现插件包时，检查 `hooks/hooks.json` 是否存在
2. 如果有 SessionStart hook，执行其 command
3. 将 `additionalContext` 作为 system-reminder 注入
4. 同时，插件的 skills 通过正常的 SkillStore 发现流程注册

这样 superpowers 的 SessionStart Hook 可以原样运行，无需修改。

### 4.3 SkillStore 扩展伪码

```typescript
async discover(): Promise<SkillMetadata[]> {
  // 四源并行扫描
  const [builtinSkills, pluginSkills, userSkills, projectSkills] = await Promise.all([
    this.#scanDir(builtinSkillsDir(), 'builtin'),
    this.#scanPlugins(),    // ← 新增
    this.#scanDir(userSkillsDir(), 'user'),
    this.#scanDir(projectSkillsDir(), 'project'),
  ])

  // 按优先级合并：后者覆盖前者
  for (const meta of [...builtinSkills, ...pluginSkills, ...userSkills, ...projectSkills]) {
    this.#skills.set(meta.name, meta)
  }
}

async #scanPlugins(): Promise<SkillMetadata[]> {
  const results: SkillMetadata[] = []
  // 扫描 ~/.ccode/plugins/ 和 .ccode/plugins/ 下的每个插件包
  for (const pluginDir of pluginDirs) {
    const pluginName = basename(pluginDir)
    const skills = await this.#scanDir(join(pluginDir, 'skills'), 'plugin')
    for (const skill of skills) {
      skill.name = `${pluginName}:${skill.name}`   // 添加命名空间
      skill.pluginName = pluginName
      results.push(skill)
    }
  }
  return results
}
```

## 五、兼容性设计

### 5.1 与 Superpowers 的兼容

| Superpowers 机制 | cCli 对应 |
|------------------|-----------|
| `.claude-plugin/plugin.json` | 读取但仅用于展示信息，不是必需文件 |
| `skills/*/SKILL.md` | SkillStore 直接扫描，完全兼容 |
| `hooks/hooks.json` → SessionStart | HookManager 执行，additionalContext 注入 |
| `hooks/run-hook.cmd` | cCli 直接读 hooks.json 中的 command 执行，不走 run-hook.cmd |
| Skill 引用 `superpowers:xxx` | 命名空间天然匹配 |

**用户安装 superpowers 只需一步**：
```bash
ln -s /path/to/superpowers ~/.ccode/plugins/superpowers
```

### 5.2 一期行为不变

- builtin/user/project 三层 skill 无命名空间，行为完全不变
- 现有 `skill({ name: "commit" })` 调用正常工作
- system prompt 中的 skills 列表格式兼容
