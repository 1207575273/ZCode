## 四、踩坑记录

### 1. Anthropic stream 不能提前 break

**现象**：Anthropic 协议测试报 `request ended without sending any chunks`

**根因**：AnthropicProvider 的流程是：
```
for await (event of stream) → yield text chunks
stream.finalMessage() → yield usage + tool_calls + done
```
测试代码用 `break` 提前退出 `for await` → generator 被终止 → `finalMessage()` 永远不会执行 → SDK 报错。

**修复**：去掉 `break`，让流完整消费完。对于测试场景（`maxTokens: 32`），流很快就结束。

### 2. Provider error chunk 被忽略

**现象**：OpenRouter 测试显示"未收到有效响应"，看不到具体错误

**根因**：OpenAI 兼容 provider 出错时 yield `{ type: 'error' }` chunk，不会 throw。测试代码只检查了 `text` 和 `done`，漏了 `error`。

**修复**：加上 `if (chunk.type === 'error') streamError = chunk.error` 并返回给前端。

### 3. exactOptionalPropertyTypes 与 baseURL

**现象**：`baseURL: provCfg.baseURL`（可能是 undefined）传给 Provider 构造函数报 TS 错误

**修复**：用条件展开 `...(provCfg.baseURL ? { baseURL: provCfg.baseURL } : {})`

## 五、供应商协议选择指南

| 供应商 | 协议 | Base URL | 说明 |
|--------|------|----------|------|
| Anthropic 官方 | `anthropic` | 不填（默认） | 直连 Anthropic API |
| OpenAI 官方 | `openai` | 不填（默认） | 直连 OpenAI API |
| 智谱 GLM | `openai` | `https://open.bigmodel.cn/api/coding/paas/v4` | OpenAI 兼容协议 |
| DeepSeek | `openai` | `https://api.deepseek.com` | OpenAI 兼容协议 |
| MiniMax | `anthropic` | `https://api.minimax.chat/v1` | Anthropic 兼容协议 |
| OpenRouter | `openai` | `https://openrouter.ai/api/v1` | OpenAI 兼容协议（即使代理 Claude 模型） |
| Ollama 本地 | `openai` | `http://localhost:11434/v1` | OpenAI 兼容协议 |

**判断标准**：看供应商的 API 接口格式，不看它背后跑的是什么模型。
OpenRouter 虽然代理了 Claude，但它的接口是 `/v1/chat/completions`（OpenAI 格式），所以选 `openai`。

## 六、API Key 特征速查表

配置时容易把不同供应商的 Key 搞混（尤其是代理平台），通过前缀可以快速识别：

| 供应商 | Key 前缀 | 示例 | 说明 |
|--------|----------|------|------|
| Anthropic | `sk-ant-api` | `sk-ant-api03-xxxx` | 官方 Claude API |
| OpenAI | `sk-` | `sk-proj-xxxx` 或 `sk-xxxx` | 官方 GPT API |
| OpenRouter | `sk-or-v1-` | `sk-or-v1-993xxxxx` | 多模型代理平台 |
| 智谱 GLM | 无固定前缀 | `xxxxxxx.yyyyyyyy` | 通常是 `.` 分隔的两段 |
| DeepSeek | `sk-` | `sk-xxxxxxxx` | 与 OpenAI 格式类似 |
| MiniMax | 无固定前缀 | 较长数字字母串 | — |

### 常见配错场景

| 现象 | 原因 |
|------|------|
| OpenRouter 报 `User not found` (401) | Key 无效/过期，或者误填了 Anthropic 的 key（`sk-ant-` 开头） |
| OpenRouter 报 `Missing Authentication header` | Base URL 不对（缺少 `/v1`），或 LangChain 未正确传递 apiKey |
| GLM 报 401 | Key 格式不对，GLM 的 key 是两段式（`id.secret`） |
| 测试通过但对话无内容 | Base URL 少了路径（如 `https://openrouter.ai/` 应为 `https://openrouter.ai/api/v1`） |

### Base URL 易错点

LangChain 底层的 OpenAI SDK 会在 Base URL 后拼接 `/chat/completions`，所以：

```
✅ https://openrouter.ai/api/v1     → 实际请求 https://openrouter.ai/api/v1/chat/completions
❌ https://openrouter.ai/api         → 实际请求 https://openrouter.ai/api/chat/completions（404）
❌ https://openrouter.ai/             → 实际请求 https://openrouter.ai/chat/completions（404）
```

填 Base URL 时**不要加** `/chat/completions` 后缀，SDK 会自动拼。

## 七、LangChain ChatOpenAI 认证传递

### 踩坑：configuration 对象与 apiKey 的关系

LangChain 的 `ChatOpenAI` 使用自定义 Base URL 时，`configuration` 对象会直接传给底层 OpenAI SDK client。
为确保 apiKey 被正确传递，必须在 `configuration` 里也显式包含 `apiKey`：

```typescript
// ✅ 正确：configuration 里包含 apiKey
new ChatOpenAI({
  apiKey: config.apiKey,
  model: request.model,
  configuration: { baseURL: config.baseURL, apiKey: config.apiKey }
})

// ❌ 可能丢失认证：configuration 里没有 apiKey
new ChatOpenAI({
  apiKey: config.apiKey,
  model: request.model,
  configuration: { baseURL: config.baseURL }
})
```

原因：`ChatOpenAI` 的 `apiKey` 参数映射到内部 `openAIApiKey`，
但 `configuration` 直接覆盖底层 SDK 的 client 配置，
如果 `configuration` 里没有 `apiKey`，SDK client 可能拿不到认证信息。
