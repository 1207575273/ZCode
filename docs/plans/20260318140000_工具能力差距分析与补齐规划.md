## 三、差距分析

### 3.1 完全缺失的能力（红色）

| 缺失 | Claude Code 工具 | 影响 | 优先级 |
|------|-----------------|------|--------|
| **任务规划可视化** | TodoWrite | Agent 无法显式展示任务分解计划，用户看不到"接下来要做什么" | **P0** |
| **后台任务输出读取** | TaskOutput | bash run_in_background 启动的任务无法读取实时输出 | **P1** |
| **图片/PDF 读取** | FileRead (image/pdf) | 无法处理截图、设计稿、PDF 文档 | P2 |

### 3.2 功能不完整的能力（黄色）

| 差距 | Claude Code 能力 | cCli 当前 | 优先级 |
|------|-----------------|----------|--------|
| **Agent 后台运行** | `run_in_background: true` + `TaskOutput` 读取 | 不支持 | **P1** |
| **Agent 模型选择** | `model: "sonnet" / "opus" / "haiku"` | 子 Agent 继承父 Agent 模型 | P2 |
| **Agent Worktree 隔离** | `isolation: "worktree"` | 不支持 | P2 |
| **Agent 权限模式** | `mode: "acceptEdits" / "bypassPermissions" / "plan"` | 主 Agent 派发即授权 | P2 |

### 3.3 不需要补的能力

| Claude Code 工具 | 理由 |
|-----------------|------|
| WebFetch / WebSearch | 已通过 MCP 扩展覆盖（web-search-prime, deepwiki） |
| NotebookEdit | Jupyter 场景非核心（Python 生态可通过 MCP 扩展） |
| Config | 已有 Web Dashboard 设置管理 + `/model` 指令 |
| EnterWorktree / ExitWorktree | P2，后续与 Agent 隔离一起做 |
| ExitPlanMode | 与 TodoWrite 一起做 |
| SubscribePolling | MCP 高级能力，非核心 |
| MCP 资源管理 | 当前 MCP 工具调用已覆盖核心场景 |

## 四、TodoWrite 与 Agent 的关系（重要概念澄清）

### 4.1 两个独立工具

```
TodoWrite = 任务看板（纯数据 + UI 展示，追踪进度）
Agent     = 任务执行者（有自己的 AgentLoop，实际干活）
```

**两者正交，互不依赖。** TodoWrite 不知道 Agent 的存在，Agent 也不知道 TodoWrite 的存在。是 LLM 的推理将两者关联——LLM 决定"完成了某步，该更新 TodoWrite 了"。

### 4.2 场景一：不走 SubAgent，主 Agent 自己干

```
用户: "帮我重构这个模块"

LLM 调用 TodoWrite → 创建任务计划:
  📋 任务计划
  ⬜ 1. 读取现有代码
  ⬜ 2. 分析依赖关系
  ⬜ 3. 拆分函数
  ⬜ 4. 写测试

主 Agent 自己开始执行:
  → 调 read_file                → LLM 调 TodoWrite 更新 task 1 → ✅
  → 调 grep 分析依赖            → LLM 调 TodoWrite 更新 task 2 → ✅
  → 调 edit_file 拆分函数       → LLM 调 TodoWrite 更新 task 3 → 🔄
  → 调 bash 跑测试              → LLM 调 TodoWrite 更新 task 4 → ✅
```

**TodoWrite 在这里是"进度看板"**——LLM 主动汇报自己做到哪了，用户实时看到。

### 4.3 场景二：走 SubAgent，分派任务

```
用户: "帮我重构这 3 个文件"

LLM 调用 TodoWrite → 创建任务计划:
  📋 任务计划
  ⬜ 1. 重构 foo.ts
  ⬜ 2. 重构 bar.ts
  ⬜ 3. 重构 baz.ts

LLM 调用 dispatch_agent("重构 foo.ts")
  → 子 Agent 独立执行 → 完成
  → 主 Agent 调 TodoWrite 更新 task 1 → ✅

LLM 调用 dispatch_agent("重构 bar.ts")
  → 子 Agent 独立执行 → 完成
  → 主 Agent 调 TodoWrite 更新 task 2 → ✅

LLM 调用 dispatch_agent("重构 baz.ts")
  → 子 Agent 独立执行 → 完成
  → 主 Agent 调 TodoWrite 更新 task 3 → ✅
```

**TodoWrite 在这里是"任务编排看板"**——主 Agent 规划，子 Agent 执行，主 Agent 更新状态。子 Agent 完全不知道 TodoWrite 的存在。

### 4.4 架构模式映射

| 模式 | 对应 | 关系 |
|------|------|------|
| **Planning** | TodoWrite | 任务分解可视化 |
| **ReAct** | AgentLoop | 边想边做的执行循环 |
| **Tool Use** | 原子工具 | 实际操作 |
| **Multi-Agent** | dispatch_agent | 任务派发给子 Agent |
| **Reflection** | 未实现 | 自检环节 |

**TodoWrite 是 Planning 模式的落地工具**。它让 LLM 的隐式规划变成显式的、可视化的、可追踪的任务列表。不管是主 Agent 自己干还是派发子 Agent，TodoWrite 都只是"看板"——记录什么要做、做到哪了。

### 4.5 为什么这个理解很重要

实现 TodoWrite 时**不能**把它和 dispatch_agent 耦合。它就是一个独立的状态管理工具：

```typescript
// 输入：全量覆盖 todo 列表
interface TodoWriteArgs {
  todos: { content: string; status: 'pending' | 'in_progress' | 'completed' }[]
}

// 输出：更新前后的差异
interface TodoWriteResult {
  oldTodos: Todo[]
  newTodos: Todo[]
}
```

LLM 会在以下时机自主调用 TodoWrite：
- **任务开始时**：创建初始计划
- **完成一个步骤后**：更新对应 task 的状态
- **遇到变化时**：调整计划（新增/删除/重排 task）
- **全部完成时**：所有 task 标记为 completed

这些都是 LLM 自主决策的，不需要代码层面编排。

## 五、补齐规划（原第四章内容）

### 4.1 Phase 1: TodoWrite — 任务规划工具（P0）

**目标**：让 Agent 能显式创建和管理任务计划，CLI 和 Web 都能展示任务列表。

**Claude Code 的 TodoWrite**：

```typescript
interface TodoWriteInput {
  todos: {
    content: string                                    // 任务描述
    status: "pending" | "in_progress" | "completed"   // 三态
    activeForm: string                                 // 当前活跃表单
  }[]
}

interface TodoWriteOutput {
  oldTodos: Todo[]     // 更新前
  newTodos: Todo[]     // 更新后
  verificationNudgeNeeded?: boolean
}
```

**cCli 的 TodoWrite 设计**：

```typescript
// src/tools/todo-write.ts

interface TodoItem {
  id: string
  content: string
  status: 'pending' | 'in_progress' | 'completed'
}

interface TodoWriteArgs {
  todos: TodoItem[]  // 全量覆盖
}
```

**存储**：内存中维护，不持久化（session 级生命周期）。通过 EventBus 广播状态变更，CLI 和 Web 实时渲染。

**CLI 渲染**：

```
📋 任务计划 (2/5 完成)
  ✅ 1. 读取配置文件
  ✅ 2. 分析当前架构
  🔄 3. 编写新模块          ← 当前
  ⬜ 4. 编写测试
  ⬜ 5. 更新文档
```

**Web 渲染**：进度条 + checkbox 列表，与 CLI 实时同步。

**新增事件**：

```typescript
// AgentEvent
| { type: 'todo_update'; todos: TodoItem[] }

// BridgeEvent
| { type: 'todo_update'; todos: TodoItem[] }
```

**改动文件**：

| 文件 | 改动 |
|------|------|
| `src/tools/todo-write.ts` | 新增工具 |
| `src/core/agent-loop.ts` | AgentEvent 加 `todo_update` |
| `src/core/bootstrap.ts` | 注册 TodoWriteTool |
| `src/ui/useChat.ts` | 消费 `todo_update` 事件，维护 todos 状态 |
| `src/ui/TodoPanel.tsx` | 新增 CLI 渲染组件 |
| `src/ui/App.tsx` | 显示 TodoPanel |
| `web/src/types.ts` | 加 `todo_update` 事件 |
| `web/src/pages/ChatPage.tsx` | 渲染任务列表 |
| `src/core/event-bus.ts` | BridgeEvent 加 `todo_update` |
| `src/server/bridge/client.ts` | 透传 `todo_update` |

**预估工作量**：3-4 小时

### 4.2 Phase 2: TaskOutput — 后台任务输出读取（P1）

**目标**：补全 bash `run_in_background` 的闭环——启动后台任务后可以读取其输出。

**Claude Code 的 TaskOutput**：

```typescript
interface TaskOutputInput {
  task_id: string     // 任务 ID
  block: boolean      // 是否阻塞等待
  timeout: number     // 最大等待时间(ms)
}
```

**cCli 设计**：

```typescript
// src/tools/task-output.ts

interface TaskOutputArgs {
  pid: number          // 进程 PID（复用 process-tracker 的 ID）
  block?: boolean      // 是否等待完成
  timeout?: number     // 超时(ms)
}
```

**核心改动**：
- `process-tracker.ts` 需要改造——后台进程不能用 `stdio: 'ignore'`，要用管道捕获 stdout/stderr
- 新增 `task-output.ts` 工具——从 tracker 读取已捕获的输出

**预估工作量**：2-3 小时

### 4.3 Phase 3: Agent 能力增强（P2）

| 能力 | 改动 | 工作量 |
|------|------|--------|
| 模型选择 | dispatch_agent 加 `model` 参数，子 Agent 可指定不同模型 | 1h |
| 后台运行 | dispatch_agent 加 `run_in_background`，复用 TaskOutput 读取结果 | 2h |
| Worktree 隔离 | 新增 git worktree 管理模块 | 4h |
| 权限模式 | 子 Agent 支持 `mode` 参数控制权限策略 | 2h |

### 4.4 Phase 4: 图片/PDF 读取（P2）

- FileRead 支持图片返回 base64
- FileRead 支持 PDF 文本提取
- 需要新增依赖（pdf-parse 或 pdfjs-dist）
