# 多行输入支持实现总结

> 日期: 2026-03-22
> 状态: 已完成

---

## 一、问题

cCli 的输入框只支持单行输入，按 Enter 直接提交。用户无法输入多行文本（如粘贴代码片段、写多段描述）。

Claude Code CLI 支持通过 Shift+Alt+Enter 换行，输入区域自动展开为多行。

---

## 二、根因分析

### 终端按键识别的限制

大多数终端对 `Enter` 和 `Shift+Enter` 发送**同一个字节序列** `\r`，代码层面无法区分。

但 `Alt+Enter` 可以区分——Alt 键会在按键前追加 ESC 字节（`\x1b\r`），终端能正确传递。

| 按键组合 | 终端字节 | 可区分 |
|----------|----------|--------|
| Enter | `\r` | — |
| Shift+Enter | `\r` | ❌ 与 Enter 相同 |
| Alt+Enter | `\x1b\r` | ✅ |
| Shift+Alt+Enter | `\x1b\r` | ✅ |

### 原有组件的限制

原来使用的 `ink-text-input` 是**单行输入组件**，没有多行能力，也没有 keyBindings 配置。

---

## 三、解决方案

### 替换为 ink-multiline-input

[ink-multiline-input](https://www.npmjs.com/package/ink-multiline-input) 是专门为 Ink 设计的多行文本输入组件，原生支持：

- 多行编辑（自动展开行数）
- 方向键导航（上下左右）
- 自动滚动（超出 maxRows 时）
- **可配置的 keyBindings**（自定义 submit 和 newline 的按键）

### 按键配置

```typescript
<MultilineInput
  keyBindings={{
    // Enter = 提交
    submit: (key) => key.return && !key.meta && !key.shift,
    // Alt+Enter 或 Shift+Alt+Enter = 换行
    newline: (key) => key.return && key.meta,
  }}
  rows={1}       // 默认 1 行（单行外观）
  maxRows={10}   // 最多展开 10 行
/>
```

## 六、Home/End 键支持

> 日期: 2026-03-23
> 状态: 已完成

### 问题

多行输入场景下，用户期望按 Home 键跳到**当前行**行首、按 End 键跳到**当前行**行尾。
但 Ink 的标准 `useInput` 不暴露 `home`/`end` 布尔属性，这两个按键被直接丢弃。

### 根因深入分析

#### 1. Ink 的按键解析链

Ink 底层 `parseKeypress` 能正确识别 Home/End（解析为 `name === "home"/"end"`），
但 `useInput` 构造的 Key 接口**没有 `home`/`end` 字段**。
更关键的是，`nonAlphanumericKeys` 包含 `"home"/"end"`，
导致 `useInput` 将 `input` 清空为 `''`，所有 key 布尔值也为 `false`——
Home/End 在 `useInput` 层面变成了一个**不可区分的"空事件"**。

#### 2. 跨平台转义序列差异（核心踩坑点）

不同终端 / 操作系统对 Home/End 发送的字节序列完全不同：

| 环境 | Home 序列 | End 序列 | 备注 |
|------|-----------|----------|------|
| Linux/Mac xterm | `\x1b[H` (1b5b48) | `\x1b[F` (1b5b46) | ANSI 标准 |
| rxvt | `\x1b[1~` / `\x1b[7~` | `\x1b[4~` / `\x1b[8~` | rxvt 变体 |
| xterm/gnome | `\x1bOH` | `\x1bOF` | Application mode |
| **Windows CMD/PowerShell** | **`e047` / `0047`** | **`e04f` / `004f`** | **非 ANSI，原生扫描码** |
| Windows Terminal | `\x1b[H` (1b5b48) | `\x1b[F` (1b5b46) | 兼容 ANSI |

**Windows 原生扫描码 `e047`/`0047` 是关键盲区**——
所有基于 ANSI 字符串匹配的方案在 Windows CMD/PowerShell 下都会失败，
必须同时匹配 **hex 编码**才能覆盖。

#### 3. 失败方案历程

| 方案 | 失败原因 |
|------|----------|
| 循环发送方向键 `handler('', LEFT_KEY)` × N | React 批量更新，闭包捕获 stale state，只移动 1 格 |
| `stdin.on('data')` 拦截 | Ink 可能使用 paused mode（readable + read()），data 事件不一定触发 |
| `internal_eventEmitter` hook | 通过 `useStdin()` 取到的上下文无法可靠访问内部 emitter |
| monkey-patch `stdin.read()` | 可行但侵入性强，且 Windows 扫描码匹配不全 |
| 只匹配 ANSI 字符串序列 | 遗漏 Windows 原生扫描码（`e047` 等），导致 Windows 环境不生效 |

### 最终方案

#### 架构：ControlledMultilineInput + 自管 cursor + stdin.prependListener

```
stdin raw data ──prependListener──→ 拦截 Home/End（hex + string 双重匹配）
                                         ↓
                                  setCursorIndex(行首/行尾)
                                         ↓
Ink useInput ─────────────────→ 空事件守卫过滤 ──→ 常规按键处理
```

1. **`ControlledMultilineInput`**：纯展示组件，接受外部 `cursorIndex` prop，
   自己管理全部键盘逻辑和 cursor 状态
2. **`stdin.prependListener('data')`**：在 Ink 处理之前拦截原始字节流
3. **双重匹配**：同时检查 `data.toString('hex')` 和 `data.toString('utf8')`，
   覆盖 ANSI 转义序列 + Windows 原生扫描码
4. **Ctrl+A / Ctrl+E** 作为兜底快捷键（走 useInput 通道，100% 可靠）

#### 核心拦截代码

```typescript
const onData = (data: Buffer) => {
  const hex = data.toString('hex')
  const str = data.toString('utf8')

  // 双重匹配：ANSI 字符串 + Windows 原生扫描码 hex
  const isHome =
    str === '\x1b[H' || str === '\x1b[1~' || str === '\x1b[7~' || str === '\x1bOH' ||
    hex === 'e047' || hex === '0047' || hex === '1b5b48'

  const isEnd =
    str === '\x1b[F' || str === '\x1b[4~' || str === '\x1b[8~' || str === '\x1bOF' ||
    hex === 'e04f' || hex === '004f' || hex === '1b5b46'

  if (isHome) {
    // 向前找最近的换行符 → 当前行首
    const lastNewline = findLastNewline(val, cur - 1)
    setCursorIndex(lastNewline === -1 ? 0 : lastNewline + 1)
  } else if (isEnd) {
    // 向后找最近的换行符 → 当前行尾
    const nextNewline = findNextNewline(val, cur)
    setCursorIndex(nextNewline === -1 ? val.length : nextNewline)
  }
}

stdin.prependListener('data', onData)
```

#### 行首/行尾计算（同时兼容 `\n` 和 `\r`）

```typescript
function findLastNewline(str: string, fromIdx: number): number {
  return Math.max(str.lastIndexOf('\n', fromIdx), str.lastIndexOf('\r', fromIdx))
}

function findNextNewline(str: string, fromIdx: number): number {
  const n = str.indexOf('\n', fromIdx)
  const r = str.indexOf('\r', fromIdx)
  if (n === -1) return r
  if (r === -1) return n
  return Math.min(n, r)
}
```

### 经验总结

| 教训 | 说明 |
|------|------|
| Windows 扫描码≠ANSI | Windows CMD/PowerShell 的 Home/End 发送 `e047`/`e04f` 等原生扫描码，不是标准 ANSI 转义序列。**必须同时匹配 hex 编码和字符串** |
| React 状态批量更新 | 同步循环调用 setState 在闭包里只读到同一个 stale value，实际只生效 1 次 |
| Ink useInput 的盲区 | Key 接口缺少 home/end 字段，且 input 被清空为 ''。只能绕过 useInput 直接操作 stdin |
| 用 Controlled 组件 | 当需要精确控制内部状态时，用 `ControlledMultilineInput` 而非 hack 黑盒 `MultilineInput` |
| stdin.prependListener | 确保在 Ink 内部 handler 之前触发，不影响 Ink 正常处理 |
| ref 保存最新值 | stdin listener 通过 `valueRef` / `cursorRef` 读取最新状态，避免频繁重建 listener |
| 双通道兜底 | Home/End（stdin 拦截）+ Ctrl+A/Ctrl+E（useInput），确保所有终端环境至少有一条路径可用 |

## 八、被放弃的方案

| 方案 | 放弃原因 |
|------|----------|
| 拦截 Shift+Enter | 终端不区分 Shift+Enter 和 Enter，发送相同字节 |
| 反斜杠续行（`\` + Enter） | 可行但不直观，用户体验差 |
| 手动 useInput 拦截 + TextInput 拼接 | 复杂，渲染不自然，且 TextInput 本身是单行 |
| Kitty 键盘协议 | 仅部分终端支持（iTerm2/Kitty/WezTerm），Windows Terminal 不支持 |
| 循环发送方向键 × N | React 状态批量更新导致只移动 1 格 |
| `internal_eventEmitter` hook | 无法通过 `useStdin()` 可靠访问 Ink 内部 emitter |
| monkey-patch `stdin.read()` | 侵入性强，且仍然遗漏 Windows 扫描码 |
| 只匹配 ANSI 字符串 | 遗漏 Windows 原生扫描码（`e047` 等） |
| `keyHandledRef` 状态同步 | stdin 和 useInput 之间的竞态导致下一个按键丢失 |
