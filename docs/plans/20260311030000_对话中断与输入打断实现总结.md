# F18 对话中断与输入打断 — 实现总结

## 1. 实现目标

LLM 响应期间，用户可通过以下方式中断：
1. **Escape / Ctrl+C** — 立即中止当前流式响应
2. **输入新消息并 Enter** — 中断当前响应并开启新一轮对话

中断后：
- 被打断的部分回复保留在消息历史和 UI 中
- 按 Claude Code 模式记录 `[Request interrupted by user]` 到 session 日志
- 双击 Ctrl+C（2 秒内）才退出程序

---

## 2. 核心架构

```
用户按 Escape / Ctrl+C
  │
  ▼ App.tsx useInput hook（始终活跃，内部判断 isStreaming）
  │
  ▼ abort() → AbortController.abort()
  │
  ▼ LangChain Provider 捕获 abort
  │
  ├── 路径 A（直接 throw）: catch 直接捕获 AbortError
  └── 路径 B（yield error chunk）: error chunk 含 "aborted" → 重新 throw AbortError
  │
  ▼ agent-loop catch(isAbortError) → yield llm_usage(stopReason: 'abort') → throw
  │
  ▼ useChat catch(isAbortError)
  │
  ├── 保存 accumulated 部分回复到 messages + session 日志
  ├── 记录 [Request interrupted by user] 到 session 日志
  └── finally → setIsStreaming(false)

用户在 streaming 期间输入新消息并 Enter
  │
  ▼ InputBar handleSubmit → onInterruptSubmit(text)
  │
  ▼ App.tsx → interruptAndSubmit(text)
  │
  ├── abort() — 中止当前流
  └── setTimeout(() => submit(text), 0) — 等状态更新后发起新请求
```

## 4. 关键实现细节

### 4.1 isAbortError — 健壮的 abort 检测

```typescript
export function isAbortError(err: unknown): boolean {
  if (!(err instanceof Error)) return false
  if (err.name === 'AbortError') return true
  if (err.message.toLowerCase().includes('aborted')) return true
  return false
}
```
- Node.js 原生 fetch 抛 `name='AbortError'`
- LangChain 等库可能包装为普通 Error，`message` 含 "aborted" 但 `name` 不是 `'AbortError'`
- 两者都需要覆盖

### 4.2 error chunk → 重新 throw

```typescript
// agent-loop.ts #callLLM
if (mapped.type === 'error') {
  const errorMsg = chunk.error ?? 'unknown error'
  // Provider 将 abort 包装为 error chunk → 重新抛出
  if (errorMsg.toLowerCase().includes('aborted')) {
    const abortErr = new Error(errorMsg)
    abortErr.name = 'AbortError'
    throw abortErr
  }
  yield makeLlmError(errorMsg, outputTokens)
  yield mapped
  return { toolCalls: [], aborted: true }
}
```

**为什么需要这个**：LangChain provider 可能捕获 abort 后不 throw，而是 yield `{ type: 'error' }` chunk。agent-loop 的 error chunk 路径会正常 return（不 throw），导致 useChat 走不到 catch 分支。

### 4.3 useChat 中断处理

```typescript
} catch (err) {
  if (isAbortError(err)) {
    if (accumulated) {
      setMessages(prev => [...prev, partialMsg])
      sessionLogger.logAssistantMessage(accumulated, currentModel)
    }
    sessionLogger.logUserMessage('[Request interrupted by user]')
  } else {
    setError(err instanceof Error ? err.message : String(err))
  }
}
```

### 4.4 双击 Ctrl+C 退出

```typescript
const lastCtrlCRef = useRef(0)
const DOUBLE_CTRLC_MS = 2000

useInput((input, key) => {
  const isCtrlC = input === 'c' && key.ctrl
  if (isStreaming && pendingPermission == null) {
    if (key.escape || isCtrlC) {
      abort()
      appendSystemMessage('⏹ 已中断响应')
    }
  } else if (isCtrlC && !isStreaming) {
    const now = Date.now()
    if (now - lastCtrlCRef.current < DOUBLE_CTRLC_MS) {
      exit()
    } else {
      lastCtrlCRef.current = now
      appendSystemMessage('再次 Ctrl+C 退出')
    }
  }
})
```

### 4.5 exitOnCtrlC: false

```typescript
// bin/zcli.ts
render(React.createElement(App, { ... }), { exitOnCtrlC: false })
```

Ink 默认 `exitOnCtrlC: true`，Ctrl+C 直接 `process.exit()`，`useInput` 根本收不到。必须禁用。

## 5. 设计决策

| # | 决策 | 理由 |
|---|------|------|
| 1 | `setTimeout(0)` 延迟 submit | abort 后 finally 块需要一个 tick 来更新 `isStreaming` |
| 2 | Escape 和 Ctrl+C 都支持 | Claude Code 两者都支持 |
| 3 | 全局 useInput 而非 isActive 条件 | 需要同时处理 streaming（中断）和空闲（双击退出）两种场景 |
| 4 | `streaming` 替代 `disabled` | 语义更清晰：不是"禁用"，是"可打断" |
| 5 | 部分回复保留在消息历史 | 与 Claude Code 一致 |
| 6 | `[Request interrupted by user]` | Claude Code 的 session 日志格式，LLM 下一轮能感知上轮被打断 |
| 7 | 双击 Ctrl+C 退出 | 防误触，参照 Claude Code 行为 |
| 8 | error chunk 重新 throw | LangChain 不保证 abort 一定 throw，可能 yield error chunk |

## 6. 踩坑记录

### 坑 1：Ink exitOnCtrlC 默认行为

**现象**：Ctrl+C 直接退出，useInput 的中断逻辑不生效。
**根因**：Ink 默认 `exitOnCtrlC: true`，在 raw mode 下拦截 Ctrl+C 直接 `process.exit()`。
**修复**：`render()` 传 `{ exitOnCtrlC: false }`。

### 坑 2：AbortError 不叫 AbortError

**现象**：中断后日志 `stopReason: "error"` 而不是 `"abort"`。
**根因**：LangChain 包装后的 abort 错误 `name` 不是 `'AbortError'`，而是普通 Error + `message: "This operation was aborted"`。
**修复**：`isAbortError()` 同时检查 `name` 和 `message`。

### 坑 3：Provider yield error chunk 而非 throw

**现象**：`isAbortError` 修复后中断仍未正确记录。日志仍无 `[Request interrupted by user]`。
**根因**：LangChain provider 捕获 abort 后不 throw，而是 yield `{ type: 'error', error: 'This operation was aborted' }` 作为 chunk。agent-loop 的 error chunk 处理路径正常 return `{ aborted: true }`（不 throw），run() 看到 `aborted=true` 就静默返回。useChat 的 `for await` 正常结束，catch(AbortError) 永远不执行。
**修复**：error chunk 包含 "aborted" 时重新 `throw new Error()` 并设置 `name='AbortError'`，使 catch 路径正确触发。
