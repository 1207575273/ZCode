# F17 并行工具执行 — 实现总结

## 1. 实现目标

LLM 单次回复返回多个 tool_calls 时，安全工具并行执行以缩短耗时，危险工具保持串行（需逐个权限确认）。

**性能收益**：3 个安全工具各 200ms → 串行 600ms → 并行 ~200ms（3x 加速）。

---

## 2. 核心架构

```
LLM 返回 toolCalls: [glob, grep, bash, read_file]
  │
  ▼ classifyToolCalls()
  │
  ├── safe:      [glob, grep, read_file]  ← 并行 Promise.allSettled
  └── dangerous: [bash]                   ← 串行 yield* executeOneTool
```

**分类规则**：

| 条件 | 分类 |
|------|------|
| 工具未注册 | dangerous |
| `registry.isDangerous()` 返回 true | dangerous |
| 已注册且 isDangerous 为 false | safe |

**内置工具分类**：

| safe（无副作用） | dangerous（有副作用/需确认） |
|-------------------|----------------------------|
| read_file, glob, grep | bash, write_file, edit_file |

## 4. 关键实现细节

### 4.1 parallel-executor.ts

两个核心导出函数：

**`classifyToolCalls(toolCalls, registry)`** — 纯函数，O(n) 单次遍历分组。

**`executeSafeToolsInParallel(toolCalls, registry, onEvent, ctx, maxParallel?)`**：
- `Promise.allSettled` 并行执行，单工具失败不影响其他工具
- `maxParallel` 默认 5，超过时分批执行（slice + allSettled）
- 通过 `onEvent` 回调收集 `tool_start` / `tool_done` 事件（不使用 yield*）
- 返回结果按原始 toolCalls 顺序排列（预分配数组 + originalIndex 回写）
- `resultSummary` 截断到 200 字符，追加 `...`

### 4.2 agent-loop.ts 变更

**AgentConfig 扩展**：
```typescript
interface AgentConfig {
  model: string
  provider: string
  signal?: AbortSignal | undefined
  parallelTools?: boolean | undefined    // 默认 true
  maxParallelTools?: number | undefined  // 默认 5
}
```

**新方法 `#executeToolCalls`**：
```
parallelTools === false → 全部串行（兼容模式）
parallelTools !== false → classifyToolCalls 分组
  → safe:      executeSafeToolsInParallel → 收集事件 → yield → history 追加
  → dangerous: 逐个 yield* #executeOneTool（保留权限交互）
```

### 4.3 事件流保证

- `tool_start` 在工具启动时立即发出（并行工具几乎同时）
- `tool_done` 在工具完成时发出（完成顺序不固定）
- 安全工具全部完成后，才开始执行危险工具
- history 按原始 toolCalls 顺序追加（保证 LLM 看到确定性结果）

### 4.4 ToolRegistry.has()

新增 `has(name: string): boolean`，基于内部 `Map.has()`，替代 `getAll().some()` 的 O(n) 检查。

## 5. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| 1 | 新建 `parallel-executor.ts` | 关注点分离，不膨胀 agent-loop.ts |
| 2 | 事件回调而非 yield* | Promise.allSettled 内部无法 yield |
| 3 | 未注册工具视为 dangerous | 安全优先，未知工具不自动并行 |
| 4 | history 按原始顺序追加 | LLM 看到确定性结果，避免不同运行产生不同上下文 |
| 5 | parallelTools 默认 true | 并行是明确的性能提升，默认启用 |
| 6 | 分批而非信号量 | slice + allSettled 实现简单，对工具场景足够 |
| 7 | 先并行安全组再串行危险组 | 最大化并行度，不打断权限交互 |

### 7.2 手动验证（Pipe Mode）

agent-loop.ts 内置了调试日志，当多工具触发时会在 stderr 输出分组信息。


观察 UI 中是否几乎同时出现多个工具的执行状态。

### 7.4 验证串行回退

在 `src/core/pipe-runner.ts` 中临时传入 `parallelTools: false`：

```typescript
const loop = new AgentLoop(provider, registry, {
  model: modelName,
  provider: providerName,
  signal: controller.signal,
  parallelTools: false, // 临时添加，验证串行模式
})
```

运行同样的多工具 prompt，确认工具逐个执行（无 `[parallel]` 日志）。

### 7.5 注意事项

- **并行触发取决于 LLM 决策**：不同模型对多工具调用的倾向不同。Claude 和 GPT-4o 较擅长一次返回多个 tool_calls，小模型可能倾向逐个调用。
- **调试日志**：`[parallel]` 日志仅在 2+ 工具时输出，单工具不打印。
- **stderr 输出**：调试日志写入 stderr 不影响 `--json` 的 stdout 结构化输出。
