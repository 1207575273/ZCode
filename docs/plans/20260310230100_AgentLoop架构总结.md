# AgentLoop 架构总结

## 概述

AgentLoop 是 cCli 的多轮执行引擎，负责 LLM ↔ 工具的循环调用。一次 `run()` 调用对应一次用户提问的完整处理过程，通过 AsyncGenerator 将所有中间状态 yield 出去，由调用方（useChat）和观察者（SessionLogger）各取所需。

**文件位置**：`src/core/agent-loop.ts`（约 240 行，单文件分区组织）

## AgentEvent 类型系统

### 业务事件（UI 消费）

| 类型 | 用途 | 关键字段 |
|-----|------|---------|
| `text` | LLM 流式文本 | `text` |
| `tool_start` | 工具开始执行 | `toolName`, `toolCallId`, `args` |
| `tool_done` | 工具执行完成 | `toolName`, `durationMs`, `success`, `resultSummary?` |
| `permission_request` | 危险工具等待用户确认 | `toolName`, `args`, `resolve` |
| `error` | 运行时错误 | `error` |
| `done` | 本次 run() 完成 | — |

### 观测事件（SessionLogger 消费）

| 类型 | 用途 | 关键字段 |
|-----|------|---------|
| `llm_start` | LLM 调用开始 | `provider`, `model`, `messageCount` |
| `llm_usage` | LLM 调用结束 | `inputTokens`, `outputTokens`, `stopReason` |
| `llm_error` | LLM 错误 | `error`, `partialOutputTokens?` |
| `tool_fallback` | 工具降级 | `toolName`, `fromLevel`, `toLevel`, `reason` |
| `permission_grant` | 权限授予 | `toolName`, `always` |

## 方法分解

### `run()` — 主循环（15 行）

```
for turn < MAX_TURNS:
  llmResult = yield* #callLLM(history)
  if aborted → return
  if no toolCalls → yield done, return
  for each toolCall:
    yield* #executeOneTool(tc, history)
```

- 使用 `yield*` 委托子方法的事件流
- history 数组在整个 run() 生命周期内共享引用
- MAX_TURNS 防止死循环

### `#callLLM()` — LLM 流式调用

**签名**：`AsyncGenerator<AgentEvent, { toolCalls: ToolCallContent[]; aborted: boolean }>`

利用 AsyncGenerator 的 return 值传递结构化结果：
- yield 出 `llm_start` → `text*` → `llm_usage | llm_error`
- return `{ toolCalls, aborted }` 给调用方

**事件流**：
```
yield llm_start
for await (chunk of provider.chat()):
  yield text / error (via #mapChunk)
yield llm_usage
return { toolCalls, aborted: false }
```

### `#mapChunk()` — 纯函数映射

将 Provider 的 StreamChunk 映射为 AgentEvent：
- `text` chunk → `text` event
- `tool_call` chunk → 收集到 pendingToolCalls（不产生事件）
- `error` chunk → `error` event
- `usage` / `done` → null（不产生业务事件）

### `#executeOneTool()` — 工具执行

```
yield tool_start
allowed = yield* #checkPermission(tc)
if !allowed → push rejection to history, yield tool_done(failed)
result = await registry.execute(...)
push result to history
yield tool_done
```

### `#checkPermission()` — 权限检查

```
if !isDangerous → return true
yield permission_request (含 resolve 回调)
allowed = await promise
if allowed → yield permission_grant
return allowed
```

利用 Promise + resolve 回调实现暂停等待：yield 出 `permission_request` 后，UI 层调用 `resolve(true/false)` 恢复执行。

## 关键设计模式

### 1. AsyncGenerator yield* 委托

run() 不直接处理 LLM 调用和工具执行的细节，而是通过 `yield*` 委托给子方法。子方法的 yield 事件直接穿透到调用方，同时通过 return 值传递结构化结果。

```typescript
// run() 中：
const llmResult = yield* this.#callLLM(history)
// llmResult.toolCalls 是 #callLLM 的 return 值
// 期间 #callLLM yield 的所有事件直接传给 run() 的消费者
```

### 2. 观测事件与业务事件分离

AgentEvent 联合类型包含两类事件：
- 业务事件（text/tool_start/done/error）→ UI 渲染
- 观测事件（llm_start/usage/error）→ SessionLogger 日志

两类事件走同一个事件流，消费者各自过滤。AgentLoop 不知道谁在消费，保持完全解耦。

### 3. History 引用共享

messages 数组通过引用传递给所有子方法。工具执行结果直接 push 到 history，无需返回再合并。

### 4. exactOptionalPropertyTypes 兼容

所有可选字段使用条件展开或三元表达式，避免赋值 `undefined`：

```typescript
// makeLlmError — 根据 partialTokens 构造不同对象形状
return partialTokens > 0
  ? { type: 'llm_error', error, partialOutputTokens: partialTokens }
  : { type: 'llm_error', error }
```

## 与 SessionLogger 的集成

AgentLoop 本身不依赖 SessionLogger。集成在 `useChat.ts` 中完成：

```typescript
// useChat.ts
for await (const event of agentLoop.run(messages)) {
  sessionLogger.consume(event)  // 观测
  // ... UI 处理
}
```

SessionLogger 的 `consume()` 按 event.type switch 映射为 SessionEvent，不关心事件来源。

## 演进方向

- **流式工具调用**：当前 tool_call chunk 收集完再执行，未来可支持流式参数
- **并行工具执行**：当前顺序执行 toolCalls，可优化为并行
- **重试策略**：llm_error 后可加入自动重试机制
- **上下文压缩**：history 超长时可触发摘要压缩
