## 一、Claude Code 插件目录结构逆向分析

### 1.1 顶层结构

```
~/.claude/plugins/
├── installed_plugins.json        ← 全局插件注册表（版本、路径、安装时间）
├── blocklist.json                ← 屏蔽列表
├── known_marketplaces.json       ← 已知的 marketplace 源
├── install-counts-cache.json     ← 安装计数缓存
├── marketplaces/                 ← marketplace 仓库（git clone）
│   └── claude-code-plugins/
│       └── .claude-plugin/marketplace.json
├── cache/                        ← 插件缓存（实际代码）
│   ├── claude-plugins-official/  ← marketplace 名
│   │   ├── context7/             ← 插件名
│   │   │   └── 78497c524da3/     ← 版本（git commit hash）
│   │   ├── github/
│   │   ├── frontend-design/
│   │   └── superpowers/
│   │       ├── 4.3.1/            ← 历史版本
│   │       ├── 5.0.0/
│   │       ├── 5.0.2/
│   │       └── 5.0.4/            ← 最新版本
│   └── temp_git_*/               ← 临时 git clone 目录
└── ...
```

## 二、关键发现

### 2.1 版本管理

- 同一插件保留**多个版本**（superpowers 有 4.3.1 / 5.0.0 / 5.0.2 / 5.0.4）
- `installed_plugins.json` 指向当前活跃版本的路径
- 版本标识：语义版本号（superpowers）或 git commit hash（context7/github）

### 2.2 Marketplace 机制

- 插件按 marketplace 分组：`cache/<marketplace>/<plugin>/<version>/`
- `known_marketplaces.json` 记录 marketplace 源地址
- `marketplaces/` 目录存放 marketplace 仓库的 git clone
- 安装过程：从 marketplace 仓库读取插件列表 → git clone 到 cache → 记录到 installed_plugins.json

### 2.3 兼容性评估

| Claude Code 特性 | cCli 当前状态 | 兼容度 |
|-----------------|-------------|--------|
| `skills/` 目录扫描 | ✅ `#scanPlugins()` 已实现 | 完全兼容 |
| `hooks/hooks.json` | ✅ `HookManager` 已实现 | 完全兼容 |
| `SKILL.md` 格式 | ✅ frontmatter 解析已实现 | 完全兼容 |
| `.claude-plugin/plugin.json` | 🔲 未读取 | 可选（不影响功能） |
| `installed_plugins.json` 注册表 | 🔲 无 | cCli 用目录扫描替代 |
| Marketplace 机制 | 🔲 无 | 不需要（手动管理） |
| 多版本共存 | 🔲 无 | 不需要（只保留一个版本） |
| `agents/` 目录 | 🔲 不支持 | 后续可扩展 |
| `commands/` 目录 | 🔲 不支持 | 后续可扩展 |

### 2.4 核心结论

**cCli 的 Skills + Hooks 已经完全兼容 Claude Code 的插件格式**。只需要把 `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/` 复制到 `~/.ccode/plugins/<plugin>/` 就能用。

缺失的是**管理能力**——没有 UI 来查看、导入、删除插件和 MCP。

## 三、cCli 插件/MCP 管理方案

### 3.1 设计原则

1. **不做 marketplace**——太重了，手动管理 + 从 Claude Code 导入足够
2. **不做多版本**——只保留一个版本，升级就是覆盖
3. **维护一个简单的注册表**——`~/.ccode/plugins.json`，记录安装来源和时间
4. **Web UI CRUD**——在 Dashboard 设置页新增 Plugins + MCP 管理 tab

### 3.2 注册表设计（`~/.ccode/plugins.json`）

```json
{
  "plugins": [
    {
      "name": "superpowers",
      "installPath": "~/.ccode/plugins/superpowers",
      "source": "claude-code",
      "sourceDetail": "~/.claude/plugins/cache/claude-plugins-official/superpowers/5.0.4",
      "version": "5.0.4",
      "installedAt": "2026-03-17T12:00:00Z",
      "skillCount": 14,
      "hasHooks": true
    },
    {
      "name": "frontend-design",
      "installPath": "~/.ccode/plugins/frontend-design",
      "source": "manual",
      "version": "unknown",
      "installedAt": "2026-03-17T12:00:00Z",
      "skillCount": 1,
      "hasHooks": false
    }
  ]
}
```

### 3.4 后端 API

```
# Plugins
GET  /api/plugins                    ← 已安装插件列表
GET  /api/plugins/claude-available   ← 从 Claude Code 可导入的插件
POST /api/plugins/import-claude      ← 从 Claude Code 导入（复制目录）
POST /api/plugins/delete             ← 删除插件

# MCP
GET  /api/mcp/servers                ← 已配置的 MCP Server 列表（合并三层来源）
POST /api/mcp/servers/add            ← 添加 MCP Server（写入 ~/.ccode/.mcp.json）
POST /api/mcp/servers/delete         ← 删除 MCP Server
```

### 3.5 导入 Claude Code 插件的核心逻辑

```typescript
// 1. 读取 ~/.claude/plugins/installed_plugins.json
// 2. 解析每个插件的 installPath + version
// 3. 提供选择界面
// 4. 用户确认后 cp -r installPath → ~/.ccode/plugins/<pluginName>/
// 5. 写入 ~/.ccode/plugins.json 注册表
```

不需要处理 marketplace 协议、git 操作、版本比较——就是**读已安装列表 + 复制目录**。

## 四、实施优先级

| 优先级 | 功能 | 工作量 |
|--------|------|--------|
| P0 | Plugins 管理 tab（已安装列表 + 删除） | 小 |
| P0 | 从 Claude Code 导入（读 installed_plugins.json + 复制） | 中 |
| P1 | MCP 管理 tab（列表 + 添加 + 删除） | 中 |
| P2 | 从 Git 仓库克隆插件 | 中 |
| P2 | 项目级插件/MCP 显示（只读） | 小 |
| P3 | plugins.json 注册表 | 小 |
