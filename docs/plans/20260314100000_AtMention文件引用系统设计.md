# @ Mention 文件引用系统设计

## 一、需求概述

在用户输入中支持 `@` 引用项目文件，提供自动补全建议，将文件路径信息传递给 LLM，由 LLM 自行决定是否使用 Read 工具加载内容。

**核心行为**：
```
用户输入: @src/hooks/types.ts 这个文件的设计思路是什么
                    ↓ 客户端预处理
LLM 收到: "用户引用了文件 src/hooks/types.ts（绝对路径: D:\...\src\hooks\types.ts）。\n\n@src/hooks/types.ts 这个文件的设计思路是什么"
                    ↓ LLM 自主决策
LLM 调用 Read 工具读取文件 → 回答问题
```

## 二、架构总览

```
┌─────────────────────────────────────────────────────────────┐
│                        @ Mention 系统                        │
├──────────────┬──────────────┬───────────────┬───────────────┤
│  FileIndex   │  FileWatcher │  AtResolver   │  AtSuggestion │

├──────────────┼──────────────┼───────────────┼───────────────┤

│ .zcliignore  │              │ context 注入   │ frecency 排序 │
└──────────────┴──────────────┴───────────────┴───────────────┘
```

## 三、模块设计

### 3.1 FileIndex — 文件索引器

**职责**：启动时扫描项目目录树，维护一个扁平的相对路径列表。

```typescript
// src/file-index/file-index.ts

export class FileIndex {
  /** 相对路径列表（排序后），内存中只存字符串数组 */
  private paths: string[] = []
  /** ignore 实例，管理排除规则 */
  private ig: Ignore

  constructor(private cwd: string) {}

  /** 首次全量扫描 */
  async scan(): Promise<void>

  /** 增量更新：添加/删除路径 */
  add(relativePath: string): void
  remove(relativePath: string): void

  /** 模糊搜索，返回匹配结果（带评分） */
  search(query: string, limit?: number): SearchResult[]

  /** 获取全部路径（用于调试/导出） */
  getAll(): string[]

  /** 路径总数 */
  get size(): number
}

interface SearchResult {
  path: string       // 相对路径
  score: number      // 匹配分数
  positions: number[] // 高亮位置
}
```

**扫描策略**：
- 使用 `fast-glob` 的 Stream API，内存友好
- 只收集文件路径，不读内容
- 存储为 `string[]`（相对路径），10 万文件约 10MB 内存

**排除规则优先级**（高覆盖低）：
1. 内置硬编码：`.git`, `node_modules`, `.next`, `dist`, `build`, `__pycache__`, `.zcli`
2. `.gitignore`（项目根目录 + 递归子目录）
3. `.zcliignore`（项目根目录，语法同 .gitignore）

### 3.2 FileWatcher — 变更监听器

**职责**：监听文件系统变更，增量更新 FileIndex。

```typescript
// src/file-index/file-watcher.ts

export class FileWatcher {
  constructor(
    private cwd: string,
    private index: FileIndex,
    private ig: Ignore,
  ) {}

  /** 启动监听 */
  start(): void

  /** 停止监听（app 退出时调用） */
  stop(): void
}
```

**技术选型**：Node.js 内置 `fs.watch` + `recursive: true`
- **零外部依赖**，Node 18.17+ / Bun 原生支持
- Windows 使用 ReadDirectoryChangesW（内核级），macOS 使用 FSEvents，Linux 使用 inotify
- 收到 `rename`（新建/删除/重命名）事件后通过 `ignore` 过滤，再调用 `FileIndex.add()` / `remove()`
- 比 chokidar 更轻量，不存在 chokidar Windows 锁目录问题

> **备选方案**：如果 `fs.watch` 在某些极端场景下不可靠，退回 `chokidar` v4（纯 JS，内部也用 `fs.watch`，加了跨平台兼容层）。
> **未考虑**：`@parcel/watcher` 原生 C++ addon 性能最佳，但需要 node-gyp 编译，不兼容 Bun，排除。

**防抖**：文件变更事件 debounce 100ms，批量更新索引。

### 3.3 AtResolver — 引用解析器

**职责**：解析用户输入中的 `@path` 引用，生成 LLM context 提示。

```typescript
// src/file-index/at-resolver.ts

export interface AtReference {
  raw: string          // 原始匹配文本，如 "@src/hooks/types.ts"
  relativePath: string // 相对路径
  absolutePath: string // 绝对路径
  lineRange?: {        // 可选行范围
    start: number
    end?: number
  }
  start: number        // 在输入字符串中的起始位置
  end: number          // 结束位置
}

export class AtResolver {
  constructor(
    private cwd: string,
    private index: FileIndex,
  ) {}

  /** 从用户输入中解析所有 @ 引用 */
  parse(input: string): AtReference[]

  /** 生成注入给 LLM 的 context 文本 */
  buildContext(refs: AtReference[]): string

  /** 预处理用户消息：context + 原始输入 */
  resolve(input: string): { context: string; rawInput: string }
}
```

**解析规则**：
```
@<path>          → 引用整个文件
@<path>#<line>   → 引用特定行
@<path>#<s>-<e>  → 引用行范围

正则: (?:^|\s)@([\w.\/\\-]+(?:#\d+(?:-\d+)?)?)
```

**context 注入格式**：
```xml
<file-references>
<file path="src/hooks/types.ts" absolute="D:\a_dev_work\...\src\hooks\types.ts" />
<file path="src/ui/App.tsx" absolute="D:\a_dev_work\...\src\ui\App.tsx" lines="100-200" />
</file-references>
```

LLM 看到路径后自行决定是否用 Read 工具读取。不注入文件内容，省 token。

### 3.4 AtSuggestion — 建议浮层 UI

**职责**：输入 `@` 后展示文件建议列表，支持模糊匹配和键盘导航。

```typescript
// src/ui/AtSuggestion.tsx

interface AtSuggestionProps {
  /** 模糊搜索结果 */
  items: AtSuggestionItem[]
  /** 当前选中索引 */
  selectedIndex: number
}

interface AtSuggestionItem {
  /** 相对路径 */
  path: string
  /** 文件名（高亮显示） */
  filename: string
  /** 父目录（dimColor 显示） */
  directory: string
  /** 匹配高亮位置 */
  positions?: number[]
}
```

**与 CommandSuggestion 的关系**：
- **独立组件**，不复用 CommandSuggestion
- 理由：未来 `@` 可能扩展支持 `@agent`、`@mcp-resource` 等类型，展示样式和数据结构不同
- 共享的逻辑（窗口滚动、键盘导航）可提取为通用 hook `useSuggestionNavigation`

**展示设计**：
```
┌──────────────────────────────────────┐
│  📄 types.ts       src/hooks/        │  ← 选中态（cyan 高亮）
│     store.ts       src/skills/engine │
│     hook-runner.ts src/hooks/        │
│     App.tsx        src/ui/           │
│  (4/127) ↑↓ 滚动                     │
└──────────────────────────────────────┘
```

- 文件名左对齐加粗，目录 dimColor 右侧显示
- 最多显示 9 项，循环滚动（同 CommandSuggestion）
- 匹配字符高亮

## 四、集成到现有架构

### 4.1 启动流程

```
bin/zcli.ts (交互模式)
  └→ bootstrap: ensureFileIndexReady()
       ├→ FileIndex.scan()        // 全量扫描
       └→ FileWatcher.start()     // 启动监听
```

`ensureFileIndexReady()` 在 `ensureSkillsDiscovered()` 之后调用，不阻塞首帧渲染。

### 4.2 输入流改造

```
App.tsx
  ├→ inputValue 变化
  │   ├→ 以 "/" 开头 → CommandSuggestion（现有逻辑）
  │   └→ 包含 "@" → AtSuggestion（新增逻辑）
  │
  └→ handleSubmit(input)
       ├→ AtResolver.resolve(input)
       │   ├→ context: "<file-references>..."
       │   └→ rawInput: 原始输入
       └→ submit(context + rawInput)  // context 拼接到用户消息前面
```

### 4.3 键盘交互

| 触发 | 行为 |
|------|------|
| 输入 `@` | 弹出 AtSuggestion，展示按 frecency 排序的文件列表 |
| `@` 后继续输入 | 模糊匹配过滤 |
| ↑↓ 方向键 | 导航选中项 |
| Tab | 补全选中文件路径到输入框（`@src/hooks/types.ts `） |
| Enter | 提交当前输入 |
| Esc | 关闭建议浮层 |

**@检测条件**：
- `@` 前是空白字符或位于输入起始位置
- `@` 后到光标之间无空格

### 4.4 tsconfig 路径别名

```json
{
  "@file-index": ["src/file-index"],
  "@file-index/*": ["src/file-index/*"]
}
```

## 六、依赖新增

| 包名 | 用途 | 大小 | 备注 |
|------|------|------|------|
| `fast-glob` | 目录扫描 | ~50KB | 纯 JS，Bun 兼容 |
| `ignore` | .gitignore 规则解析 | ~15KB | 纯 JS，ESLint/Prettier 都在用 |
| `fzf` (fzf-for-js) | 模糊匹配 | ~7.5KB | fzf 算法忠实移植，Smith-Waterman 评分，匹配质量最佳 |

文件监听使用 Node.js 内置 `fs.watch`（零依赖）。

> **模糊匹配备选**：`uFuzzy`（~7.5KB）— 零依赖、无需预建索引、启动 <1ms。
> 优势：内存更省，速度更快；劣势：匹配质量略低于 fzf 算法（无 Smith-Waterman 评分，不感知词边界/驼峰）。
> 如果未来发现 fzf-for-js 在超大文件列表（10万+）下性能不够，可切换到 uFuzzy。

## 七、未来扩展
