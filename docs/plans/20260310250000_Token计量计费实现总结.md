# F10 Token 计量计费实现总结

## 概述

F10 为 cCli 引入了 **Token 计量计费系统**，在每次 LLM 调用后自动记录四维 token 消耗（input/output/cacheRead/cacheWrite）到 SQLite，匹配计价规则计算费用，并在 StatusBar 和 `/usage` 指令中展示。**支持多币种**（USD/CNY），国际厂商按美元计价，国内厂商（GLM、DeepSeek）按人民币计价，UI 自动显示对应货币符号。

**核心设计**：AgentLoop 产生 `llm_usage` 事件 → TokenMeter 独立观察者消费 → 匹配 pricing_rules → 写入 usage_logs（含币种） → 维护会话统计（按币种分组）。

---

## 架构

```
AgentLoop                   SessionLogger (F9)          SessionStore → JSONL
  │                             │
  ├─ yield llm_usage ──────────► consume()
  │       │
  │       └────────────────────► TokenMeter (F10)        SQLite → usage_logs
  │                                 │
  │                                 ├─ #resolveRule()    → pricing_rules 匹配
  │                                 ├─ #calculateCost()  → 四维费用计算
  │                                 ├─ getSessionStats() → 内存累计（StatusBar）
  │                                 ├─ getTodayStats()   → SQL 聚合
  │                                 └─ getMonthStats()   → SQL 聚合
  │
  └─ yield text/done ─────────► UI 渲染
```

**关键特征**：
- **独立观察者**：TokenMeter 与 SessionLogger 平级，互不依赖，各自消费 AgentEvent
- **四维 token**：input / output / cacheRead / cacheWrite 全链路贯通
- **多币种支持**：pricing_rules 带 `currency` 字段，费用按币种分组统计和展示（$ / ¥）
- **懒加载 SQLite**：首次 `getDb()` 时创建 `~/.ccode/data/ccode.db`
- **内聚计价**：pricing_rules 匹配和费用计算全部封装在 TokenMeter 内部
- **prepared statement 缓存**：构造时预编译 SQL，规则匹配按 provider+model 缓存
- **本地时区**：getTodayStats / getMonthStats 使用本地日期计算边界

## 数据库设计

### usage_logs 表

每次 LLM 调用产生一条记录，记录四维 token 消耗和费用。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| session_id | TEXT NOT NULL | 所属会话 ID |
| timestamp | TEXT NOT NULL | 记录时间 (ISO 8601) |
| provider | TEXT NOT NULL | LLM 供应商 |
| model | TEXT NOT NULL | 模型标识 |
| input_tokens | INTEGER NOT NULL | 输入 token 数 |
| output_tokens | INTEGER NOT NULL | 输出 token 数 |
| cache_read | INTEGER NOT NULL | 缓存读取 token 数 |
| cache_write | INTEGER NOT NULL | 缓存写入 token 数 |
| duration_ms | INTEGER | LLM 调用耗时（毫秒） |
| cost_amount | REAL | 费用（对应币种金额），无匹配规则时为 NULL |
| cost_currency | TEXT NOT NULL | 费用币种（USD/CNY），取自匹配的 pricing_rule |
| pricing_rule_id | INTEGER | 匹配的计价规则 ID |

### pricing_rules 表

按 provider + model 通配符匹配，支持时间范围和优先级。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| provider | TEXT NOT NULL | LLM 供应商 |
| model_pattern | TEXT NOT NULL | 匹配模式（末尾 `*` 通配符） |
| input_price | REAL NOT NULL | 输入价格 (货币/百万 token) |
| output_price | REAL NOT NULL | 输出价格 (货币/百万 token) |
| cache_read_price | REAL NOT NULL | 缓存读取价格 (货币/百万 token) |
| cache_write_price | REAL NOT NULL | 缓存写入价格 (货币/百万 token) |
| currency | TEXT NOT NULL | 价格币种（USD = 美元，CNY = 人民币） |
| effective_from | TEXT NOT NULL | 生效起始日期 |
| effective_to | TEXT | 截止日期（NULL 表示永久有效） |
| source | TEXT | 价格来源说明 |
| priority | INTEGER NOT NULL | 匹配优先级（越大越优先） |

### schema_comments 表

SQLite 不支持列级 COMMENT，用此表存储字段说明，可运行时查询。

| 字段 | 类型 | 说明 |
|------|------|------|
| table_name | TEXT NOT NULL | 所属表名 |
| column_name | TEXT NOT NULL | 字段名 |
| comment | TEXT NOT NULL | 字段说明 |

## 费用计算公式

```
cost = (inputTokens × input_price
      + outputTokens × output_price
      + cacheReadTokens × cache_read_price
      + cacheWriteTokens × cache_write_price) / 1,000,000

# cost 单位取决于匹配到的 pricing_rule.currency（USD 或 CNY）
```

计价规则匹配逻辑：
1. 按 `provider` 精确匹配
2. 按 `effective_from <= now` 且 `effective_to IS NULL OR > now` 过滤时间范围
3. 按 `priority DESC, effective_from DESC` 排序
4. 逐条用 `model_pattern` 通配符匹配（仅支持末尾 `*`）
5. 首个匹配命中即返回，费用币种取自 `pricing_rule.currency`；无匹配则 cost 为 NULL

**多币种展示**：
- `SessionCostStats.costByCurrency`: `Record<string, number>`，按币种分组累计（如 `{ USD: 1.23, CNY: 4.56 }`）
- `getTodayStats()` / `getMonthStats()`: SQL 按 `cost_currency GROUP BY` 返回 `AggregateStats[]` 数组
- UI 显示：USD → `$`，CNY → `¥`，多币种用 `+` 连接（如 `$0.0225 + ¥0.0036`）

## 决策记录

1. **TokenMeter 独立于 SessionLogger**：两者平级消费 AgentEvent，各自负责 SQLite 和 JSONL。好处：完全解耦，可独立测试，关注点分离。

2. **内置默认计价规则**：hardcode 在代码中，首次建库时 seed 到 SQLite。好处：开箱即用，无需外部配置。用 `INSERT WHERE NOT EXISTS` 保证幂等。

3. **better-sqlite3 同步 API**：CLI 场景下写入频率低（每次 LLM 调用一次），微秒级同步写入不阻塞。相比 async 方案代码更简洁，无需处理 Promise。

4. **零新目录**：用户明确要求减少新建文件夹。计价逻辑内聚在 TokenMeter（私有方法），db.ts 放在已有的 persistence/ 下。

5. **prepared statement 缓存**：构造时预编译 4 条 SQL，pricing rule 按 provider+model 缓存。bind() 切换时自动失效。避免每次 consume 重复解析 SQL。

6. **本地时区**："今日"和"本月"使用本地日期计算，符合用户直觉（UTC+8 凌晨不会看到"昨天"的数据）。

7. **schema_comments 元数据表**：SQLite 不支持列级 COMMENT，单独建表存储字段说明，可运行时查询 `SELECT * FROM schema_comments WHERE table_name = 'usage_logs'`。

8. **多币种而非统一汇率**：国内厂商（GLM、DeepSeek）按人民币计价，国际厂商按美元计价。不做汇率换算，直接按原始币种存储和展示。原因：汇率波动大且用户关心的是"花了多少人民币/美元"而非等值换算。`costByCurrency: Record<string, number>` 按币种分组，UI 用 `$` / `¥` 符号区分。

9. **今日/月统计按币种 GROUP BY**：`getTodayStats()` / `getMonthStats()` 返回 `AggregateStats[]` 数组而非单行。每个币种一行，前端自行合并展示。好处：SQL 层面简单清晰，不做跨币种合并。
