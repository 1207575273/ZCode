# Agent Loop 架构选型讨论

> 日期: 2026-03-03 | 状态: **已决策 → 方案 C**

---

## 背景与核心诉求

ZCli 的 Agent Loop 是整个产品的核心引擎。LLM 调用层的选型直接影响：

- **多模型能力**：任何支持 OpenAI 兼容协议的供应商都能接入（F2）
- **可控性**：Agent Loop 的编排逻辑（SubAgent、会话恢复、权限确认等）必须完全自主
- **CLI 启动速度**：目标冷启动 < 500ms
- **长期可维护性**：避免被第三方框架的版本节奏绑架

---

## 四个候选方案

### 方案 A：纯 OpenAI 兼容协议（原生 HTTP）

自研 HTTP Client，只对接 `/v1/chat/completions` 协议。

```typescript
interface ProviderConfig {
  baseURL: string   // 唯一需要变的东西
  apiKey: string
  model: string
}
// 任何 OpenAI 兼容服务只换 baseURL:
// Anthropic → https://api.anthropic.com/v1/
// Gemini    → https://generativelanguage.googleapis.com/v1beta/openai/
// Ollama    → http://localhost:11434/v1/
// 任意代理  → https://your-proxy.com/v1/
```

需要自己实现：SSE 流式响应解析、tool calling 协议组装（各家有细微差异）、错误重试/超时/abort、token 数提取。

| 维度 | 评估 |
|------|------|
| 依赖 | 零 |
| 多模型 | ✅ 天然支持 OpenAI 兼容协议 |
| 可控性 | ✅ 最高 |
| 开发成本 | ❌ 高，边界情况多 |
| 高级特性 | ❌ Anthropic cache control、extended thinking 无法直接用 |

---

### 方案 B：LangChain / LangGraph TS

用 `@langchain/core` 做 LLM 抽象，`@langchain/langgraph` 做 Agent 状态机。

> 参考 `a_res/Codara`：该项目用 LangChain 的 `initChatModel` 做多模型路由，是方案 B 的真实落地案例。

```typescript
import { initChatModel } from "langchain/chat_models/universal"
// Codara 的做法：配置路由表，factory.create("default") 创建模型实例
```

| 维度 | 评估 |
|------|------|
| 依赖 | ❌ 重（10+ 包，node_modules 膨胀） |
| 多模型 | ✅ 内置多模型抽象 |
| 可控性 | ❌ 最低，Agent Loop 核心逻辑被框架接管 |
| 调试体验 | ❌ 穿透抽象层极其痛苦，错误是框架堆栈 |
| 版本维护 | ❌ LangChain JS breaking change 频率高 |
| 启动速度 | ❌ 最慢 |

Codara 选了 LangChain，但它的定位是快速原型，不追求极致体验和长期可维护性。ZCli 目标不同，不应照搬。

---

### 方案 C：Vercel AI SDK（推荐）

用 `ai` 核心包 + `@ai-sdk/openai`（OpenAI 兼容协议封装）。

```typescript
import { streamText, tool } from "ai"
import { createOpenAI } from "@ai-sdk/openai"

// 任何 OpenAI 兼容服务，只换 baseURL
const provider = createOpenAI({ baseURL: "...", apiKey: "..." })

const result = streamText({
  model: provider("claude-opus-4-6"),
  messages,
  tools: { /* ZCli 工具集 */ },
  maxSteps: 10,
})
```

**Vercel AI SDK 是什么**：对 LLM API 调用的薄 TypeScript 封装。核心只做三件事：发请求、解析 SSE 流、处理 tool calling 协议。不管 Agent 怎么编排，那是 ZCli 自己的事。

| 维度 | 评估 |
|------|------|
| 依赖 | ✅ 轻（2-3 包） |
| 多模型 | ✅ `@ai-sdk/openai` 原生就是 OpenAI 兼容协议，换 baseURL 即可 |
| 可控性 | ✅ 高，Agent Loop 编排完全自研 |
| 调试体验 | ✅ 薄抽象，问题定位清晰 |
| 高级特性 | ✅ 可选引入 `@ai-sdk/anthropic` 解锁 Claude 独有能力 |
| 替换成本 | ✅ 低，自研 LLMProvider 接口包一层，上层不感知 |

---

### 方案 D：Claude Agent SDK（本次新增讨论）

> 由外部建议引入，需要先澄清这个 SDK 究竟是什么。

#### D1：claude-agent-sdk（Python SDK，a_res 中有源码）

**实际工作原理**（读源码得出，不是文档描述）：

```python
# _internal/transport/subprocess_cli.py — 核心实现
# 通过 subprocess 把 Claude Code CLI 作为子进程启动
# 再通过 stdin/stdout JSON 流与其通信
```

这不是一个调用 LLM API 的 SDK，**它是一个远程控制 Claude Code CLI 的 SDK**。

```
你的程序
  │
  │ subprocess spawn
  ▼
Claude Code CLI 进程 (claude)
  │
  │ 内部调用
  ▼
Anthropic API（固定，Claude only）
```

**致命约束**：
- 必须在本机安装 Claude Code CLI 才能工作
- 底层永远是 Claude，无法接入 OpenAI / Gemini / 本地模型
- 子进程模式，启动开销和通信延迟不可控
- 这是 Python SDK，ZCli 是 TypeScript 项目，无法直接使用

**结论**：与 ZCli「多模型」的核心需求根本矛盾，直接排除。

#### D2：@anthropic-ai/sdk（TypeScript 官方 SDK，a_res 中有源码）

这才是真正的 Anthropic TypeScript SDK，直接调 Anthropic API。

```typescript
import Anthropic from "@anthropic-ai/sdk"

const client = new Anthropic({ apiKey: "..." })
const message = await client.messages.create({
  model: "claude-opus-4-6",
  messages: [{ role: "user", content: "Hello" }],
  max_tokens: 1024,
})
```

| 维度 | 评估 |
|------|------|
| 依赖 | ✅ 轻，官方维护 |
| 多模型 | ❌ **只能用 Claude**，不支持 OpenAI / Gemini / Ollama |
| 高级特性 | ✅ 最全，extended thinking、cache control、citations 全支持 |
| 可控性 | ✅ 高，Agent Loop 自研 |

**结论**：作为 Claude 的「原生 Provider」是合适的，但不能作为多模型的统一调用层。它的定位应该是：**当用户选择走 Claude 原生协议而非 OpenAI 兼容协议时的补充选项**。

## 方案对比总表

| 维度 | A. 原生 HTTP | B. LangChain | C. Vercel AI SDK | D1. Claude Agent SDK | D2. @anthropic-ai/sdk |
|------|:-----------:|:------------:|:----------------:|:--------------------:|:---------------------:|
| 多模型支持 | ✅ | ✅ | ✅ | ❌ Claude only | ❌ Claude only |
| 依赖量 | 零 | 重 | 轻 | 需安装 Claude Code | 轻 |
| Agent Loop 自主 | ✅ | ❌ | ✅ | ❌ | ✅ |
| 开发效率 | 低 | 高 | 中高 | 高但受限 | 中 |
| CLI 启动速度 | 最快 | 最慢 | 快 | 慢(subprocess) | 快 |
| 调试体验 | 好 | 差 | 好 | 差(黑盒子进程) | 好 |
| Anthropic 高级特性 | ❌ 需自研 | 部分 | 可选引入 | ✅ | ✅ |
| 可替换性 | 无需替换 | 极难 | 低成本 | 无法替换 | 低成本 |
| 长期维护成本 | 自研维护 | 高(频繁breaking) | 低 | 高(依赖CLI版本) | 低 |

## 最终决策：方案 C 为主干 + D2 作为 Claude 原生通道

```
┌─────────────────────────────────────────────────────────┐
│              Agent Loop (完全自研)                        │
│  消息管理 → 轮次控制 → SubAgent 调度 → 权限确认           │
│  上下文压缩 → 并行工具调用 → 会话恢复                     │
│              ↑ 这一层是 ZCli 的灵魂，不交给任何框架        │
├─────────────────────────────────────────────────────────┤
│              ZCli LLMProvider 接口 (自研，隔离底层)        │
│  chat(request) → AsyncIterable<ChatChunk>                │
│  底层换什么 SDK 上层不感知                                 │
├──────────────────────┬──────────────────────────────────┤
│  主通道 (90% 场景)    │  原生通道 (需要独有能力时)         │
│                       │                                  │
│  @ai-sdk/openai       │  @ai-sdk/anthropic               │
│  OpenAI 兼容协议       │  (extended thinking,             │
│  换 baseURL 接任意     │   cache control, citations)      │
│  供应商               │                                  │
│                       │  @ai-sdk/google                  │
│  Anthropic ✓          │  (grounding, code execution)     │
│  OpenAI ✓             │                                  │
│  Gemini ✓             │  Raw HTTP fetch                  │
│  Ollama ✓             │  (极端自定义场景逃生口)            │
│  任意代理 ✓            │                                  │
└──────────────────────┴──────────────────────────────────┘
```

## 决策理由总结

| 问题 | 答案 |
|------|------|
| 为什么不选 LangChain (B)？ | Agent Loop 是 ZCli 核心竞争力，不能交给框架；依赖重拖慢启动；Codara 用了它但 Codara 不追求极致可控 |
| 为什么不选 Claude Agent SDK (D1)？ | 它是控制 Claude Code CLI 子进程的工具，不是 LLM 调用库；绑死 Claude；与多模型需求根本矛盾 |
| 为什么不把 @anthropic-ai/sdk (D2) 作为主通道？ | 只支持 Claude，无法满足多模型需求；但作为 Claude 原生特性的补充通道合理 |
| 为什么选 Vercel AI SDK (C)？ | 轻量、OpenAI 兼容协议优先、Agent Loop 自主控制、可替换性强、可选原生 Provider 获取高级特性 |
| 纯原生 HTTP (A) 为什么不选主干？ | SSE + tool calling 协议处理工作量大、各家差异需逐个踩坑；在 Vercel AI SDK 包一层后作为逃生口即可 |
