> 日期: 2026-03-19
> 状态: 设计中（待打磨）
> 数据来源: Claude Code CLI 逆向分析、Anthropic 官方文档、Codex CLI 横向对比

## 一、问题陈述

cCli 当前的对话机制是**无限增长的朴素累积**——每轮 LLM 调用把完整历史发送一次，没有任何裁剪、压缩或记忆机制。这意味着：

1. **长对话必崩**：200K 上下文窗口用完后 API 直接报错，用户丢失所有上下文
2. **无法感知风险**：用户不知道当前用了多少 token，没有任何预警
3. **tool 输出无控制**：一次 `grep` 返回 5000 行结果全部进入历史，快速吞噬窗口
4. **跨会话无记忆**：每次新对话从零开始，无法积累项目知识和用户偏好
5. **成本浪费**：重复发送大量已过时的 tool 输出，浪费输入 token 费用

这是 cCli 走向生产可用的**最大架构短板**。

> **观察方式**：可以通过 `cCli/debug.log` 查看每次 LLM 调用发送的完整 messages 数组和 API 返回的 token 统计，
> 直观感受 history 的膨胀过程。

### 1.1 消息流时序详解 — 以实际场景说明

#### 场景 1：用户第一次发"你好"

```
useChat.submit("你好")
  │
  ├─ 1. 构建 history
  │    messages 状态：[]（空）
  │    + 新 userMsg: { role: 'user', content: '你好' }
  │    过滤 system 消息，只保留 user/assistant
  │
  │    history = [
  │      { role: 'user', content: '你好' }       ← 仅此 1 条
  │    ]
  │
  ├─ 2. new AgentLoop(provider, registry, config)
  │
  ├─ 3. loop.run(history)
  │    │
  │    │  内部 copy: const history = [{ role: 'user', content: '你好' }]
  │    │
  │    ├─ turn 0: #callLLM(history)
  │    │    发送给 API：
  │    │    {
  │    │      model: 'claude-sonnet-4-6',
  │    │      systemPrompt: '...(CCODE.md + Skills + Hooks)...',   ← ~3-5K tokens 固定开销
  │    │      tools: [bash, read_file, write_file, ...12 个],     ← ~5-15K tokens 固定开销
  │    │      messages: [
  │    │        { role: 'user', content: '你好' }                  ← ~2 tokens
  │    │      ]
  │    │    }
  │    │
  │    │    API 返回：text "你好！有什么可以帮你的？"
  │    │    toolCalls: []（无工具调用）
  │    │    inputTokens: 15,234（精确值，含 system + tools + messages）
  │    │
  │    └─ toolCalls 为空 → done
  │
  └─ 4. messages 状态更新为：
       [
         { role: 'user',      content: '你好' },
         { role: 'assistant', content: '你好！有什么可以帮你的？' }
       ]
```

**第一次调用 inputTokens ≈ 15K**——几乎全是 system prompt + tool definitions 的固定开销。

#### 场景 2：用户第二次发"你好"

```
useChat.submit("你好")
  │
  ├─ 1. 构建 history（关键：携带全部历史，无裁剪）
  │    messages 状态：[user "你好", assistant "你好！..."]
  │    + 新 userMsg: { role: 'user', content: '你好' }
  │
  │    history = [
  │      { role: 'user',      content: '你好' },                    ← 第 1 轮（旧）
  │      { role: 'assistant', content: '你好！有什么可以帮你的？' },   ← 第 1 轮回复（旧）
  │      { role: 'user',      content: '你好' },                    ← 第 2 轮（新）
  │    ]
  │
  ├─ 2. loop.run(history)
  │    发送给 API：
  │    {
  │      systemPrompt: '...',     ← 和第一次完全相同
  │      tools: [...],            ← 和第一次完全相同
  │      messages: [              ← 全部 3 条，包含已过时的第 1 轮历史
  │        { role: 'user',      content: '你好' },
  │        { role: 'assistant', content: '你好！...' },
  │        { role: 'user',      content: '你好' },
  │      ]
  │    }
  │    inputTokens: 15,260  ← 比第一次多 ~26 tokens（2 条历史消息的开销）
  │
  └─ messages 状态更新为 4 条
```

**第二次 inputTokens 几乎不变（+26 tokens），因为纯文本对话增量很小。**

#### 场景 3：涉及工具调用（"帮我看看 package.json"）—— 问题开始暴露

```
useChat.submit("帮我看看 package.json")
  │
  ├─ history = [
  │    ...前面所有历史消息...,                           ← 全量携带，无裁剪
  │    { role: 'user', content: '帮我看看 package.json' }
  │  ]
  │
  ├─ loop.run(history)
  │    │
  │    ├─ turn 0: #callLLM(history)
  │    │    inputTokens: 15,400
  │    │    API 返回 toolCalls: [{ toolName: 'read_file', args: { path: 'package.json' } }]
  │    │
  │    ├─ #executeOneTool(tc, history)
  │    │    → read_file 执行，读取 package.json（200 行）
  │    │    → 关键！tool 结果直接 push 到 history：
  │    │      history.push({
  │    │        role: 'user',
  │    │        content: '[Tool read_file result]: {"name":"ccode",...整个文件内容...}'
  │    │      })
  │    │    → 这一条消息可能就有 ~2000 tokens
  │    │
  │    ├─ turn 1: #callLLM(history)        ← 第二轮 LLM 调用，history 已膨胀
  │    │    messages: 所有旧历史 + 新用户消息 + 完整 tool 结果
  │    │    inputTokens: 17,400             ← 多了 ~2000 tokens（package.json 内容）
  │    │    API 返回文本，无更多工具调用
  │    │
  │    └─ done
  │
  └─ tool 结果永远留在 history 里，后续每轮 LLM 调用都会重复发送
```

**tool 结果是 `role: 'user'` 追加到 history，之后的每一次 LLM 调用都背着它。**

#### 1.2 增长曲线：为什么 tool 结果是"罪魁祸首"

```
第 1 轮：inputTokens ≈ 15K
  固定开销: system(4K) + tools(10K) + messages(1 条, ~20 tokens)

第 2 轮：inputTokens ≈ 15.3K
  固定开销: 同上 + messages(3 条, ~60 tokens)
  → 纯文本对话几乎不增长

第 5 轮（涉及 grep + read_file）：inputTokens ≈ 35K
  固定开销(14K) + messages(15 条, ~21K)
                    其中 tool results 占 ~18K   ← 开始膨胀

第 10 轮（大量工具调用）：inputTokens ≈ 80K
  固定开销(14K) + messages(40 条, ~66K)
                    其中 tool results 占 ~55K   ← 大部分是旧 tool 结果

第 20 轮：inputTokens ≈ 128K → 默认窗口用完
  无任何防线，API 直接报 context_length_exceeded 错误
  用户丢失所有上下文，必须开新对话
```

**结论**：

1. **纯文本对话**增长极慢（每轮 +几十 tokens），聊 100 轮也不会满
2. **工具调用**是上下文膨胀的核心原因，一次 `grep` 返回 5000 行 = 上万 tokens
3. **旧 tool 结果**价值衰减最快（3 轮前的 read_file 结果大概率已过时），但永远不被清除
4. **固定开销**（system + tools ≈ 14K）每次调用都重复发送，但可通过 Prompt Caching 优化

这就是为什么上下文管理的**第一优先级是清理旧 tool 结果**，而不是压缩对话文本。

## 二、上下文窗口定义 — 一次 LLM 调用包含什么

### 2.1 窗口组成结构

一个完整的上下文窗口由以下部分组成，每部分都消耗 token 配额：

```
┌─────────────────────────────────────────────────────┐
│                 Model Context Window                 │
│                (e.g. 200K tokens)                     │
│                                                      │
│  ┌──────────────────────────────────────────────┐   │
│  │ System Prompt（系统提示）                      │   │  ← 固定开销，每次调用都发送
│  │  ├─ 基础指令（角色、规则、约束）               │   │
│  │  ├─ CCODE.md / CLAUDE.md 用户指令               │   │
│  │  ├─ Skills 列表描述                            │   │
│  │  └─ Hook 注入的 additionalContext              │   │
│  │                               ~3K-5K tokens    │   │
│  ├──────────────────────────────────────────────┤   │
│  │ Tool Definitions（工具 JSON Schema）           │   │  ← 固定开销，随工具数增长
│  │  ├─ 内置工具 × 12（bash/read/write/edit/...）  │   │
│  │  ├─ MCP 动态工具 × N                           │   │
│  │  └─ Skill 工具                                 │   │
│  │                               ~5K-15K tokens   │   │
│  ├──────────────────────────────────────────────┤   │
│  │ Conversation History（对话历史）               │   │  ← 持续增长，窗口主要消耗者
│  │  ├─ User Messages（用户消息）                   │   │
│  │  │    含 @ 文件引用注入的内容                   │   │
│  │  ├─ Assistant Messages（助手回复）              │   │
│  │  │    含 thinking blocks（如启用 extended）      │   │
│  │  ├─ Tool Use Blocks（工具调用请求）             │   │
│  │  │    LLM 发出的 tool_use content blocks       │   │
│  │  └─ Tool Results（工具执行结果）                │   │  ← 最大消耗者
│  │       grep 5000 行、bash 编译输出、             │   │
│  │       read_file 大文件等                        │   │
│  │                            ~0K-180K tokens     │   │
│  ├──────────────────────────────────────────────┤   │
│  │ Output Reserve（输出预留）                     │   │  ← 必须预留，否则 LLM 无法回复
│  │                              ~4K-16K tokens    │   │
│  └──────────────────────────────────────────────┘   │
│                                                      │
│  Effective Window = Total - Output Reserve            │
│  Auto-compact 阈值 = Effective Window × 98%          │
└─────────────────────────────────────────────────────┘
```

### 2.2 各组成部分的特征

| 组成部分 | 大小特征 | 可缓存 | 可压缩 | 增长模式 |
|----------|----------|--------|--------|----------|
| **System Prompt** | 固定 ~3-5K | ✅ 高命中率（首次缓存） | ❌ 不应压缩 | 恒定 |
| **Tool Definitions** | 半固定 ~5-15K | ✅ 工具集不变时 | ❌ 不应压缩 | MCP 工具增减时变化 |
| **User Messages** | 变量，通常较小 | ✅ 前缀缓存 | ✅ 可摘要 | 每轮对话增长 |
| **Assistant Messages** | 变量，较大 | ✅ 前缀缓存 | ✅ 可摘要 | 每轮对话增长 |
| **Tool Use Blocks** | 变量，较小 | ✅ 前缀缓存 | ✅ 可清除 | 每次工具调用增长 |
| **Tool Results** | **变量，可以非常大** | ✅ 前缀缓存 | **✅ 优先清除** | 工具执行后增长 |
| **Output Reserve** | 固定，模型相关 | — | ❌ | 恒定 |

### 2.3 Effective Window 计算

```
Total Window        = 128,000 tokens（cCli 默认值，支持按模型/配置调整）
Output Reserve      = 16,384 tokens（max_tokens 配置）
Effective Window    = 128,000 - 16,384 = 111,616 tokens

Fixed Overhead      = System Prompt (~4K) + Tool Definitions (~10K) = ~14K
Available for Conv  = 111,616 - 14,000 = ~97K tokens

Auto-compact at     = 97K × 95% ≈ 92K tokens（对话历史部分）
Warning at          = 97K × 70% ≈ 68K tokens
```

**多模型窗口差异**：

| 模型 | 窗口大小 | Effective（扣除 16K 输出） |
|------|----------|---------------------------|
| Claude Sonnet/Opus 4.6 | 200K | ~184K |
| GPT-4o | 128K | ~112K |
| GLM-4 | 128K | ~112K |
| DeepSeek | 64K | ~48K |

cCli 默认按 **128K** 计算。支持通过配置调整（`config.contextWindow`），后续可按模型自动适配。
窗口大小影响 auto-compact 阈值——DeepSeek 64K 比 Claude 200K 更早触发压缩。

### 2.4 Claude Code 的窗口管理策略

**按优先级递减清理**：

```
1. 最先清理：旧的 Tool Results（占比最大、信息时效性最短）
   → Context Editing API 自动清除，保留最近 3 个
   → 大输出保存到磁盘，上下文中只留文件引用

2. 其次压缩：对话历史（User + Assistant + Tool Use）
   → /compact 生成结构化摘要替换

3. 始终保留：System Prompt + Tool Definitions
   → 不可压缩，每次调用必须完整发送
   → 利用 Prompt Caching 降低重复发送成本

4. 特殊处理：
   → MCP 工具描述超过窗口 10% 时自动 defer（按需加载）
   → Skill 描述限制在窗口 2% 以内
   → 图片/PDF 块在 compact 时剥离
```

### 2.5 cCli 上下文窗口的目标结构

```typescript
/** 上下文窗口状态快照 */
interface ContextWindowState {
  /** 模型的总窗口大小（tokens） */
  totalWindow: number
  /** 输出预留（max_tokens） */
  outputReserve: number
  /** 有效窗口 = total - reserve */
  effectiveWindow: number

  /** 各组成部分占用 */
  usage: {
    systemPrompt: number     // 系统提示 tokens
    toolDefinitions: number  // 工具 Schema tokens
    userMessages: number     // 用户消息 tokens
    assistantMessages: number // 助手回复 tokens
    toolResults: number      // 工具执行结果 tokens（最大消耗者）
    total: number            // 合计
  }

  /** 百分比 = usage.total / effectiveWindow */
  usedPercentage: number
  /** 剩余可用 */
  remaining: number

  /** 状态级别 */
  level: 'normal' | 'warning' | 'critical' | 'overflow'
  //       <70%      70-85%      85-98%       >98%
}
```

## 三、业界方案横向对比

### 3.1 Claude Code（Anthropic 官方）

**三层递进防线**：

| 层级 | 机制 | 触发条件 | 效果 |
|------|------|----------|------|
| **第一层** | Context Editing（API 层） | input tokens 达到配置阈值 | 服务端自动清除旧 tool_use 结果，保留最近 N 个 |
| **第二层** | /compact 压缩 | 手动或 ~98% 自动触发 | 对话历史 → LLM 生成结构化摘要，60-80% 压缩率 |
| **第三层** | Memory 持久记忆 | LLM 主动判断 | 文件级跨会话记忆，分 user/feedback/project/reference 四类 |

**Context Editing 详细机制**（API 参数）：
```typescript
context_management: {
  edits: [{
    type: 'clear_tool_uses_20250919',
    trigger: { type: 'input_tokens', value: 100000 },  // 触发阈值
    keep: { type: 'tool_uses', value: 3 },              // 保留最近 3 个
    exclude_tools: ['memory'],                           // memory 工具不清除
  }]
}
```

**Compact 压缩 Prompt**：
```
Summarize our conversation above. This summary will be the only
context available when the conversation continues, so preserve
critical information including:
- What was accomplished（已完成的工作）
- Current work in progress（正在进行的工作）
- Files involved（涉及的文件）
- Next steps（下一步计划）
- Key user requests or constraints（关键用户要求和约束）
```

**Memory 工具**：

`memory_20250818` 是 Anthropic API 级别的**内置工具类型**，与 `computer_use` 类似——不需要你定义 JSON Schema，
模型在训练阶段已经学会了何时以及如何调用它。声明方式：

```typescript
// 普通工具 — 需要完整 Schema 定义
tools: [{ name: 'bash', description: '...', input_schema: { ... } }]

// API 内置工具 — 只声明类型，模型已知道怎么用
tools: [{ type: 'memory_20250818', name: 'memory' }]
```

**核心设计：API 定义接口协议，存储完全客户端实现**。Anthropic 不帮你存任何东西。
你收到 tool_use 调用后自己决定存哪里（文件系统/数据库/云存储）。

操作命令（6 个）：`view` / `create` / `str_replace` / `insert` / `delete` / `rename`

调用流程示例：
```
Claude → tool_use: { name: "memory", input: { command: "view", path: "/memories" } }
你的应用 → tool_result: "1.5K /memories/project_context.md\n2.0K /memories/user_prefs.md"
Claude → tool_use: { name: "memory", input: { command: "create", path: "/memories/debug_notes.md", file_text: "..." } }
你的应用 → tool_result: "File created successfully at: /memories/debug_notes.md"
```

与 Context Editing 配合：清除前 Claude 先把关键信息存到 memory，清除后按需读回。

**Claude Code CLI 的 Memory 存储路径实证**：

```
~/.claude/
├── CLAUDE.md                             ← 全局用户指令（只读，非 memory）
├── projects/
│   └── D--a-dev-work-claude-cli-z01/     ← 项目 slug
│       ├── CLAUDE.md                     ← 项目级指令（只读，非 memory）
│       └── memory/                       ← 项目级记忆（LLM 读写）
│           ├── MEMORY.md                 ← 索引文件（前 200 行加载到上下文）
│           └── f13-subagent.md           ← 具体记忆文件
```

**指令文件 vs 记忆文件的本质区别**：

| 维度 | CLAUDE.md（指令） | memory/（记忆） |
|------|-------------------|-----------------|
| 维护者 | 用户手动编写 | **LLM 通过 tool_use 读写** |
| 加载时机 | 每次会话启动注入 system prompt | LLM 按需 view 读取 |
| 作用范围 | 全局 / 项目级 | **项目级，按 project-slug 隔离** |
| 内容性质 | 固定规则、编码规范 | 动态积累的项目知识、用户偏好、决策记录 |
| 与 compact 关系 | 不受影响（在 system prompt 中） | **compact 的信息救生圈** — 压缩前存，压缩后读 |

**对 cCli 的设计意义**：

cCli 是多模型支持的（Claude/GLM/DeepSeek/OpenAI），不能直接用 `memory_20250818` API 内置类型
（只有 Anthropic provider 支持）。应该**自己实现一个 `memory` 工具**注册到 ToolRegistry，
所有 provider 统一使用。API 内置工具的接口设计（6 个命令）可以直接参考。

cCli 的 memory 存储目录应映射为：
```
~/.ccode/projects/<project-slug>/memory/
├── MEMORY.md           ← 索引
└── *.md                ← 具体记忆文件
```

### 3.2 Codex CLI（OpenAI）

**压缩策略**：
- 阈值：默认 180K-244K（取决于模型），95% 安全边际
- 重建历史：初始上下文 + **最近 ~20K token 原始消息** + 摘要

**压缩 Prompt**：
```
Create a handoff summary for another LLM that will resume the task.
Include:
- Current progress and key decisions made
- Important context, constraints, or user preferences
- What remains to be done (clear next steps)
- Any critical data, examples, or references needed
```

**关键差异**：不全量替换，保留近期原始消息 + 摘要。包含重试逻辑和指数退避。
明确警告用户："Long conversations and multiple compactions can cause the model to be less accurate."

### 3.3 方案对比

| 维度 | Claude Code | Codex CLI |
|------|------------|-----------|
| 压缩触发 | 手动 `/compact` + 自动(~98% effective window) | 自动(95% window) |
| 摘要策略 | **全量替换** — 摘要完全替代历史 | **摘要+保留近期** — 摘要 + 最近 ~20K 原始消息 |
| tool 输出管理 | API Context Editing（服务端清除旧 tool_use） | 无 |
| 大输出处理 | 保存到磁盘，留文件引用 | 无特殊处理 |
| 记忆系统 | Memory 工具（API 内置，LLM 读写） | 无 |
| 多次压缩 | 信息累积损失 | 有降级警告 |
| 状态保持 | Session Name / Plan Mode / TodoWrite 保留 | 基本状态保留 |

### 4.3 token 计数精度问题

当前 `countTokens()` 是 `text.length / 4` 的粗糙估算（±30% 误差），不足以支撑上下文管理。

**确定方案：使用 LLM 返回的 inputTokens**

每次 `llm_done` 事件已经携带精确的 `inputTokens` 数值（API 返回），不需要额外计算：

```
llm_done 事件 → inputTokens: 85432
  → 这就是本次 LLM 调用消耗的精确输入 token 数
  → 等于 system prompt + tool defs + 完整 history 的 token 总量
  → 直接作为 ContextWindowState.usage.total
```

**优势**：零额外成本、精确、已有数据（llm_done 事件里就有）
**劣势**：滞后一轮（只有调完 LLM 才知道），首次调用前无数据

首次调用前用 `length/4` 粗估即可，第一次 `llm_done` 后切换到精确值。

### 4.4 compact 对持久化的影响

compact 替换 messages 后，JSONL 的处理策略：

```
compact 前的 JSONL：
  session_start → user → assistant → tool_call_start → tool_call_end → user → assistant → ...

compact 后追加：
  ... → compact_event { summary, compactedMessageCount, tokensBefore, tokensAfter } → user → assistant → ...

compact 不删除历史消息，只追加一个 compact_event 标记压缩点。
```

**resume 恢复策略**：
- 从 compact_event 之后开始加载（摘要 + 后续新消息）
- compact_event 之前的消息仍可审计（完整 JSONL 保留）
- 如果用户 resume 到 compact 之前的分叉点，走原有的 fork 逻辑

**SessionEvent 新增类型**：
```typescript
type SessionEventType = ... | 'compact'

// compact 事件字段
{
  type: 'compact',
  summary: string,           // 压缩摘要文本
  compactedMessageCount: number, // 被压缩的消息数
  tokensBefore: number,      // 压缩前 token 数
  tokensAfter: number,       // 压缩后 token 数
  preservedRecentCount: number,  // 保留的近期原始消息数
}
```

## 五、架构分层 — ContextManager 中间层 + 策略模式

### 5.1 核心设计决策：上下文管理不是 Tool，也不耦合 AgentLoop

**为什么不能是 Tool？**

Tool 是 LLM 决定何时调用的。但上下文管理的核心场景是 history 已经接近溢出，
必须在 LLM 调用**之前**就完成裁剪/压缩——LLM 还没被调用，没机会调 tool。
LLM 不知道自己的上下文有多大，只有系统知道（从 API 返回的 `inputTokens`）。

**为什么不耦合 AgentLoop？**

AgentLoop 的职责是 **LLM ↔ 工具的多轮执行循环**（单一职责）。
如果把 context 裁剪、compact 触发、tool 结果清理都塞进去，会变成上帝类。

**正确做法：独立的中间层拦截器。**

### 5.2 三层分离架构

```
useChat.submit("帮我重构这个模块")
  │
  ├─ 1. 构建 rawHistory（全量，从 messages state 读取）
  │
  ├─ 2. ContextManager.prepare(rawHistory, strategy)     ← 新的中间层
  │     │
  │     ├─ 检查 token 使用率（上次 llm_done 的 inputTokens）
  │     │
  │     ├─ 如果 > 阈值：触发 compact
  │     │   → 按当前策略生成摘要（见 5.3 策略模式）
  │     │   → 替换/裁剪旧历史
  │     │
  │     ├─ 裁剪旧 tool results（替换为占位符）
  │     │
  │     └─ 返回 optimizedHistory
  │
  ├─ 3. AgentLoop.run(optimizedHistory)                  ← AgentLoop 不变
  │     → LLM ↔ 工具多轮循环（只管跑，不关心 history 怎么来的）
  │
  └─ 4. 事件消费（useChat 处理 UI 更新）
```

**各层职责**：

| 层 | 模块 | 职责 | 知道什么 |
|---|------|------|---------|
| **UI 层** | useChat | 消息状态管理、事件消费、React 渲染 | 用户消息、streaming state |
| **上下文层** | ContextManager | 历史裁剪、compact 触发、tool 结果清理 | token 预算、阈值、策略 |
| **执行层** | AgentLoop | LLM ↔ 工具多轮循环 | 只管执行，不关心 history 来源 |
| **感知层** | ContextTracker | token 使用率追踪、级别判定 | 窗口大小、inputTokens |

**什么是 Tool，什么不是**：

| 能力 | 实现方式 | 原因 |
|------|----------|------|
| auto-compact | ContextManager 内部 | 系统自动触发，LLM 不参与决策 |
| tool 结果裁剪 | ContextManager 内部 | LLM 调用前拦截，LLM 无感知 |
| `/compact` | 斜杠指令 → 调用 ContextManager | 用户主动触发 |
| `/context` | 斜杠指令 → 读取 ContextTracker | 用户查看状态 |
| `memory` | **Tool（LLM 调用）** | LLM 决定什么值得记住，这是 LLM 判断力的价值 |

### 5.3 Compact 策略模式 — 可插拔的压缩算法

ContextManager 内部使用**策略模式**，支持多种 compact 算法，用户可切换：

```typescript
/** 压缩策略接口 */
interface CompactStrategy {
  readonly name: string
  readonly description: string

  /**
   * 执行压缩：接收完整历史，返回压缩后的历史。
   * provider 用于调用 LLM 生成摘要。
   */
  compact(
    history: Message[],
    provider: LLMProvider,
    options: CompactOptions,
  ): Promise<Message[]>
}

interface CompactOptions {
  model: string
  focus?: string           // 用户指定的保留重点
  systemPrompt?: string    // 当前 system prompt（摘要时需要）
}
```

**内置三种策略**：

#### 策略 A：全量替换（Claude Code 同款）

```typescript
class FullReplaceStrategy implements CompactStrategy {
  name = 'full-replace'
  description = 'Claude Code 同款：LLM 生成摘要完全替换历史'

  async compact(history, provider, options): Promise<Message[]> {
    // 1. 预清理（剥离 image/PDF 等）
    // 2. 发给 LLM 生成结构化摘要
    // 3. 返回 [{ role: 'user', content: summary }]
    return [{ role: 'user', content: summary }]
  }
}
```

- **优势**：压缩率最高（60-80%），实现简单
- **劣势**：多次压缩信息损失大，近期上下文精度丢失
- **适用**：窗口较大的模型（Claude 200K），单次长对话

#### 策略 B：摘要 + 保留近期（Codex CLI 同款）

```typescript
class SummaryWithRecentStrategy implements CompactStrategy {
  name = 'summary-with-recent'
  description = 'Codex 同款：摘要 + 保留最近 N 条原始消息'

  /** 保留的近期消息 token 预算（默认 ~20K） */
  recentTokenBudget = 20_000

  async compact(history, provider, options): Promise<Message[]> {
    // 1. 切分：远期历史 vs 近期消息
    // 2. 远期历史 → LLM 生成摘要
    // 3. 返回 [{ role: 'user', content: summary }, ...recentMessages]
    return [{ role: 'user', content: summary }, ...recentMessages]
  }
}
```

- **优势**：近期精度高，渐进降级
- **劣势**：压缩率略低
- **适用**：多模型场景（GLM/DeepSeek 摘要质量不如 Claude），保留原始消息是安全网

#### 策略 C：仅裁剪 Tool Results（轻量级，不调 LLM）

```typescript
class ToolResultTrimStrategy implements CompactStrategy {
  name = 'tool-trim'
  description = '仅清理旧 tool 结果，不生成摘要（零 LLM 调用成本）'

  /** 保留最近 N 个 tool 结果 */
  keepRecentToolResults = 5

  async compact(history, provider, options): Promise<Message[]> {
    // 遍历 history，将旧 tool 结果替换为占位符
    // 不调用 LLM，零成本
    return trimmedHistory
  }
}
```

- **优势**：零额外 LLM 成本，快速
- **劣势**：只清理 tool 输出，对话文本不压缩
- **适用**：tool 输出是主要膨胀源的场景（大多数编码对话）

### 5.4 策略选择与配置

```typescript
// 默认策略：摘要 + 保留近期（最稳妥）
const DEFAULT_STRATEGY = 'summary-with-recent'

// 用户可通过配置或命令切换：
// config.json: { "compactStrategy": "full-replace" }
// /compact --strategy full-replace
// /compact --strategy tool-trim

// 策略注册表
const strategies = new Map<string, CompactStrategy>([
  ['full-replace', new FullReplaceStrategy()],
  ['summary-with-recent', new SummaryWithRecentStrategy()],
  ['tool-trim', new ToolResultTrimStrategy()],
])
```

**策略推荐矩阵**：

| 场景 | 推荐策略 | 理由 |
|------|----------|------|
| 日常编码（tool 调用多） | `tool-trim` | 零成本，清理 tool 输出即可释放大量空间 |
| 长对话 + Claude 模型 | `full-replace` | 200K 窗口 + Claude 摘要质量高 |
| 长对话 + GLM/DeepSeek | `summary-with-recent` | 保留近期消息兜底，摘要质量不依赖 |
| 手动精细控制 | `/compact --strategy X` | 用户按需选择 |

**auto-compact 的策略级联**：

```
auto-compact 触发（>= 95%）
  │
  ├─ 第一次尝试：tool-trim（零成本快速释放）
  │   → 如果释放后 < 70%：完成
  │   → 如果仍然 >= 70%：继续
  │
  └─ 第二次尝试：当前配置的策略（full-replace 或 summary-with-recent）
      → LLM 生成摘要，深度压缩
```

### 5.5 ContextManager 完整接口

```typescript
class ContextManager {
  #tracker: ContextTracker
  #strategy: CompactStrategy

  /** 设置压缩策略 */
  setStrategy(name: string): void

  /**
   * 准备 history — 在 AgentLoop.run() 之前调用。
   *
   * 1. 检查使用率 → 是否需要 auto-compact
   * 2. 裁剪旧 tool results
   * 3. 返回优化后的 history
   */
  async prepare(
    rawHistory: Message[],
    provider: LLMProvider,
    options: CompactOptions,
  ): Promise<{ history: Message[]; compacted: boolean }>

  /**
   * 手动 compact — /compact 命令调用。
   * 可指定策略覆盖默认。
   */
  async compact(
    rawHistory: Message[],
    provider: LLMProvider,
    options: CompactOptions & { strategy?: string },
  ): Promise<{ history: Message[]; summary: string; tokensBefore: number; tokensAfter: number }>

  /** 获取当前状态 */
  getState(): ContextWindowState
}
```

### 5.3 Codex CLI 的差异化方案

```
messages = [
  { role: 'system', content: systemPrompt },
  { role: 'user', content: handoffSummary },     ← 摘要
  ...recentMessages.slice(-N)                     ← 保留最近 ~20K token 原始消息

```

**为什么保留近期原始消息？**
- 避免"摘要→摘要"的累积信息损失
- 用户最近说的话和最近的 tool 结果最重要
- 摘要覆盖远期上下文，原始消息保证近期精度

### 5.4 多次压缩的信息衰减

```
第 1 次 compact: 150K → 40K（保留 ~80% 关键信息）
第 2 次 compact: 再次增长到 150K → 40K（第 1 次的摘要被再次摘要）
第 3 次 compact: 早期信息已经过两轮摘要，精度显著下降

信息衰减曲线：
信息保留率 ≈ 0.8^(compact次数)
3 次后: ~51%
5 次后: ~33%
```

**缓解策略**：
- Memory 系统：关键信息写入文件，不受 compact 影响
- Focus 参数：指定保留重点，减少不相关信息的摘要占比
- 近期消息保留（Codex 方案）：最近的上下文不压缩

## 七、设计决策（已确认）

### 决策 1：Compact 策略 → 确定方案 B（摘要 + 保留近期）

| 方案 | 优势 | 劣势 |
|------|------|------|
| A. 全量替换（Claude Code） | 压缩率最高，实现简单 | 多次压缩信息损失大 |
| **✅ B. 摘要+保留近期（Codex CLI）** | 近期精度高，渐进降级 | 压缩率略低，实现较复杂 |
| C. A+B 混合 | 兼顾两者 | 需要更精细的策略 |

**理由**：
- cCli 面向的模型多样（GLM/DeepSeek 的摘要质量不如 Claude），保留原始消息是安全网
- 近期上下文最有价值，不值得压缩
- 多次 compact 时，近期消息保证精度不至于快速衰减

### 决策 2：tool 输出管理 → 确定方案 B（客户端裁剪），Context Editing 后续

| C. API Context Editing | Anthropic API beta 能力，暂不依赖 |

**理由**：
- Context Editing 是 beta 功能且仅限 Anthropic，不适合作为基础方案
- 客户端裁剪所有 provider 统一可用，实现可控
- 后续 Anthropic Context Editing 正式发布后可作为增强层叠加

### 决策 3：记忆实现 → 确定方案 B（自实现 Memory 工具）

| 方案 | 说明 |
|------|------|
| A. Anthropic API 内置 `memory_20250818` | 仅限 Anthropic provider |
| **✅ B. 自实现 Memory 工具** | 所有 provider 通用 |
| C. B + 自动记忆提取 | 后续迭代 |

**理由**：
- 多模型架构，不依赖 Anthropic 专有工具类型

### Phase 3: Memory 工具（P2，跨会话知识）

```
目标：LLM 能主动积累和检索跨会话知识

实现方式：实现一个普通的 MemoryTool（implements Tool），注册到 ToolRegistry
  → 不需要定义新的 Tool 接口，复用现有 Tool.execute(args, ctx)
  → 所有 provider 统一可用（Claude/GLM/DeepSeek/OpenAI）
  → 参数设计参考 Anthropic API 内置工具的 6 个命令

存储路径（项目级隔离，对齐 Claude Code）：
  ~/.ccode/projects/<project-slug>/memory/
  ├── MEMORY.md           ← 索引文件（前 200 行加载到 system prompt）
  └── *.md                ← 具体记忆文件（LLM 按需 view 读取）

参数设计（LLM 通过 args.command 路由，和 bash 工具接收 command 参数同一模式）：
  memory({ command: "view", path: "/memories" })              → 目录列表 / 文件内容
  memory({ command: "create", path: "notes.md", file_text })  → 创建新记忆
  memory({ command: "str_replace", path, old_str, new_str })  → 更新记忆
  memory({ command: "insert", path, insert_line, insert_text })→ 插入内容
  memory({ command: "delete", path })                         → 删除记忆
  memory({ command: "rename", old_path, new_path })           → 重命名

系统提示注入 Memory Protocol：
  "ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE.
   As you make progress, record status/progress/thoughts in your memory.
   ASSUME INTERRUPTION: context window might be reset at any moment."

与 /compact 联动：
  - compact 前 LLM 先将关键信息写入 memory
  - compact 后 LLM 从 memory 读回需要的上下文
  - memory 工具调用在 Context Editing 中标记为 exclude（不被清除）

改动：
  - 新增 src/tools/memory.ts — Memory 工具实现
  - 新增 src/core/memory-store.ts — 文件级存储（路径安全校验）
  - bootstrap.ts: 注册 MemoryTool + 启动时加载 MEMORY.md 到 system prompt
  - system-prompt: 注入 Memory Protocol
预估：4-6h
```
