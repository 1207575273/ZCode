# 上下文窗口与记忆系统 — 架构分析与规划

> 日期: 2026-03-19
> 状态: 设计中（待打磨）
> 数据来源: Claude Code CLI 逆向分析、Anthropic 官方文档、Codex CLI / OpenCode / Amp 横向对比

---

## 一、问题陈述

ZCli 当前的对话机制是**无限增长的朴素累积**——每轮 LLM 调用把完整历史发送一次，没有任何裁剪、压缩或记忆机制。这意味着：

1. **长对话必崩**：200K 上下文窗口用完后 API 直接报错，用户丢失所有上下文
2. **无法感知风险**：用户不知道当前用了多少 token，没有任何预警
3. **tool 输出无控制**：一次 `grep` 返回 5000 行结果全部进入历史，快速吞噬窗口
4. **跨会话无记忆**：每次新对话从零开始，无法积累项目知识和用户偏好
5. **成本浪费**：重复发送大量已过时的 tool 输出，浪费输入 token 费用

这是 ZCli 走向生产可用的**最大架构短板**。

## 二、业界方案横向对比

### 2.1 Claude Code（Anthropic 官方）

**三层递进防线**：

| 层级 | 机制 | 触发条件 | 效果 |
|------|------|----------|------|
| **第一层** | Context Editing（API 层） | input tokens 达到配置阈值 | 服务端自动清除旧 tool_use 结果，保留最近 N 个 |
| **第二层** | /compact 压缩 | 手动或 ~98% 自动触发 | 对话历史 → LLM 生成结构化摘要，60-80% 压缩率 |
| **第三层** | Memory 持久记忆 | LLM 主动判断 | 文件级跨会话记忆，分 user/feedback/project/reference 四类 |

**Context Editing 详细机制**（API 参数）：
```typescript
context_management: {
  edits: [{
    type: 'clear_tool_uses_20250919',
    trigger: { type: 'input_tokens', value: 100000 },  // 触发阈值
    keep: { type: 'tool_uses', value: 3 },              // 保留最近 3 个
    exclude_tools: ['memory'],                           // memory 工具不清除
  }]
}
```

**Compact 压缩 Prompt**：
```
Summarize our conversation above. This summary will be the only
context available when the conversation continues, so preserve
critical information including:
- What was accomplished（已完成的工作）
- Current work in progress（正在进行的工作）
- Files involved（涉及的文件）
- Next steps（下一步计划）
- Key user requests or constraints（关键用户要求和约束）
```

**Memory 工具**：
- 类型：`memory_20250818`（API 内置工具）
- 操作：view / create / str_replace / insert / delete / rename
- 存储：客户端实现，Claude 通过 tool_use 调用
- 与 Context Editing 配合：清除前存记忆，清除后按需读回

### 2.2 Codex CLI（OpenAI）

```
token 阈值触发 → 压缩 prompt:
"Create a handoff summary for another LLM that will resume the task.
 Include: Current progress, Important context, What remains, Critical data"

关键差异：保留最近 ~20K token 原始消息 + 摘要（不全部替换）
```

### 2.3 OpenCode（SST）

```
双重机制：
1. token overflow → 完整对话摘要
2. tool 输出修剪 → 超过 40K 保护窗口的旧 tool 输出被裁剪（需可回收 ≥20K）
   → 标记为 assistant message with "summary" type
```

### 2.4 Amp（Sourcegraph）

```
哲学："everything in the context window has an influence on the output"
完全不做自动压缩，改用：
- Handoff：分析线程 → 提取关键信息 → 开新线程
- Fork：在特定点复制上下文
- Thread References：按需引用其他线程信息
```

### 2.5 方案对比总结

| 维度 | Claude Code | Codex CLI | OpenCode | Amp |
|------|------------|-----------|----------|-----|
| 压缩触发 | 手动+自动(98%) | 自动(95%) | 自动(overflow) | 不压缩 |
| 摘要策略 | 全量替换 | 摘要+保留近期 | 全量替换 | Handoff/Fork |
| tool 输出管理 | API Context Editing | 无 | 40K 保护窗口修剪 | 无 |
| 记忆系统 | Memory 工具(API) | 无 | 无 | Thread References |
| 多次压缩 | 信息累积损失 | 有降级警告 | 有降级警告 | 不适用 |

## 三、ZCli 当前状态

### 3.1 已有能力

| 能力 | 实现位置 | 状态 |
|------|----------|------|
| token 使用量记录 | `observability/token-meter.ts` | ✅ 但仅用于计费，不用于上下文管理 |
| token 计数估算 | `providers/anthropic.ts:countTokens()` | ⚠️ 简单 `text.length/4` 启发式 |
| 最大轮次限制 | `agent-loop.ts:maxTurns=50` | ✅ 防止无限循环，但不防 token 溢出 |
| 系统提示缓存 | `bootstrap.ts:cachedSystemPrompt` | ✅ 构建一次复用，但缺 cache_control |
| 会话持久化 | `persistence/session-store.ts` | ✅ JSONL 存储，支持 resume/fork |
| compact hook 预留 | `bootstrap.ts:runSessionStartHooks('compact')` | ⚠️ 占位符，未实现 |

### 3.2 完全缺失

| 能力 | 影响 |
|------|------|
| **上下文使用率追踪** | 用户不知道何时快满，无法主动干预 |
| **`/compact` 命令** | 长对话无法压缩，满了直接崩 |
| **auto-compact** | 无自动防线，完全依赖用户手动管理 |
| **tool 结果裁剪** | 一次大输出可以瞬间吞噬大量窗口 |
| **Context Editing** | 未利用 Anthropic API 的服务端清理能力 |
| **Memory 工具** | 跨会话无知识积累 |
| **上下文组成分析** | 无法知道窗口被什么占满了（system prompt? tool 输出? 历史?） |

### 3.3 消息流当前架构

```
useChat.submit()
  → 构建 messages: Message[]
    → 从 state 读取全部历史（无裁剪）
  → new AgentLoop(provider, registry, config)
  → agentLoop.run(messages)
    → 每轮：provider.chat({ messages: [...history], tools, systemPrompt })
    → tool 结果 append 到 history
    → 下一轮带着更大的 history 继续
    → 直到 LLM 不再调用工具 或 maxTurns
  → 返回所有事件

问题：history 只增不减，每轮 LLM 调用的 inputTokens 持续增长
```

## 四、/compact 核心算法深度解析

### 4.1 触发路径

```
用户执行 /compact [focus]  或  auto-compact 检测到阈值
  │
  ├─ 1. 预清理（减少发送量）
  │     → 剥离 image content blocks
  │     → 移除 PDF document blocks
  │     → 清除空 "(no content)" 占位文本
  │
  ├─ 2. 构建压缩请求
  │     → system: "You are a conversation summarizer..."
  │     → messages: [完整对话历史]
  │     → user: compact prompt（见下方）
  │     → 可选 focus 参数聚焦特定方面
  │
  ├─ 3. 调用 LLM 生成摘要
  │     → 使用当前模型（不切换低成本模型）
  │     → 返回结构化摘要文本
  │
  ├─ 4. 替换对话历史
  │     → messages = [{ role: 'user', content: compactSummary }]
  │     → 保留 session 元数据
  │     → 保留 TodoWrite 状态
  │     → 保留 plan mode 状态
  │
  └─ 5. 后续对话基于摘要继续
        → LLM 基于摘要理解上下文
        → 精度有损但可持续工作
```

### 4.2 摘要内容结构

Claude Code 的 compact 产出大致包含：

```markdown
## 已完成的工作
- 实现了 SubAgent 后台运行功能
- 创建了 subagent-store.ts 内存状态管理
- 完成了 Web 端 SubAgentCard 折叠卡片

## 正在进行的工作
- Skills 市场安装功能调试中
- npx skills add 的临时目录方案已实现

## 涉及的关键文件
- src/tools/dispatch-agent.ts — run_in_background 逻辑
- src/tools/subagent-store.ts — 内存状态
- web/src/components/SubAgentCard.tsx — Web UI

## 下一步计划
- 测试 skills 安装端到端流程
- 实现上下文窗口管理

## 用户约束和偏好
- 中文注释和文档
- 先看方案再写代码
- Git 作者用 codeYang
```

### 4.3 Codex CLI 的差异化方案

```
messages = [
  { role: 'system', content: systemPrompt },
  { role: 'user', content: handoffSummary },     ← 摘要
  ...recentMessages.slice(-N)                     ← 保留最近 ~20K token 原始消息
]
```

**为什么保留近期原始消息？**
- 避免"摘要→摘要"的累积信息损失
- 用户最近说的话和最近的 tool 结果最重要
- 摘要覆盖远期上下文，原始消息保证近期精度

### 4.4 多次压缩的信息衰减

```
第 1 次 compact: 150K → 40K（保留 ~80% 关键信息）
第 2 次 compact: 再次增长到 150K → 40K（第 1 次的摘要被再次摘要）
第 3 次 compact: 早期信息已经过两轮摘要，精度显著下降

信息衰减曲线：
信息保留率 ≈ 0.8^(compact次数)
3 次后: ~51%
5 次后: ~33%
```

**缓解策略**：
- Memory 系统：关键信息写入文件，不受 compact 影响
- Focus 参数：指定保留重点，减少不相关信息的摘要占比
- 近期消息保留（Codex 方案）：最近的上下文不压缩

## 五、设计决策点（待讨论）

### 决策 1：Compact 策略选择

| 方案 | 优势 | 劣势 |
|------|------|------|
| **A. 全量替换**（Claude Code） | 压缩率最高，实现简单 | 多次压缩信息损失大 |
| **B. 摘要+保留近期**（Codex CLI） | 近期精度高，渐进降级 | 压缩率略低，实现较复杂 |
| **C. Handoff 开新会话**（Amp） | 无信息损失 | 用户体验断裂 |
| **D. A+B 混合** | 兼顾两者 | 需要更精细的策略 |

**倾向方案 B**——参考 Codex CLI，摘要 + 保留最近 N 条原始消息。理由：
- ZCli 面向的模型多样（GLM/DeepSeek 的摘要质量不如 Claude），保留原始消息是安全网
- 近期上下文最有价值，不值得压缩

### 决策 2：tool 输出管理策略

| 方案 | 说明 |
|------|------|
| **A. 发送前裁剪** | tool 输出超过 N token 时截断，只保留头尾 |
| **B. 历史中清除** | 旧的 tool 结果在下次 LLM 调用前替换为 `(output truncated, use read_file to see full content)` |
| **C. API Context Editing** | 利用 Anthropic API 的 `context_management.edits` 自动清除 |
| **D. B+C 混合** | Anthropic 用 C，其他 provider 用 B |

**倾向方案 D**——Anthropic provider 利用原生 API 能力，其他 provider 客户端模拟。

### 决策 3：记忆系统范围

| 方案 | 说明 |
|------|------|
| **A. 纯文件级**（当前 CLAUDE.md） | 只读，用户手动维护 |
| **B. LLM 读写的 Memory 工具** | 类 Claude Code，LLM 通过 tool_use 操作 |
| **C. 自动记忆提取** | compact 时自动将关键信息写入 memory |
| **D. B+C 完整方案** | 工具 + 自动提取 |

**倾向方案 B 先行**，C 后续。Memory 工具让 LLM 自主决定什么值得记住，不需要复杂的自动提取规则。

### 决策 4：上下文组成可视化

Claude Code 有 `/context` 命令查看上下文被什么占满。ZCli 是否需要？

```
/context 输出示例：
  System Prompt:     2,800 tokens (1.4%)
  Tool Definitions:  8,200 tokens (4.1%)
  Conversation:    156,000 tokens (78%)
    - User messages:   12,000
    - Assistant:       44,000
    - Tool results:   100,000  ← 罪魁祸首
  Available:         33,000 tokens (16.5%)
```

## 六、实施路线（待确认）

### Phase 0: 上下文感知（P0，前置条件）

```
目标：让用户和系统知道当前上下文状态
改动：
  - agent-loop.ts: 累积 inputTokens，每轮 LLM 调用后更新
  - 新增 context-tracker.ts: 跟踪窗口使用率、各组成部分占比
  - CLI 状态栏: 显示 "Context: 45% (90K/200K)"
  - Web header: 同步显示
  - 70% 黄色警告、85% 红色警告
预估：3-4h
```

### Phase 1: /compact 手动压缩（P0，核心能力）

```
目标：用户可以主动压缩对话，或系统自动压缩
改动：
  - 新增 /compact 斜杠指令
  - 新增 compact 逻辑：构建压缩 prompt → 调 LLM → 替换历史
  - 策略选择：摘要 + 保留最近 N 条消息（Codex 方案）
  - auto-compact: 达到阈值自动触发
  - Web 端同步通知
预估：4-6h
```

### Phase 2: tool 输出管理（P1，成本优化）

```
目标：控制 tool 输出对上下文的侵蚀
改动：
  - tool 结果超过 N token 时截断（保留头尾 + 摘要）
  - 旧 tool 结果在后续轮次中替换为占位符
  - Anthropic provider: 启用 Context Editing API
预估：3-4h
```

### Phase 3: Memory 工具（P2，跨会话知识）

```
目标：LLM 能主动积累和检索跨会话知识
改动：
  - 新增 memory 工具（view/create/update/delete）
  - 存储目录：~/.zcli/memory/
  - 系统提示注入 Memory Protocol
  - 与 /compact 联动：压缩前 LLM 先存记忆
预估：4-6h
```
