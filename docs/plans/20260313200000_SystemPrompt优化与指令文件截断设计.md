# System Prompt 优化与指令文件截断设计

## 目标

优化 system prompt 的构建与发送策略，提升 LLM API 的 prompt cache 命中率，降低 token 消耗。

## 背景

与有经验的开发者讨论后达成的共识：

1. **System prompt 应该一次构建、全程复用** — 前缀不变时 Anthropic API 的 prompt caching 命中率最高
2. **指令文件首次加载 400 行** — 超出部分截断，LLM 需要时自行用 Read 工具加载完整内容
3. **启动后不动态修改 system prompt** — 必须重启生效，context 压缩也不参与

## 设计方案

### 1. 指令文件截断（400 行上限）

`formatInstructionsPrompt()` 对每个指令文件内容做行数检查：

```typescript
const MAX_INSTRUCTION_LINES = 400

if (lines.length > MAX_INSTRUCTION_LINES) {
  body = lines.slice(0, MAX_INSTRUCTION_LINES).join('\n')
  body += `\n\n[... 已截断，共 ${lines.length} 行。需要完整内容时用 Read 工具读取: ${source}]`
}
```

截断提示包含完整路径，LLM 可按需自行读取。400 行足以覆盖绝大多数项目的 CLAUDE.md / CCODE.md。

### 2. System Prompt 一次构建缓存

`bootstrap.ts` 新增 `buildSystemPrompt()` 和 `getSystemPrompt()`：

```typescript
let cachedSystemPrompt: string | undefined

export function buildSystemPrompt(hookContext: string): void {
  if (cachedSystemPrompt !== undefined) return  // 幂等
  const parts = [
    getInstructionsPrompt(),   // 指令文件（已截断）
    getSkillsSystemPrompt(),   // Skills 注册信息
    hookContext,               // SessionStart Hook 注入
  ].filter(Boolean)
  cachedSystemPrompt = parts.length > 0 ? parts.join('\n\n') : undefined
}

export function getSystemPrompt(): string | undefined {
  return cachedSystemPrompt
}
```

**调用方（REPL 的 `useChat.submit()` 和 Pipe Mode 的 `runPipe()`）**：

```typescript
// 构建 system prompt（幂等，只构建一次）
const hookContext = await runSessionStartHooks('startup')
buildSystemPrompt(hookContext)
const systemPrompt = getSystemPrompt()

// 传给 AgentLoop
const loop = new AgentLoop(provider, registry, {
  ...(systemPrompt !== undefined ? { systemPrompt } : {}),
})
```

### 3. 初始化顺序

必须按以下顺序调用，确保各组件就绪后再构建 system prompt：

```
ensureSkillsDiscovered()     → Skills 目录扫描
ensureHooksDiscovered()      → Hooks 发现
ensureMcpInitialized()       → MCP 连接
ensureInstructionsLoaded()   → 指令文件加载
runSessionStartHooks()       → 执行 SessionStart hooks
buildSystemPrompt()          → 一次性构建并缓存
```

### 4. Cache 命中优化原理

Anthropic API 的 prompt caching 基于前缀匹配：

```
请求 1: [system prompt][user msg 1]
请求 2: [system prompt][user msg 1][assistant msg 1][user msg 2]
                        ↑ 前缀完全一致 → cache 命中
```

每次请求使用**同一个字符串引用**的 system prompt，前缀始终不变。如果每次重新拼接（字符串内容可能因浮点精度、时间戳等微妙差异而不同），cache 可能失效。

| 文件 | 改动 |

## 决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 截断行数 | 400 行 | 覆盖绝大多数 CLAUDE.md，过长文件用 Read 按需加载 |
| 缓存策略 | 模块级变量 + 幂等函数 | 最简实现，无需额外依赖，符合"启动后不变"语义 |
| Hook 注入时机 | system prompt 构建前一次性注入 | SessionStart hook 只在启动/resume/compact 时执行 |
| 重新构建 | 需重启 | 保证 cache 前缀稳定，运行期间不热更新 |
