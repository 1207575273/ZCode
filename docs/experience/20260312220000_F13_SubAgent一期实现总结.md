# F13 SubAgent 一期实现总结

> 日期: 2026-03-12 | 特性: F13 SubAgent（父子 Agent）| 状态: 一期完成

---

## 一、实现目标

让主 Agent 能通过 `dispatch_agent` 工具派发子 Agent 独立执行任务。子 Agent 拥有完整的 AgentLoop（多轮 LLM + 工具调用），执行完毕后将结果作为 tool_result 返回给主 Agent。每个子 Agent 的完整事件历史写入独立 JSONL 文件，支持审计回溯。

---

## 二、核心设计决策

### 2.1 StreamableTool 接口（借鉴 LangChain invoke/stream 分离）

```typescript
// 基础工具接口（不动）
interface Tool {
  execute(args, ctx): Promise<ToolResult>
}

// 流式工具接口（新增）
interface StreamableTool extends Tool {
  stream(args, ctx): AsyncGenerator<unknown, ToolResult>
}
```

- Tool 接口完全不变，现有工具零改动
- StreamableTool 新增 `stream()` 方法，yield 中间事件，return 最终结果
- AgentLoop 通过 `isStreamableTool()` 类型守卫选择路径
- 事件流统一走 generator `yield*` 链路，与 `#callLLM` 模式一致，无回调

### 2.2 权限模型

- `dispatch_agent` 标记 `dangerous=false`（本身不执行危险操作）
- 子 Agent 的 AgentConfig 设置 `isSidechain: true`
- `#checkPermission` 检测到 isSidechain 直接返回 true（主 Agent 派发即授权）

### 2.3 双重递归防护

| 层级 | 机制 |
|------|------|
| 工具层 | `registry.cloneWithout('dispatch_agent')` — 子 Agent 工具集不含 dispatch_agent |
| 提示词层 | "Do NOT dispatch further sub-agents" — 兜底 |

### 2.4 独立 JSONL 审计

目录结构（与 Claude Code 对齐）：

```
~/.ccode/sessions/<projectSlug>/
  YYYYMMDDHHMMSSMMM_<sessionId>.jsonl              ← 主会话
  <sessionId>/
    subagents/
      agent-<agentId>.jsonl                          ← 子 Agent 完整日志
```

子 Agent JSONL 包含完整事件链：`session_start → user → llm_call_start → llm_call_end → [tool_call*] → assistant → session_end`，标记 `isSidechain: true` 和 `parentSessionId`。

## 四、踩坑记录

### 4.1 并行执行器缺失 ToolContext 字段

**现象：** dispatch_agent 报错 "需要 ToolContext 中的 provider 和 registry"，durationMs 为 0。

**原因：** dispatch_agent 标记 `dangerous=false`，被 `classifyToolCalls` 归为 safe 走并行路径。并行执行器传给工具的 ctx 只有 `{ cwd }`，缺少 provider/registry。

**修复：**
1. `classifyToolCalls` 中 StreamableTool 强制归为 dangerous（走串行 `#executeOneTool` → `yield* stream()`）
2. 并行路径的 ctx 也改用完整的 `buildToolContext()`（兜底保护）

### 4.2 exactOptionalPropertyTypes 类型错误

**现象：** `{ signal: config.signal }` 当 signal 为 undefined 时 TS 报错。

**修复：** 提取 `buildToolContext()` 辅助函数，用 `if (x !== undefined)` 条件赋值。

### 4.3 GLM-5 模型重复派发

**现象：** 用户要求 2 个子 Agent，实际产生了 4 个。主 Agent 第一轮派发 2 个成功返回后，第二轮又重新派发了一遍。

**结论：** GLM-5 在"什么时候该停止调用工具"这个判断上偏弱，属模型行为问题，非代码 bug。
