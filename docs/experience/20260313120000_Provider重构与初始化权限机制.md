## 二、Provider 重构

### 2.1 问题背景

- 原 `AnthropicProvider` 基于 LangChain `ChatAnthropic`，中间层对 `baseURL`/`authToken` 等参数透传不可靠
- 接入 MiniMax 等兼容 Anthropic API 的第三方 provider 时，认证头（`Authorization: Bearer`）无法正确设置
- provider 协议路由硬编码在 `NATIVE_ANTHROPIC_PROVIDERS` 集合中，新增 provider 需改代码

### 2.2 改动内容

**`src/providers/anthropic.ts` — 去 LangChain，直接用 `@anthropic-ai/sdk`**

```typescript
// 旧：LangChain 中间层
import { ChatAnthropic } from '@langchain/anthropic'
const model = new ChatAnthropic({ apiKey, model })

// 新：官方 SDK 直接调用
import Anthropic from '@anthropic-ai/sdk'
this.#client = new Anthropic({ apiKey, baseURL, defaultHeaders })
const stream = this.#client.messages.stream({ model, messages, tools })
```

关键改进：
- 流式事件直接映射：`content_block_delta` → `text` chunk，`finalMessage` → `tool_call` + `usage`
- 第三方兼容 API 认证：`defaultHeaders: { 'Authorization': 'Bearer ${apiKey}' }`
- 消息格式转换：内部 `Message` → Anthropic SDK `MessageParam`，支持 `tool_use`/`tool_result` 多轮
- `countTokens` 简化为字符数估算（不再依赖 LangChain 的 token 计数）

**`src/config/config-manager.ts` — 新增 `protocol` 字段**

```typescript
export interface ProviderConfig {
  apiKey: string
  baseURL?: string
  protocol?: 'anthropic' | 'openai'  // 新增：显式声明协议类型
  models: string[]
}
```

**`src/providers/registry.ts` — config 驱动协议路由**

```typescript
// 旧：硬编码集合
const NATIVE_ANTHROPIC_PROVIDERS = new Set(['anthropic', 'minimax'])

// 新：基于 config.protocol 动态判断
function resolveProtocol(providerName: string, protocol?: 'anthropic' | 'openai') {
  if (protocol) return protocol
  return providerName === 'anthropic' ? 'anthropic' : 'openai'
}
```

### 2.3 配置示例

```jsonc
// ~/.zcli/config.json
{
  "providers": {
    "anthropic": { "apiKey": "sk-xxx", "models": ["claude-sonnet-4-6"] },
    "minimax": {
      "apiKey": "xxx",
      "baseURL": "https://api.minimaxi.com/anthropic",
      "protocol": "anthropic",          // ← 显式声明走 Anthropic 协议
      "models": ["MiniMax-M2.5"]
    },
    "glm": {
      "apiKey": "xxx",
      "baseURL": "https://open.bigmodel.cn/api/coding/paas/v4",
      // 不写 protocol → 默认走 openai 兼容
      "models": ["glm-5"]
    }
  }
}
```

新增任何 provider 只需改配置文件，零代码改动。

## 三、启动初始化机制

### 3.1 问题背景

- 首次使用时 `~/.zcli/` 目录和配置文件可能不存在，导致运行报错
- 配置文件损坏时没有恢复机制
- 缺少项目级配置目录（`.zcli/`），无法承载权限等项目级设置

### 3.2 `src/core/initializer.ts`

统一前置检查，`bin/zcli.ts` 入口最早调用 `initialize()`：

| 检查项 | 行为 |
|--------|------|
| `~/.zcli/` 目录 | 不存在则创建 |
| `~/.zcli/config.json` | 不存在 → 写默认模板；存在但缺字段 → 补全；JSON 损坏 → 备份 `.bak` + 重置 |
| `~/.zcli/.mcp.json` | 不存在 → 写空模板 `{ "mcpServers": {} }` |
| `<项目>/.zcli/settings.local.json` | 不存在 → 写空权限模板 `{ "permissions": { "allow": [] } }` |
| 启动诊断 | 当前 provider 的 apiKey 为空 → stderr 警告 |

```typescript
// bin/zcli.ts
const initResult = initialize()
if (initResult.created.length > 0) {
  process.stderr.write(`[init] 已创建: ${initResult.created.join(', ')}\n`)
}
for (const warn of initResult.warnings) {
  process.stderr.write(`[init] ⚠ ${warn}\n`)
}
```

### 3.3 设计原则

- **幂等**：已存在的文件不会被覆盖
- **非阻塞**：诊断失败、备份失败不阻塞启动
- **诊断信息走 stderr**：不污染交互 UI 或管道 stdout

## 四、项目级权限机制

### 4.1 问题背景

- 原有权限机制：`dangerous=true` 的工具每次都弹 PermissionDialog 确认
- 只有 session 级白名单（选 "always" 后本次会话免确认），重启后失效
- 缺少持久化的项目级权限配置

### 4.2 `src/config/permissions.ts`

```typescript
export class PermissionManager {
  constructor(rules: string[], registeredTools: string[])
  isAllowed(toolName: string): boolean
  static fromProjectDir(projectDir?: string, registeredTools?: string[]): PermissionManager
}
```

**规则格式**（兼容 Claude Code 风格）：

| 规则 | 含义 | 匹配原理 |
|------|------|----------|
| `Bash(*)` | bash 工具免确认 | 友好名 "bash" → 注册表精确匹配 "bash" |
| `Read(*)` | read_file 免确认 | 友好名 "read" → 注册表前缀匹配 "read_file" |
| `Edit(*)` | edit_file 免确认 | 友好名 "edit" → 注册表前缀匹配 "edit_file" |
| `mcp__*` | 所有 MCP 工具免确认 | 前缀通配符 |
| `mcp__ctx7__query` | 特定 MCP 工具 | 精确匹配 |
| `bash` | 直接用内部工具名 | 精确匹配 |

**核心设计：基于 ToolRegistry 动态解析，不硬编码工具映射表。**

友好名解析流程：
1. 提取 `FriendlyName(*)` 中的名称，转小写
2. 在 `registeredTools` 中查找精确匹配（如 "bash" → "bash"）
3. 找不到则查找前缀+下划线匹配（如 "read" → "read_file"）
4. 仍找不到 → 规则被忽略（不报错）

### 4.3 权限检查优先级

```
工具调用
  ↓
① isSidechain（子 Agent）→ 自动放行
  ↓
② isDangerous === false → 自动放行
  ↓
③ 项目级白名单（settings.local.json）→ 匹配则放行
  ↓
④ Session 级白名单（用户选 "always"）→ 匹配则放行
  ↓
⑤ PermissionDialog 弹窗询问用户
```

### 4.4 接入点

**交互模式（`useChat.ts`）**：

```typescript
// 首次 submit 时基于实际注册工具构建权限管理器
if (!permissionManagerRef.current) {
  const toolNames = registry.getAll().map(t => t.name)
  permissionManagerRef.current = PermissionManager.fromProjectDir(process.cwd(), toolNames)
}

// permission_request 事件处理
if (permissionManagerRef.current?.isAllowed(event.toolName)) {
  event.resolve(true)  // 项目级白名单放行
} else if (allowedToolsRef.current.has(event.toolName)) {
  event.resolve(true)  // session 级白名单放行
} else {
  setPendingPermission(...)  // 弹窗询问
}
```

**管道模式（`pipe-runner.ts`）**：

```typescript
const toolNames = registry.getAll().map(t => t.name)
const pm = PermissionManager.fromProjectDir(process.cwd(), toolNames)
if (pm.isAllowed(event.toolName)) {
  event.resolve(true)         // 项目级白名单放行
} else {
  event.resolve(options.yes)  // --yes 全部放行，否则拒绝
}
```

### 4.5 配置文件示例

```jsonc
// <项目>/.zcli/settings.local.json
{
  "permissions": {
    "allow": [
      "Bash(*)",           // bash 命令免确认
      "Read(*)",           // 读文件免确认
      "Write(*)",          // 写文件免确认
      "Edit(*)",           // 编辑文件免确认
      "Glob(*)",           // 文件搜索免确认
      "Grep(*)",           // 内容搜索免确认
      "mcp__*"             // 所有 MCP 工具免确认
    ]
  }
}
```

默认模板为 `allow: []`（空），遵循原有询问机制。用户按需添加规则。

## 七、经验总结

1. **去中间层**：LangChain 封装对标准场景好用，但第三方兼容 API 的认证/baseURL 透传不可靠。直接用官方 SDK 更可控，流式事件映射也更精确。
2. **config 驱动 > 代码硬编码**：`protocol` 字段让新增 provider 变成纯配置操作，消除了 `NATIVE_ANTHROPIC_PROVIDERS` 集合的维护负担。
3. **权限基于注册表**：不硬编码工具映射表，而是从 ToolRegistry 实际注册的工具名动态推导。新增/移除工具时权限系统自动适配。
4. **初始化防御**：配置文件损坏时先备份再重置，比直接覆盖更安全；诊断信息走 stderr 不干扰正常输出。
