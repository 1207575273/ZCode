# Skills 二期 — 外置插件加载与 Hook 系统实现总结

## 一、功能概述

在 Skills 一期（内置三层发现 + SkillTool 工具调用）基础上，新增三大能力：

1. **外置插件包加载** — 兼容 superpowers 等 Claude Code 插件目录结构，自动扫描 `~/.ccode/plugins/` 和 `.ccode/plugins/`
2. **命名空间** — 插件内 Skill 自动加 `<插件名>:` 前缀（如 `superpowers:brainstorming`），避免命名冲突
3. **Hook 事件系统** — SessionStart / PreToolUse / PostToolUse 三种事件钩子，支持插件包自带 Hook 和用户自定义 Hook

---

## 二、核心架构变更

### 2.1 SkillStore 四源扫描

```
builtin → plugin → user → project（优先级递增，后者覆盖前者）
```

新增 `#scanPlugins()` 方法，扫描用户级/项目级插件目录，为每个插件包内的 Skill 添加命名空间前缀。

### 2.2 Skill 支撑文件元数据

`skill` 工具返回内容时，在 body 前附加 `<skill-context>` 标签：

```xml
<skill-context name="superpowers:brainstorming" base-dir="/path/to/skills/brainstorming">
Supporting files (use Read tool to access):
- visual-companion.md
- spec-document-reviewer-prompt.md
</skill-context>
```

LLM 据此知道有哪些支撑文件可用 Read 工具读取。

### 2.3 Hook 系统

```
HookRunner（子进程执行 + JSON 解析 + 超时控制）
    ↑
HookManager（发现合并 + matcher 匹配 + 调度执行）
    ↑
bootstrap.ts（初始化 + SessionStart 执行 + 导出 hookManager）
    ↑
agent-loop.ts（PreToolUse / PostToolUse 工具前后钩子）
```

三层来源叠加（插件 → 项目级 → 用户级），同事件多个 Hook 顺序执行。

## 五、兼容性

### 与 superpowers 兼容

- `skills/*/SKILL.md` 直接扫描，完全兼容
- `hooks/hooks.json` → SessionStart hook 自动执行
- `additionalContext` / `hookSpecificOutput.additionalContext` / `additional_context` 三种格式均兼容
- 命名空间 `superpowers:xxx` 天然匹配

### 安装方式

```bash
ln -s /path/to/superpowers ~/.ccode/plugins/superpowers
```

### 一期行为不变

- builtin/user/project 三层 Skill 无命名空间
- 现有 `skill({ name: "commit" })` 正常工作
- system prompt 格式兼容

## 六、设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 插件加载方式 | 目录扫描（A 方案） | 工作量小、实用性强，不需要 marketplace |
| 命名空间 | `<插件名>:<skill名>` | 与 Claude Code 一致，避免冲突 |
| Hook 范围 | 基础集（3 种事件） | 覆盖启动注入 + 工具拦截，为未来扩展留基础 |
| Hook 来源 | 三层叠加 | 与 Claude Code 行为一致，灵活性高 |
| 支撑文件 | `<skill-context>` 元数据 | 不预加载全部文件（避免 token 浪费），让 LLM 按需读取 |
| HookRunner 超时 | 手动 setTimeout + kill | Windows 下 spawn 的 timeout 选项不可靠 |
