# 备用屏幕缓冲区与终端 Resize 渲染优化总结

## 背景问题

cCli 终端 UI（基于 Ink/React）存在两个渲染残留问题：

1. **首次发消息时 WelcomeScreen 闪现**：用户输入第一条消息后，旧的 WelcomeScreen 内容短暂出现在 ChatView 上方，然后才被清除。
2. **终端拉伸时内容错乱**：无论是 WelcomeScreen 还是 ChatView，拉伸终端窗口时旧帧内容残留，布局混乱。

## 根因分析

### 问题 1：useEffect 清屏时序

原方案在 `useEffect` 中执行 `process.stdout.write('\x1b[2J\x1b[H')` 清屏：

```typescript
// ❌ 旧方案
useEffect(() => {
  if (started && !hasClearedRef.current) {
    hasClearedRef.current = true
    process.stdout.write('\x1b[2J\x1b[H')
  }
}, [started])
```

**问题**：`useEffect` 在 React **渲染之后**才执行，但 Ink 的 `<Static>` 组件在**渲染期间**就冻结帧内容。时序：

```
setMessages() → React render → <Static> 冻结（WelcomeScreen 残留在此帧中）→ useEffect 清屏（太晚了）
```

### 问题 2：Ink 差分渲染 vs 终端宽度变化

Ink 使用差分渲染策略：对比上一帧，只更新变化的行。当终端宽度突变时，旧帧按旧宽度排列的字符无法被正确覆盖（行数、字符位置都变了），导致残留。

## 解决方案

### 方案 A：备用屏幕缓冲区（Alternate Screen Buffer）

采用与 vim/less/htop 相同的机制 — `\x1b[?1049h` / `\x1b[?1049l`：

- 进入对话时切换到**备用屏幕**（独立画布），WelcomeScreen 内容完全隔离
- 退出 app 时还原**主屏幕**，终端恢复到进入前的状态
- 备用屏幕内 resize 不会泄露主屏幕内容

**关键：清屏必须在状态变化之前同步执行**，而非 `useEffect` 中异步执行：

```typescript
// ✅ 新方案：在 handleSubmit 回调中同步切换
if (!hasClearedRef.current) {
  hasClearedRef.current = true
  enterAlternateScreen()  // \x1b[?1049h\x1b[2J\x1b[H
}
submit(trimmed)  // 之后才触发 state 变化 → React 重渲染
```

覆盖所有入口路径：
- `handleSubmit` 中的普通消息提交
- `handleSubmit` 中的 Skill fallback 提交
- `--resume` 启动参数触发的 `loadSession`
- ResumePanel 选择恢复会话

退出时在 `bin/ccli.ts` 的 `exitGracefully()` 和 `process.on('exit')` 中调用 `leaveAlternateScreen()`，确保 resume 提示打印在主屏幕上。

### 方案 B：Debounce Resize + 清屏

改造 `useTerminalSize` hook：

```typescript
const onResize = () => {
  if (timerRef.current) clearTimeout(timerRef.current)
  timerRef.current = setTimeout(() => {
    // 清屏：可见区域 + scrollback 缓冲区
    process.stdout.write('\x1b[2J\x1b[3J\x1b[H')
    // 更新 state 触发一次干净的 React 重渲染
    setColumns(stdout.columns ?? 80)
    setRows(stdout.rows ?? 24)
  }, RESIZE_DEBOUNCE_MS)  // 150ms
}
```

效果：拉伸过程中不频繁重绘，停止 150ms 后统一清屏 + 重绘一帧。

## 终端转义序列速查

| 序列 | 作用 | 场景 |
|------|------|------|
| `\x1b[2J` | 清除可见区域 | 基础清屏 |
| `\x1b[3J` | 清除 scrollback 缓冲区 | 防止滚动回看到旧内容 |
| `\x1b[H` | 光标归位到 (0,0) | 配合清屏使用 |
| `\x1b[?1049h` | 进入备用屏幕 | 全屏 TUI 应用（vim/less） |
| `\x1b[?1049l` | 离开备用屏幕，还原主屏幕 | TUI 退出时 |

| 文件 | 改动 |

## 经验要点

1. **Ink + `<Static>` 的时序陷阱**：`<Static>` 在 render 时冻结内容，`useEffect` 在 render 后才执行。需要副作用在 render 前生效时，必须在触发 state 变化的回调中同步执行。
2. **差分渲染 vs 终端尺寸**：Ink 的差分渲染假设终端宽度不变。宽度突变时必须清屏重绘，debounce 避免频繁清屏闪烁。
3. **备用屏幕是全屏 TUI 的标准做法**：Claude Code 等成熟 CLI 工具都使用备用屏幕隔离对话内容和原终端。退出时还原，体验一致。
4. **`\x1b[3J` 不要忘记**：只清可见区域不清 scrollback，resize 或滚动时旧内容会"浮"出来。
