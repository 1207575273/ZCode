# Windows/MSYS 平台兼容性修复总结

> 日期：2026-03-12
> 提交：`fix(tools): Windows/MSYS 平台兼容性修复 — 路径转换 + bash 超时/后台 + SubAgent 信任`

---

## 一、背景

使用 GLM-5 模型在 Windows + Git Bash 环境下运行 ZCli Demo 时，连续暴露出三类平台兼容性问题：

1. **文件工具路径解析错误**：Git Bash 返回的 cwd 是 MSYS 格式 `/c/Users/...`，Node.js `path.resolve()` 将其误解为相对路径，拼出 `C:\c\Users\...`
2. **bash 工具超时与后台能力不足**：默认 30s 超时对 `mvn spring-boot:run` 等长任务太短；缺少后台运行机制
3. **SubAgent 结果不被信任**：弱模型（GLM-5）收到子 Agent 结果后，不信任并重复执行相同操作，产生 23 轮无意义循环

---

## 二、问题分析与修复

### 2.1 P0 — MSYS 路径转换 Bug

**根因**：

```
cwd = '/c/Users/ThinkPad/Desktop/demo'   // Git Bash 返回的 MSYS 格式
path.resolve(cwd, 'hello.py')
→ 'C:\c\Users\ThinkPad\Desktop\demo\hello.py'  // ❌ 多了个 \c\
```

Windows 的 `path.resolve()` 看到 `/c/Users/...` 时，识别出前导 `/` 认定为"当前盘根目录下的路径"，于是拼出 `C:\` + `c\Users\...`。

**修复方案**：

新增 `src/platform/path-utils.ts`，提供 `resolvePath(cwd, rawPath)` 函数：

```typescript
// MSYS 格式 /c/Users/... → Windows 格式 C:\Users\...
function msysToWin(p: string): string {
  const match = /^\/([a-zA-Z])(\/.*)?$/.exec(p)
  if (!match) return p
  const drive = match[1]!.toUpperCase()
  const rest = (match[2] ?? '').replace(/\//g, '\\')
  return `${drive}:${rest || '\\'}`
}

export function resolvePath(cwd: string, rawPath: string): string {
  const { isWindows } = detectPlatform()
  if (isWindows) {
    return resolve(msysToWin(cwd), msysToWin(rawPath))
  }
  return resolve(cwd, rawPath)
}
```

**影响范围**：`read-file`、`write-file`、`edit-file`、`grep` 四个工具统一替换 `resolve()` → `resolvePath()`。

**测试**：`tests/unit/platform/path-utils.test.ts`，7 个用例覆盖 MSYS 各种格式 + 非 MSYS 路径透传。

### 2.2 P0 — bash 工具超时与后台增强

**问题清单**：

| 问题 | 原状 | 修复后 |
|------|------|--------|
| 默认超时太短 | 30s 硬编码 | 120s 默认，支持 `timeout` 参数（上限 600s） |
| 无后台运行能力 | 不支持 | 新增 `run_in_background` 参数 |
| 空命令无校验 | 直接执行空串 | 卫语句返回错误 |
| 超时无明确提示 | 通用错误信息 | `Command timed out after N milliseconds` |
| 后台进程崩溃主进程 | execa promise reject 未捕获 | `child.catch(() => {})` 静默处理 |
| `taskkill` MSYS 路径冲突 | `kill PID` 提示 | Windows 上提示 `taskkill //F //PID` |

**后台模式关键实现**：

```typescript
#runBackground(shell, command, cwd): ToolResult {
  const child = execa(shell.path, [...shell.args, command], {
    cwd, detached: true, stdio: 'ignore',
  })
  child.unref()
  // 关键：防止后台进程退出时 unhandled rejection 崩溃 Node.js
  child.catch(() => {})
  const killHint = buildKillHint(child.pid)  // 平台感知的 kill 命令
  return { success: true, output: `Background process started (pid: ${pid}). ${killHint}` }
}
```

**MSYS `taskkill` 路径转换问题**：

Git Bash (MSYS) 会自动将 `/` 开头的参数解释为 Unix 路径。`taskkill /F /PID 1234` 中的 `/F` 会被转换为 `F:\` 导致命令失败。双斜杠 `//F` 是 MSYS 约定的转义方式，表示"这是一个命令行开关，不是路径"。

```typescript
function buildKillHint(pid: number | undefined): string {
  if (pid == null) return ''
  const { isWindows } = detectPlatform()
  if (isWindows) {
    return `Use "taskkill //F //PID ${pid}" to stop it.`
  }
  return `Use "kill ${pid}" to stop it.`
}
```

**测试**：`tests/unit/tool-bash.test.ts`，11 个用例（含超时、后台、空命令、cwd 等场景）。

### 2.3 P1 — SubAgent 信任标记

**问题**：GLM-5 作为主 Agent 时，收到子 Agent 的执行结果后不信任，自行重新执行了创建文件、编译、curl 验证等全部步骤，产生 23 轮无意义循环。

**修复**：双重信任引导——

1. **工具描述层**：在 `dispatch_agent` 的 description 中加入明确指令

```
IMPORTANT: The sub-agent result is ALREADY VERIFIED.
When you receive the result, trust it and relay it to the user directly.
Do NOT re-verify, re-create files, or re-run commands that the sub-agent already completed.
```

2. **结果包装层**：`wrapSubagentResult()` 在子 Agent 返回文本前后加信任标记

```
[Sub-agent completed: 创建 Spring Boot 项目]

<子 Agent 原始输出>

[INSTRUCTION: The sub-agent has already executed all steps and verified the result.
Do NOT repeat any of the above actions. Simply relay this result to the user.]
```

**效果评估**：对 GPT-4 / Claude 等强模型效果显著，GLM-5 等弱模型仍有概率重复，属模型能力限制。

**测试**：`tests/unit/tool-dispatch-agent.test.ts` 更新 3 个断言适配包装格式。

## 四、经验教训

### 4.1 MSYS 路径是 Windows 开发的隐形地雷

Git Bash 是 Windows 上最常用的开发 Shell，但它的 MSYS 路径格式 (`/c/Users/...`) 与 Node.js 的 `path` 模块假设不兼容。**任何在 Windows 上使用 `path.resolve()` 处理外部 Shell 返回路径的地方，都需要先做 MSYS 格式检测和转换**。

### 4.2 execa 的 Promise-like 特性需要显式处理

execa 返回的 `ExecaChildProcess` 同时是 `ChildProcess` 和 `Promise`。在 `detached` 模式下，虽然调用了 `unref()`，但 Promise 链仍然存在。后台进程异常退出会产生 rejected promise，不捕获就会触发 Node.js 的 `unhandledRejection` 事件，直接崩溃主进程。**必须显式 `.catch()` 或 `await`**。

### 4.3 MSYS 的自动路径转换会破坏命令行开关

MSYS 看到 `/` 开头的参数时会尝试将其转换为 Windows 路径。这影响所有使用 `/` 前缀开关的 Windows 原生命令（`taskkill /F`、`net /y`、`reg /v` 等）。解决方案是使用双斜杠 `//F`，这是 MSYS 约定的"这不是路径"的转义方式。

### 4.4 弱模型需要更强的提示工程

对于 GLM-5 这类在工具调用场景下判断力偏弱的模型，单靠系统提示不够。需要在工具描述和返回结果中都植入行为引导，形成"双重栅栏"。即便如此，也无法 100% 保证弱模型遵从指令，这是模型能力的硬限制。
