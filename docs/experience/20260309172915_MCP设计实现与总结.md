# cCli MCP 设计实现与总结

## 一、总体架构

cCli 的 MCP（Model Context Protocol）功能实现了完整的远程/本地工具扩展能力：

```
配置加载 → 并行连接 → 工具适配 → AgentLoop 集成 → UI 状态展示
```

### 模块全景

| 模块 | 文件 | 职责 |
|------|------|------|
| 配置加载 | `src/config/mcp-config.ts` | 多文件合并、优先级、来源追踪 |
| 连接管理 | `src/mcp/mcp-manager.ts` | 并行连接、故障隔离、Transport 创建 |
| 工具适配 | `src/mcp/mcp-tool.ts` | MCP 工具 → cCli Tool 接口桥接 |
| UI 面板 | `src/ui/McpStatusView.tsx` | 全屏状态面板，Esc/q 退出 |
| 聊天集成 | `src/ui/useChat.ts` | 幂等初始化、工具注册、状态查询 |
| 指令入口 | `src/commands/mcp.ts` | `/mcp` 斜杠指令定义 |
| 顶层调度 | `src/ui/App.tsx` | 面板互斥、异步状态管理 |

## 二、配置系统

### 2.1 三级优先级（高 → 低）

```
1. ~/.ccode/.mcp.json     — cCli 专属配置，优先级最高
2. ~/.claude.json         — Claude Code 用户配置（mcpServers 字段）
3. ~/.mcp.json            — 通用 MCP 配置格式
```

同名 Server 出现在多个文件时，高优先级覆盖低优先级。

### 2.2 配置格式

```json
{
  "mcpServers": {
    "mysql": {
      "command": "python",
      "args": ["-m", "mysql_mcp"],
      "env": { "DB_HOST": "localhost" }
    },
    "deepwiki": {
      "type": "streamable-http",
      "url": "https://mcp.deepwiki.com/mcp"
    },
    "web-search": {
      "type": "http",
      "url": "https://api.example.com/mcp",
      "headers": {
        "Authorization": "Bearer xxx"
      }
    }
  }
}
```

### 2.3 支持的传输类型

| type 字段 | Transport | 说明 |
|-----------|-----------|------|
| (无，有 command) | StdioClientTransport | 本地进程，通过 stdin/stdout 通信 |
| `sse` | SSEClientTransport | Server-Sent Events |
| `streamable-http` | StreamableHTTPClientTransport | MCP 推荐的 HTTP 传输 |
| `http` | StreamableHTTPClientTransport | `.claude.json` 兼容格式 |
| (无，有 url) | StreamableHTTPClientTransport | 默认 HTTP |

### 2.4 核心接口

```typescript
interface McpServerConfig {
  command?: string
  args?: string[]
  env?: Record<string, string>
  type?: 'stdio' | 'sse' | 'streamable-http' | 'http'
  url?: string
  headers?: Record<string, string>
}

interface McpConfigWithSources {
  mcpServers: Record<string, McpServerConfig>
  serverSources: Record<string, string>  // server name → 来源文件路径
}
```

## 三、连接管理（McpManager）

### 3.1 并行连接

所有 Server 通过 `Promise.allSettled` 并行连接，单点故障不影响其余：

```typescript
const results = await Promise.allSettled(
  entries.map(async ([name, config]) => {
    const client = new Client(...)
    await client.connect(transport)
    const tools = await client.listTools()
    return { server, info }
  })
)
```

### 3.2 状态追踪

每个 Server 的状态（含失败的）都被记录：

```typescript
interface ServerInfo {
  name: string
  status: 'connected' | 'failed'
  source: string          // 来源配置文件
  toolCount: number
  toolNames: string[]
  error?: string          // 失败原因
}
```

### 3.3 容错措施

- **stderr 管道化**：`stderr: 'pipe'` 阻止子进程日志泄漏到终端
- **独立 try-catch**：每个 Server 独立捕获异常
- **优雅关闭**：`disconnectAll()` 忽略单个关闭错误

## 四、工具适配（McpTool）

### 4.1 命名空间

```
原始名：search
适配后：mcp__wikipedia__search
```

格式：`mcp__<serverName>__<toolName>`，避免多 Server 同名工具冲突。

### 4.2 权限控制

所有 MCP 工具标记为 `dangerous = true`，执行前需用户确认。用户可选择"总是允许"加入白名单。

### 4.3 执行流程

```
McpTool.execute(args)
  → client.callTool({ name, arguments })
  → 提取 text content 拼接
  → 返回 { success, output } 或 { success: false, error }
```

## 五、聊天流程集成

### 5.1 生命周期

```
首次 submit() 或 /mcp
  → ensureMcpInitialized()  [幂等，只执行一次]
    → loadMcpConfigWithSources()
    → new McpManager(config)
    → mcpManager.connectAll()

每次 submit()
  → ensureMcpInitialized()  [已初始化则跳过]
  → registerMcpTools(registry)  [注册到本轮 ToolRegistry]
  → AgentLoop.run()  [MCP 工具可被 LLM 调用]
```

### 5.2 关键函数

| 函数 | 职责 |
|------|------|
| `ensureMcpInitialized()` | 幂等初始化，连接所有 Server |
| `registerMcpTools(registry)` | 将 MCP 工具注册到 ToolRegistry |
| `getMcpInfo()` | 触发初始化 + 返回 ServerInfo[]（`/mcp` 用） |

## 六、UI 交互

### 6.1 `/mcp` 指令流程

```
用户输入 /mcp
  → McpCommand → { type: 'show_mcp_status' }
  → setMcpLoading(true)
  → getMcpInfo() 异步连接
  → 显示"MCP Server 连接中..."
  → 连接完成 → McpStatusView 全屏面板
  → 用户按 Esc/q → 关闭面板，恢复 InputBar
```

### 6.2 面板互斥层级

```
PermissionDialog > ModelPicker > McpStatusView > InputBar
```

同一时刻只渲染一个交互组件，高优先级面板阻断低优先级。

### 6.3 状态面板展示

```
Manage MCP servers
  4 servers

  MCPs (~/.ccode/.mcp.json)
    deepwiki · ✔ connected
      - mcp__deepwiki__ask_question

  MCPs (~/.mcp.json)
    mysql · ✔ connected
      - mcp__mysql__mysql_execute_read_query

Esc/q 退出
```

## 八、错误处理策略

| 场景 | 处理方式 | 用户感知 |
|------|---------|---------|
| 配置文件不存在/格式错误 | 跳过，继续加载下一个 | 透明 |
| Server 连接超时/失败 | 记录错误，不阻断其他 Server | 面板显示 ✘ failed |
| MCP 工具执行失败 | 返回 `{ success: false, error }` | 聊天中显示错误 |
| 子进程 stderr 日志 | `stderr: 'pipe'` 管道化 | 终端无噪音 |

## 九、关键设计决策

### 为什么幂等初始化？
MCP Server 连接跨越整个会话，每条消息重连会有性能开销。`ensureMcpInitialized()` 通过 flag 保证只连接一次。

### 为什么并行连接？
串行连接时 N 个 Server 的等待时间是各自延迟之和；并行后取决于最慢的一个。

### 为什么所有 MCP 工具标记 dangerous？
MCP Server 的工具功能不可预测（可能执行系统命令、网络请求），统一需要权限确认。

### 为什么用命名空间 `mcp__server__tool`？
避免多 Server 同名工具冲突，权限确认时明确来源。

### 为什么面板互斥而非叠加？
终端 UI 空间有限，互斥模式保证每个面板都能独占输入焦点。
