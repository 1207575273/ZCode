# Web UI 踩坑与前端工程经验

> 日期: 2026-03-15
> 场景: ZCli Web UI (React SPA + Vite + Bridge Server) 开发过程中遇到的真实问题
> 价值: 每个问题都是实际踩过的坑，附带根因分析和修复方案

---

## 一、Vite 前端项目的 import 路径规范

### 问题

Web 前端（`web/src/`）的 `.tsx` 文件中 import 写了 `.js` 后缀：

```typescript
// web/src/main.tsx
import { App } from './App.js'  // ← 这个 .js 后缀是问题根源
```

初期能跑是因为 `tsc` 编译产生了对应的 `.js` 文件在 `src/` 目录下，Vite 直接加载了这些 `.js` 而不是 `.tsx` 源码。清理 `.js` 产物后立即白屏——`404 Not Found: /src/App.js`。

### 根因

**CLI 项目（`zCli/src/`）** 和 **Web 前端项目（`web/src/`）** 有不同的模块解析策略：

| 项目 | 构建工具 | 模块解析 | import 后缀 |
|------|---------|---------|------------|
| CLI (`zCli/src/`) | tsup (ESM) | Node ESM 规范 | **必须带 `.js`** |
| Web (`web/src/`) | Vite (bundler) | bundler 模式 | **不能带 `.js`** |

- Node ESM 规范要求 import 带文件扩展名（`.js`），tsup 遵循此规范
- Vite 使用 bundler 模式解析模块，自动识别 `.ts`/`.tsx`，写 `.js` 反而找不到

### 修复

```typescript
// CLI 项目（保持 .js 后缀）
import { AgentLoop } from '@core/agent-loop.js'

// Web 前端项目（去掉 .js 后缀）
import { App } from './App'
import { ChatPage } from './pages/ChatPage'
import type { ChatMessage } from '../types'
```

### 经验

**两个项目虽然都是 TypeScript，但构建工具不同导致 import 规范完全相反。** 新增 Web 前端文件时不要从 CLI 代码复制 import 语句。

## 二、.js 编译产物污染

### 问题

`web/src/` 目录下混入了 `.js` 编译产物（由 IDE 的 TypeScript 服务或手动 `tsc` 生成）。这些文件被提交到 git，导致：

1. Vite 加载旧版 `.js` 而不是最新的 `.tsx`——样式缺失、颜色不生效
2. 组件行为与源码不一致——修改 `.tsx` 后无效果
3. 清理 `.js` 后白屏（import 路径依赖了这些文件）

### 根因

`web/tsconfig.json` 的 `moduleResolution: "bundler"` 不会生成 `.js`，但：
- WebStorm 的 TypeScript 服务可能在后台编译
- 手动运行 `tsc` 时如果没加 `--noEmit` 会产生 `.js`
- 这些文件一旦存在，Vite 优先解析 `.js`（因为 import 写了 `.js` 后缀）

### 修复

```gitignore
# web/.gitignore
node_modules/
dist/
# TypeScript 编译产物（Vite 直接用 .tsx 源码）
src/**/*.js
```

### 经验

**Vite 前端项目的 `src/` 下永远不应该有 `.js` 文件。** `.gitignore` 必须排除 `src/**/*.js`。如果发现样式/行为与源码不一致，第一步检查是否有 `.js` 产物干扰。

## 三、工具执行记录的渲染顺序

### 问题

Web UI 上工具执行记录出现在 AI 回复**之后**，而 CLI 端是正确的顺序：

```
预期（CLI 一致）:
  用户提问 → AI 思考文本 → 工具执行 → AI 最终回复

实际（Web 错误）:
  用户提问 → AI 全部文本 → 工具执行历史
```

### 根因

AgentLoop 的事件流是交替的：

```
text(AI思考) → tool_start → tool_done → text(AI继续) → done
```

但 Web 端把所有 `text` 累积成一个字符串，所有 `tool` 累积成一个数组，`done` 时一起输出——丢失了交替关系。

### 修复

**`tool_start` 到来时，先把已累积的流式文本固化为 assistant 消息**：

```typescript
case 'tool_start':
  // 先固化已有文本
  setStreaming(prev => {
    if (prev) {
      setMessages(msgs => [...msgs, { role: 'assistant', content: prev }])
    }
    return ''
  })
  // 再添加工具事件
  setToolEvents(prev => [...prev, { ... }])
  break
```

**`tool_done` 时立即写入消息历史**（不等 `done` 批量）：

```typescript
case 'tool_done':
  setToolEvents(prev => {
    const running = prev.find(e => e.toolCallId === event.toolCallId)
    setMessages(msgs => [...msgs, { role: 'system', toolEvents: [completed] }])
    return prev.filter(e => e.toolCallId !== event.toolCallId)
  })
  break
```

这样渲染顺序就对了：`AI文本 → 工具 → AI文本 → 工具 → AI文本`。

### 经验

**流式事件的消费不能"攒着一起输出"，必须按到达顺序实时处理。** 特别是 text 和 tool 交替出现时，每个 tool_start 都是一个"断点"，需要先把之前的文本固化。

## 四、工具历史渲染样式不一致

### 问题

工具执行中（ToolStatus 组件）效果好看——多行布局、颜色正确、参数摘要清晰。
但执行后（ToolHistoryBlock 组件）变成了一行灰色横条，对勾没颜色，输出挤在一起。

### 根因

**两个组件各写各的**——ToolStatus 管实时，ToolHistoryBlock 管历史。样式、布局、参数提取逻辑全部重复且不一致。再加上 `.js` 产物污染，ToolHistoryBlock 实际加载的是旧版代码。

### 修复

**删除 ToolHistoryBlock，统一使用 ToolStatus 一个组件**。给它加 running/done 两态：

- running：黄色脉冲 `⟳` 图标
- done：绿色 `✓` / 红色 `✗`，可点击折叠/展开结果子块

MessageBubble 中工具历史也用 ToolStatus 渲染：

```tsx
if (isSystem && message.toolEvents?.length > 0) {
  return <ToolStatus events={message.toolEvents} />
}
```

### 经验

**同一类数据的渲染永远只用一个组件。** "实时"和"历史"只是状态不同，不是两个东西。拆成两个组件必然导致样式漂移。

## 五、消息重复显示（事件回路）

### 问题

Web 发一条"你好"，页面显示三条：(web) 你好 / (web) 你好 / (cli) 你好。

### 根因

三重 echo：
1. Web 发 `chat` → Bridge → EventBus emit `user_input(web)` → Bridge 推回给 Web（echo 自己）
2. useChat 监听到 web 输入 → 调 submit → submit 内部 emit `user_input(cli)` → 又推给 Web
3. ChatPage 收到两条事件各创建一条消息

### 修复

三层防护：
- **Bridge Server**：不 echo `source: 'web'` 的 `user_input` 回给 Web 客户端
- **useChat.submit**：加 `_source` 参数，web 触发时不重复 emit
- **ChatPage**：本地 optimistic 添加用户消息，不依赖服务端 echo

### 经验

**双向通信系统必须在设计时就确定"谁负责显示"。** 原则：发送方本地显示，接收方通过事件显示。中间路由器不 echo 回发送方。

## 六、WebSocket 连接被 Vite 反代拦截

### 问题

`WebSocket connection to 'ws://localhost:9800/ws' failed: WebSocket is closed before the connection is established.`

### 根因

Bridge Server 用 `app.all('*')` 做 Vite 反代，这个通配符匹配了 `/ws` 路径。WebSocket 握手是一个带 `Upgrade: websocket` 头的 GET 请求，被当作普通 HTTP 转给 Vite，Vite 不认 WebSocket upgrade，返回普通响应，握手失败。

### 修复

```typescript
app.all('*', async (c, next) => {
  const path = new URL(c.req.url).pathname
  if (path === '/ws' || path.startsWith('/api/')) return next()  // ← 排除
  // ... Vite 反代逻辑
})
```

### 经验

**反向代理必须排除 WebSocket 和 API 路径。** `app.all('*')` 看起来方便但会拦截所有请求包括 WebSocket upgrade。注册反代路由时要显式 skip 自己的路径。

## 七、Session ID 与路由匹配

### 问题

直接访问 `http://localhost:9800`（不带 `/session/xxx`），消息不通。Web 能连上 WebSocket 但发消息没反应。

### 根因

Web 注册时 `sessionId` 为空字符串，CLI 注册的是真实 sessionId（如 `019cebb1-xxx`），两者不匹配，Bridge 路由找不到对方。

### 修复

Bridge Server 在 Web 注册空 sessionId 时，自动分配第一个活跃的 CLI session：

```typescript
if (sender.clientType === 'web' && !sid) {
  for (const cl of clients.values()) {
    if (cl.clientType === 'cli' && cl.sessionId) {
      sid = cl.sessionId
      break
    }
  }
}
```

### 经验

**WebSocket 路由匹配失败时要有降级策略。** 空 sessionId 不应该静默失败，而应该自动分配可用 session。错误应该显性化——至少在 DevTools 里能看到。

## 八、生产模式静态资源路径

### 问题

生产构建后 `node dist/bin/zcli.js --web` 访问 9800 返回 404。

### 根因

`serveStatic({ root: distDir })` 的 `distDir` 用 `import.meta.dirname` 计算，构建后 `server.js` 在 `dist/src/bridge/` 下，`../../web/dist` 指向 `dist/web/dist`——但实际构建产物在 `web/dist/`。

### 修复

生产模式用 `process.cwd()` 基准：

```typescript
const distDir = isDev
  ? join(import.meta.dirname ?? '.', '../../web/dist')  // dev: 从源码目录算
  : join(process.cwd(), 'web/dist')                      // 生产: 从 cwd 算

// serveStatic 也改为相对 cwd
app.use('/*', serveStatic({ root: './web/dist' }))
```

### 经验

**`import.meta.dirname` 在构建后指向产物目录，不是源码目录。** 静态资源路径应该用 `process.cwd()` 或者在构建时注入绝对路径。不要假设构建后的目录结构和源码一致。

## 九、React StrictMode + WebSocket 重复连接（根因级 Bug）

### 问题

AI 回复结束后 Markdown 渲染乱码——表格变纯文本、标题重复、内容混乱。刷新页面后恢复正常。

调试日志发现文本内容本身就是错的：
```
first 100: 我我目前目前拥有拥有以下工具：
```
每个 chunk 被累加了两次。

### 排查过程

1. **最初怀疑**：ReactMarkdown 渲染问题 → 尝试 requestAnimationFrame / setTimeout 分帧 → 无效
2. **加日志发现**：`done` 事件到达两次，`setMessages` 被调用两次
3. **怀疑 `useState + useEffect` 丢事件** → 改为 callback 模式 → `done` 不再丢失
4. **关键突破**：日志显示文本内容本身重复（"我我目前目前"）→ 不是渲染问题，是**数据就错了**
5. **根因定位**：React StrictMode 导致两个 WebSocket 连接同时活着

### 根因分析

React 18 StrictMode 在开发模式下会**执行两次 useEffect**（mount → cleanup → mount）。

`useWebSocket` 的 `useEffect` 调用 `connect()` 创建 WebSocket：

```
StrictMode 执行流程：
  1. Effect 首次运行: connect() → 创建 ws1
  2. Cleanup 运行: ws1.close()
  3. Effect 再次运行: connect() → 创建 ws2

  但 ws1.close() 触发 ws1.onclose:
    → wsRef.current = null
    → setTimeout(connect, 2000) → 2秒后创建 ws3!

  最终：ws2 和 ws3 同时活着，都注册到 Bridge Server
  → 每个事件被 Bridge 转发给 ws2 和 ws3
  → onEventRef.current(event) 被调两次
  → turnTextRef.current += event.text 执行两次
  → 文本翻倍
```

### 修复

```typescript
// 1. connect() 先关旧连接
const connect = useCallback(() => {
  if (wsRef.current) {
    wsRef.current.onclose = null  // 阻止触发重连
    wsRef.current.close()
    wsRef.current = null
  }
  clearTimeout(reconnectTimer.current)
  const ws = new WebSocket(WS_URL)
  // ...
}, [sessionId])

// 2. onclose 检查身份：只有当前活跃连接断开才重连
ws.onclose = () => {
  if (wsRef.current === ws) {  // ← 关键检查
    reconnectTimer.current = setTimeout(connect, RECONNECT_INTERVAL_MS)
  }
}

// 3. cleanup 时不触发重连
return () => {
  if (wsRef.current) {
    wsRef.current.onclose = null  // ← 移除 onclose
    wsRef.current.close()
  }
}
```

### 经验

**WebSocket + React StrictMode 是一个经典陷阱。** StrictMode 的 effect 重放 + WebSocket 的 onclose 自动重连 = 幽灵连接。三个防护缺一不可：

1. **connect 前关旧连接** — 幂等连接
2. **onclose 检查 `wsRef.current === ws`** — 只有"当前"连接断开才重连，"过期"连接断开忽略
3. **cleanup 时 `ws.onclose = null`** — 阻止 cleanup 的 close 触发重连

### 附带修复

同时修复了 `useWebSocket` 的另一个设计缺陷：原来用 `useState(lastEvent)` + `useEffect` 处理事件，React 18 批处理下多个事件只保留最后一个（`done` 事件可能被吞）。改为 **callback 模式**：`onEvent` 回调直接从 `ws.onmessage` 调用，每个事件都保证被处理。

## 十、前端项目与 CLI 项目的工程差异

| 维度 | CLI 项目 (zCli/src/) | Web 前端 (web/src/) |
|------|---------------------|-------------------|
| 构建工具 | tsup | Vite |
| 模块解析 | Node ESM（必须 .js 后缀） | bundler（不需要后缀） |
| TypeScript 编译 | tsup 处理 | Vite 处理 |
| .js 文件 | 正常产物（dist/） | **不应存在**（Vite 直接用 .tsx） |
| 静态资源 | 无 | dist/（Vite build 产物） |
| HMR | 无（重启 CLI） | Vite HMR 即时刷新 |
| 路径别名 | tsconfig paths + tsup | Vite resolve.alias（或 tsconfig） |

**核心教训**：两个项目虽然共处一个仓库、都用 TypeScript，但**构建链完全独立**。不能互相复制配置和 import 写法。
