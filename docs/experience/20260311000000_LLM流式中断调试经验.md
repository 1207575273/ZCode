# LLM 流式中断（Abort）调试经验

> 来源：F18 对话中断与输入打断功能实现过程中的三个 Bug
> 日期：2026-03-11

---

## 背景

实现用户在 LLM 流式响应期间按 Escape/Ctrl+C 中断对话的功能。核心链路：

```
用户按 Escape → AbortController.abort() → Provider 中断流 → agent-loop 捕获 → useChat 记录中断
```

看似简单的 abort 链路，实际踩了三个层层递进的坑。每个坑都掩盖了下一个坑的存在——只有修好前一个，才能发现后一个。

---

## 坑 1：Ink exitOnCtrlC 默认行为

### 现象
按 Ctrl+C 直接退出程序，`useInput` hook 中的中断逻辑完全不执行。

### 排查过程
- 代码逻辑看起来没问题，但 Ctrl+C 根本没触发 `useInput` 回调
- 查阅 Ink 文档发现：`render()` 默认 `exitOnCtrlC: true`

### 根因
Ink 在 raw mode 下拦截 Ctrl+C，**直接调用 `process.exit()`**，优先级高于 `useInput`。事件根本到不了我们的 hook。

### 修复
```typescript
// bin/ccli.ts
render(React.createElement(App, { ... }), { exitOnCtrlC: false })
```

### 经验
**框架的默认行为可能是你最大的敌人。** 当事件"不触发"时，先检查框架层是否拦截了事件，而不是一直在业务代码里找 bug。

---

## 坑 2：AbortError 不叫 AbortError

### 现象
修好坑 1 后，Ctrl+C 可以中断了，但 session 日志显示 `stopReason: "error"` 而非 `"abort"`。`[Request interrupted by user]` 消息也没有被记录。

### 排查过程
- 检查 agent-loop 的 catch 块，发现判断条件是 `err.name === 'AbortError'`
- 打印实际错误发现：`name: 'Error'`, `message: 'This operation was aborted'`
- Node.js 原生 fetch 抛出的确实是 `name='AbortError'`，但经过 LangChain SDK 包装后，变成了普通 Error

### 根因
LangChain 等第三方库在内部 catch 了原始 AbortError，重新包装为普通 `Error` 对象。原始的 `name='AbortError'` 信息丢失了，只在 `message` 中保留了 "aborted" 关键词。

### 修复
```typescript
export function isAbortError(err: unknown): boolean {
  if (!(err instanceof Error)) return false
  if (err.name === 'AbortError') return true
  if (err.message.toLowerCase().includes('aborted')) return true
  return false
}
```

### 经验
**不要依赖第三方库保留错误的精确类型。** 错误在传播链中经过多层包装，`name` 和 `constructor` 都可能改变。设计错误检测时，多条件宽松匹配比精确匹配更健壮。

## 坑 3：Provider yield error chunk 而非 throw

### 现象
修好坑 2 后，`isAbortError` 已经能识别包装后的错误。但 session JSONL 日志仍然显示 `stopReason: "error"`，`[Request interrupted by user]` 依然没有出现。

### 排查过程
这是最隐蔽的一个坑。关键线索来自 session JSONL：

```json
{"type":"llm_call_end", "stopReason":"error", "error":"This operation was aborted"}
```

注意 `stopReason` 是 `"error"` 而不是 `"abort"`。说明 agent-loop 的 catch(isAbortError) 分支**根本没有执行**。

追踪 agent-loop 代码流程：

1. Provider 内部 catch 了 abort 错误
2. **没有重新 throw**，而是 yield 了一个 `{ type: 'error', error: 'This operation was aborted' }` chunk
3. agent-loop 的 `#callLLM` 中，error chunk 的处理路径是：
   ```typescript
   yield makeLlmError(errorMsg, outputTokens)  // → stopReason: "error"
   yield mapped
   return { toolCalls: [], aborted: true }      // 正常 return，不 throw！
   ```
4. `run()` 方法看到 `aborted: true`，直接 `return`
5. useChat 的 `for await` **正常结束**，`catch(isAbortError)` 永远不执行

### 根因
LangChain provider 处理 abort 有**两条路径**：
- **路径 A（throw）**：直接抛出错误 → 被 catch 捕获 → `isAbortError` 判断 → 正确处理
- **路径 B（yield error chunk）**：不抛出，而是 yield 错误事件 → error chunk 被当作普通错误处理 → return 而非 throw → catch 永远不触发

坑 2 修的是路径 A，但实际走的是路径 B。

### 修复
在 error chunk 处理中增加 abort 检测，遇到 abort 相关的 error chunk 时**重新 throw**：

```typescript
if (mapped.type === 'error') {
  const errorMsg = chunk.error ?? 'unknown error'
  // Provider 将 abort 错误包装为 error chunk（不抛出）→ 重新抛出使 catch 路径生效
  if (errorMsg.toLowerCase().includes('aborted')) {
    const abortErr = new Error(errorMsg)
    abortErr.name = 'AbortError'
    throw abortErr
  }
  yield makeLlmError(errorMsg, outputTokens)
  yield mapped
  return { toolCalls: [], aborted: true }
}
```

### 经验
**一个错误可能通过两条完全不同的路径传播。** 当你修好了一条路径却发现问题仍在，不是你的修复有问题——是错误走了另一条路。

对于 async generator（`for await...of`）模式，尤其要注意：generator 可能通过 throw 传播错误（触发 catch），也可能通过 yield 传播错误（触发普通分支处理），两者的控制流完全不同。

---

## 总结：三层洋葱

这三个 bug 构成一个"洋葱模型"——每修一层才能看到下一层：

```
第 1 层（框架层）：Ink 拦截了 Ctrl+C → 事件到不了业务代码
第 2 层（错误识别层）：LangChain 改了错误的 name → 识别不出 AbortError
第 3 层（传播路径层）：LangChain yield error chunk 而非 throw → catch 永远不触发
```

### 通用调试策略

1. **从日志出发，不要猜**：每次都是通过 session JSONL 的实际输出确认问题所在
2. **验证你的假设**：代码看起来对不代表执行路径对。"catch 会被触发"是假设，日志证明它没有
3. **追踪完整路径**：不要只看代码片段，要追踪从事件源到最终处理的完整链路
4. **第三方库是黑盒**：不要假设库会按你期望的方式传播错误。同一个库的不同版本、不同 provider，行为可能不同
5. **修一个 bug 后立即验证**：不要假设修好了就行，立即测试并检查日志确认
