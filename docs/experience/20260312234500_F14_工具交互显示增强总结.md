# F14 工具交互显示增强 — 实现总结

## 一、背景与目标

对标 Claude Code 的工具交互显示哲学，提升 cCli 工具调用的信息密度和视觉体验：

- Claude Code 的工具显示：`● Bash(cd "/tmp" && mvn compile)` + `⎿ [INFO] BUILD SUCCESS`
- cCli 改造前：工具执行仅有 spinner，完成后消失，无历史记录，信息丢失严重

**核心设计原则**：头部行精简（工具名 + 参数摘要 + 耗时），输出内容降级到 `⎿` 子块，已完成工具持久化到 Static 历史区。

## 二、实现内容

### P0 — 工具头部信息增强 + 历史持久化（`480bdad`）

**改动范围**：UI 层 + useChat hook

1. **ToolStatusLine.tsx 重写**
   - `buildArgsSummary()`: 按工具类型提取参数摘要（bash→command, read/write/edit→path, grep→pattern+path, glob→pattern, dispatch_agent→description）
   - `buildOutputPreview()`: resultSummary 拆分为预览行 + 折叠提示，模仿 `⎿` 子块样式
   - 三态渲染：Running（spinner + 动作描述）、Done（✓ + 工具名 + 参数 + ⎿ 输出）、Error（✗ + 红色）

2. **ChatView.tsx 扩展**
   - `ChatMessage` 新增 `role: 'tool'` + `toolCall?: CompletedToolCall`
   - 动态区域仅渲染 `status === 'running'` 的工具事件
   - 已完成工具通过 messages 数组进入 `<Static>` 固化区

3. **useChat.ts 生命周期改造**
   - `tool_done` 时：从 toolEvents 移除 → 构建 CompletedToolCall → 写入 messages 历史
   - `toolEventsRef` 双轨模式：ref 供异步回调读取（避免闭包过期），state 驱动 React 渲染
   - SubAgent 进度事件透传到 UI

4. **format-utils.ts 新建**
   - `formatDuration()`: ms → 人类可读（`3ms` / `1.2s` / `2m 30s`）
   - `truncate()`: 超长文本截断 + `...`

**关键决策**：最初将 resultSummary 放在头部行（`✓ Write(path) 已写入...`），用户反馈太丑，改为头部只保留工具名+参数+耗时，输出内容降到 `⎿` 子块。

### P1 — 结构化元数据 + 丰富渲染（`51aaf4b`）

**改动范围**：工具层（meta 生产）+ UI 层（meta 消费）

1. **ToolResultMeta 类型定义**（`src/tools/types.ts`）
   ```typescript
   export type ToolResultMeta =
     | { type: 'edit'; path: string; addedLines: number; removedLines: number; diff: string }
     | { type: 'write'; path: string; totalLines: number; preview: string }
     | { type: 'read'; path: string; totalLines: number }
     | { type: 'grep'; matchCount: number; fileCount: number }
     | { type: 'glob'; fileCount: number }
   ```

2. **各工具返回 meta**
   - `edit-file.ts`: 计算 addedLines/removedLines，生成 `-old/+new` diff 字符串
   - `write-file.ts`: 统计总行数，取前 4 行作为 preview
   - `read-file.ts`: 统计读取总行数
   - `grep.ts`: 累计 matchCount 和 fileCount
   - `glob.ts` (未单独改动，已有 fileCount)

3. **agent-loop.ts 透传 meta**
   - `tool_done` 事件新增 `meta?: ToolResultMeta`，条件展开避免 exactOptionalPropertyTypes 报错

4. **buildMetaDisplay() 渲染**（`ToolStatusLine.tsx`）
   - edit: 头部 `+3 -1`，⎿ 块红绿 diff 行
   - write: 头部 `42 lines`，⎿ 块内容预览
   - read: 头部 `180 lines`，无输出块
   - grep: 头部 `5 matches in 3 files`，⎿ 块匹配结果预览
   - glob: 头部 `12 files`，⎿ 块文件列表预览

### P2 — 管道模式 --verbose（`d37b2d3`）

**改动范围**：入口 + pipe-runner

1. **bin/ccli.ts**: 新增 `--verbose` / `-v` 参数解析
2. **pipe-runner.ts**:
   - `tool_start` → stderr: `[tool] read_file: package.json...`
   - `tool_done` → stderr: ` ✓  42 lines  3ms`
   - 复用 `buildArgsSummary` + `formatDuration` 保持显示一致性
   - 不污染 stdout 管道数据流

## 三、架构设计要点

### 数据流

```
工具执行 → ToolResult { output, meta? }
    ↓
AgentLoop → tool_done 事件 { resultSummary, meta? }
    ↓
useChat hook → CompletedToolCall → ChatMessage(role:'tool')
    ↓
ChatView → <Static> 固化 → ToolHistoryBlock(meta→丰富渲染)
```

### exactOptionalPropertyTypes 约束

tsconfig 开启 `exactOptionalPropertyTypes` 后，`meta?: ToolResultMeta` 不能赋值 `undefined`。全链路使用条件展开：

```typescript
...(result.meta !== undefined ? { meta: result.meta } : {})
```

### 双轨 ref/state 模式

`toolEventsRef` 与 `toolEvents` 状态同步维护：
- **ref**：异步回调（AgentLoop 事件处理）中读取，避免 React 闭包捕获过期 state
- **state**：驱动 UI 重渲染

## 六、经验总结

1. **UI 信息密度控制**：头部行只放"是什么 + 花了多久"，具体内容降级到子块。信息过载比信息不足更糟糕。
2. **工具历史持久化**：Ink `<Static>` 的 items 只能追加不能修改——工具完成后必须转化为 ChatMessage 写入 messages，而非留在 toolEvents 状态中。
3. **meta 透传设计**：工具层只负责"生产"结构化元数据，AgentLoop 透明转发，UI 层"消费"渲染。各层职责清晰，新增工具类型只需扩展 union type + 对应 case。
4. **管道模式与交互模式复用**：`buildArgsSummary` 和 `formatDuration` 在两种模式间共享，保持显示一致性。管道模式的 verbose 输出走 stderr 是 Unix 惯例。
